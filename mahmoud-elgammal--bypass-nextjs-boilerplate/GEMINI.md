# Repository Guidelines

## Project Structure & Module Organization
- App Router lives in `src/app` (layouts, routes, API routes).
- UI components in `src/components` (PascalCase files), providers in `src/providers`.
- State with Zustand in `src/stores`; server actions in `src/actions`.
- Reusable React hooks in `src/hooks` (files named `useX.ts`).
- Utilities and HTTP in `src/lib`; schemas in `src/schemas`; constants in `src/constants`.
- i18n config in `src/i18n`; message catalogs under `messages/<locale>/<ns>.json`.
- Tests live in `__tests__` (root or under `src/**/__tests__`).

## Coding Style & Naming Conventions
- TypeScript everywhere; 2â€‘space indent (Biome enforced).
- Components: PascalCase (`ThemeToggle.tsx`); hooks: `useX.ts` under `src/hooks`.
- Server actions in `src/actions/*.ts`; API routes under `src/app/api/**/route.ts`.
- Organize imports and fix lint with `pnpm lint` before pushing.

## Storybook
- Config at `.storybook/`. Stories co-located as `*.stories.tsx` under `src/`.
- Global CSS is loaded in `preview.tsx`; basic i18n messages are mocked.
- Example: `src/components/ThemeToggle.stories.tsx`.

## Testing Guidelines
- Framework: Vitest with `jsdom`; place specs as `__tests__/*.{spec,test}.ts(x)`.
- Prefer unit tests near features; mock DOM or network where needed.
- Keep i18n keys in sync using `pnpm i18n:check`.
- Run `pnpm test` locally; add `--coverage` or set `COVERAGE=true` to generate coverage.
- CI uploads `coverage/lcov.info` to Codecov when `CODECOV_TOKEN` is set.

## Commit & Pull Request Guidelines
- Make small, focused commits with imperative subject: `feat: add locale switcher`.
- Reference issues where relevant: `Closes #123`.
- PRs must include: summary, rationale, screenshots for UI, test plan, and any config changes.
- Ensure CI passes (build, lint, tests) before requesting review.

### Commitlint
- Conventional commits enforced via Commitlint (`.commitlintrc.cjs`). Examples: `feat(app): add locale switcher`, `fix(components): handle null props`.
- Local hook: `commit-msg` runs Commitlint (Lefthook). CI also validates commit messages.
- Commit rule: You MUST read and strictly follow the rules in `.agent/rules/commitlint/rule.md` before generating ANY commit message.

## Security & Configuration Tips
- Copy `.env.example` to `.env.local`; never commit secrets.
- Environment validated via `@t3-oss/env-nextjs`; add new vars to the schema.
- Optional: configure Sentry DSN if error reporting is desired.
- Server/runtime instrumentation in `instrumentation.ts` initializes Sentry when `SENTRY_DSN` is set.
- Scheduled security checks run via GitHub Actions: CodeQL, OSV, Trivy, and pnpm audit.

## Architecture Overview
- Next.js App Router with middleware-based locale negotiation (`middleware.ts`).
- Internationalization via `next-intl` (see `src/i18n/config.ts`).
- Client state with Zustand; schemas with Zod; styling via Tailwind CSS.
 - Million Lint integrated via Next plugin; Bundle Analyzer enabled via `ANALYZE=true`.
 - Partytown added in `src/app/layout.tsx` to offload thirdâ€‘party analytics.
- next-seo configured in `src/lib/seo.ts` and applied via `DefaultSeo` in root layout.
 - Sitemap/robots generated by `next-sitemap` (`next-sitemap.config.mjs`) on build; manifest at `src/app/manifest.ts`.
 - SWR configured via `SWRProvider` in locale layout with initial fallback for `/api/site`.
 - Browser caching: client cache utilities in `src/lib/clientCache.ts` and hook `useCachedJson` with ETag support; example component `src/components/SiteInfoClient.tsx` caches `/api/site` and lets users refresh.

## Security Headers & CSP
- Configured in `next.config.mjs` using `headers()` with a restrictive CSP.
- If you add thirdâ€‘party scripts, update `script-src`, `connect-src`, and `frame-src` directives accordingly.

## CSRF Protection
- CSRF token is issued as cookie `CSRF_TOKEN` (SameSite=Lax) by middleware.
- Forms include a hidden `"_csrf"` field (example: consent form). Server actions validate against the cookie.
- `httpJson` adds `X-CSRF-Token` for mutating requests; API routes validate it (example: `src/app/api/site/revalidate/route.ts`).

## Healthcheck
- `GET /api/health` returns `{ status: "ok" }` for uptime checks and load balancers.

## Analytics & Tags
- Consent-aware loading lives in `src/components/Analytics.tsx`.
- Google: set one of
  - `NEXT_PUBLIC_GTM_ID` (GTM-XXXXXXX) for Tag Manager, or
  - `NEXT_PUBLIC_GTAG_ID` (G-XXXXXXXXXX) for GA4.
- PostHog: set `NEXT_PUBLIC_POSTHOG_KEY` and optional `NEXT_PUBLIC_POSTHOG_HOST` (defaults to `https://app.posthog.com`).
- Tags load only after analytics consent; GTM takes precedence over GA4; PostHog can run alongside.

## Git Hooks (Lefthook)
- Hooks configured in `lefthook.yml`. Install hooks with `pnpm prepare` or `pnpm dlx lefthook install`.
- Pre-commit: formats staged files (Biome) and lints TS/JS.
- Pre-push: runs typecheck, unit tests, and i18n key parity.

## Releases
- Conventional commits drive versioning via Semantic Release (`.releaserc`).
- Branching model:
  - `main` â†’ stable releases (tags: `vX.Y.Z`).
  - `beta` â†’ pre-releases (channel `beta`, tags like `vX.Y.Z-beta.N`).
  - `alpha` â†’ pre-releases (channel `alpha`, tags like `vX.Y.Z-alpha.N`).
  - `feature/*` â†’ no releases; CI runs checks.
- CI publishes on pushes to `main`, `beta`, and `alpha` after all checks.
- Configure `GITHUB_TOKEN` (default Actions token works) and update repo URL in `.releaserc`.
- Storybook is deployed to `gh-pages` via CI. Enable GitHub Pages (source: `gh-pages` branch) to serve at `https://<owner>.github.io/<repo>/`.
- Lighthouse CI runs on the production build and uploads to temporary storage; tune thresholds in `lighthouserc.json`.

## Notifications
- Slack notifications send a summary after CI completes. Set secrets `SLACK_WEBHOOK_URL` and optional `SLACK_CHANNEL`.

## Branch Protection
- Workflow `Protect Branches` configures protection for `main`, `beta`, and `alpha`.
- Requires a repo admin token in secret `ADMIN_TOKEN` with `repo` scope. Trigger via Actions â†’ Protect Branches â†’ Run workflow.
- Enforces: required checks (`test`, `e2e`, `storybook`, `lighthouse`), 1 approving review, strict status checks, linear history.

## Dependency Updates (Renovate)
- Managed by Renovate; repo extends a local preset at `renovate-presets/shared.json5`.
- To adopt an org-wide preset later, switch `renovate.json` to extend `github:<org>/.github:renovate.json5`.
- Minor/patch updates auto-merge; Storybook, Next.js/React, TypeScript, and release tooling are grouped.
- Dependency PRs are labeled `dependencies`.
- Scheduled: GitHub Actions workflow `Renovate (Scheduled)` runs daily. Set `RENOVATE_TOKEN` (a PAT with `repo` scope) for optimal permissions; Renovate App can also be used instead.

## Vendor Patches
- Use `patch-package` for hot-fixes to dependencies.
- Create patches by editing files in `node_modules`, then run: `pnpm exec patch-package <package-name>`.
- Patches live under `patches/` and apply on install via `postinstall`.
