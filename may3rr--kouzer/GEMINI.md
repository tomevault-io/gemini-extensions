## 项目概览

`抠字儿` 是一个纯静态、单文件实现的中文论文/长文本校对工具。

- 用户把正文粘贴到页面中
- 页面按行切分文本，并用启发式规则筛出“正文”行
- 正文按批次发送到兼容 OpenAI Chat Completions 的接口
- 模型按约定的 XML 格式逐行返回修正结果
- 页面把修改内容以差异高亮的方式展示，并支持拷贝全文

这个仓库当前没有构建系统、包管理器或后端服务。`index.html` 是核心应用。

## 目录结构

- `index.html`: 整个应用的 HTML、CSS、JavaScript 都在这里
- `README.md`: 面向用户的项目介绍和示例
- `img/`: README 和页面中使用的静态图片资源

## 技术与架构约束

- 保持“纯静态页面”优先。除非用户明确要求，不要引入框架、打包器、Node 服务端或复杂依赖。
- 用户可配置任意兼容 OpenAI 的接口，但默认目标是 DeepSeek。
- API Key 目前只保存在页面运行时内存中，刷新后丢失；不要在未明确要求时改成持久化存储。
- 页面文案默认是中文，改动用户可见内容时优先保持中文语境。
- 这个项目的价值点是“轻、直接、可立即打开使用”，改动时不要破坏这一点。

## 核心代码说明

编辑前先理解这些函数，它们决定了主要行为：

- `classifyLine()`: 用启发式规则把每一行分类为正文、标题、目录、参考文献、短行等
- `checkBatch()`: 把一批正文行包装成 XML，调用模型接口
- `parseXMLResponse()`: 解析模型返回的 XML；需要兼容模型偶发输出代码块包裹的情况
- `computeDiff()`: 计算原文和修正文的字符级差异，用于高亮展示
- `startCheck()`: 主流程，负责校验输入、分批并发调用、更新进度、展示结果
- `showResults()`: 把结果渲染到页面，并统计修改数

关键常量：

- `BATCH_SIZE = 5`
- `MAX_CONCURRENT = 5`
- `SYSTEM_PROMPT`: 约束模型只做明确文字修正，不改风格，并强制 XML 输出

如果你调整接口协议、批大小、并发、提示词或结果渲染，请同步检查整条链路是否仍然自洽。

## 本地开发

这是静态站点，优先用本地 HTTP 服务预览：

```bash
python3 -m http.server 8000
```

然后访问 `http://localhost:8000`。

直接双击打开 `index.html` 在某些浏览器下也许能工作，但剪贴板、字体加载或后续扩展行为可能不稳定，不作为首选。

## 修改建议

### UI 修改

- 保持当前“极简、中文 serif、论文感”的视觉语言，除非用户要求重设计
- 同时检查桌面端和移动端布局，当前文件内已有一组窄屏样式
- 不要无故把一个单页工具改造成复杂多页结构

### 模型调用相关修改

- 保持对“OpenAI 兼容接口”的支持，不要把代码写死到某家 SDK
- 如果改动返回解析逻辑，必须考虑模型输出不稳定、代码块包裹、部分行缺失等情况
- 如果增加错误提示，优先让用户知道是 API Key、接口地址、模型名还是响应格式出了问题

### 文本处理修改

- `classifyLine()` 是这个工具避免误改封面、目录、参考文献的重要防线，改规则时要谨慎
- `showResults()` 目前按“仅正文行参与 AI 修改”设计；非正文行通常原样保留
- 差异展示是用户判断修改是否可信的关键，不要轻易弱化

## 验证清单

这个仓库目前没有自动化测试。做完改动后，至少自行检查：

1. 页面能在本地静态服务下正常打开
2. 未填写 API Key 时会阻止开始校对并显示错误
3. 粘贴 README 中的示例文本后，流程能进入进度态并产出结果
4. 结果区的插入/删除高亮仍正常显示
5. “拷贝全文”和“重置”仍可用
6. 移动端宽度下没有明显布局溢出

如果本次任务无法实际联调接口，要在交付说明里明确写出“未做真实 API 验证”。
