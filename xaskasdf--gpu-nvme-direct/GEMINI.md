## Purpose

Research data, paper drafts, figures, and documentation for **gpu-nvme-direct**:
a system enabling consumer GPUs to initiate NVMe I/O autonomously via PCIe MMIO.

**Source code is in `../gpu-nvme-direct`.**
This directory contains only research artifacts, experimental data, and publications.

## Current Status (2026-03-19)

### Results Summary

A **GeForce RTX 3090** on an **AMD B450** platform (Ryzen 5800X) autonomously
drives NVMe controllers via PCIe posted MMIO writes, with zero CPU in the data path.

| NVMe Device | Interface | Sustained BW | Link Utilization |
|-------------|-----------|-------------|-----------------|
| WD SN740 | Gen4 x4 (Gen3 on B450) | **3,350 MB/s** | **99%** of Gen3 x4 |
| WD SN530 | Gen3 x4 | 2,666 MB/s | 78% of Gen3 x4 |

**LLM Inference (Llama 3.1-70B, Q6_K, single RTX 3090):**

| Configuration | tok/s | Source |
|--------------|-------|--------|
| mmap+memcpy (baseline) | 0.028 | measured |
| gpu-nvme-direct NVMe-only (SN530) | 0.04 | measured |
| gpu-nvme-direct NVMe-only (SN740) | 0.06 | measured |
| Tiered VRAM/RAM | 0.20 | measured |
| Tiered + layer skip + Q4_K_M | **0.50** | measured |

### Milestones Achieved

1. GPU-initiated NVMe I/O on consumer hardware (first known demonstration)
2. 3,350 MB/s sustained on SN740 (99% link utilization)
3. ntransformer integration: 0.06 tok/s NVMe-only, 0.50 tok/s tiered
4. Paper v0.2 complete (18 pages, 9 figures, 23 references)
5. Published on naranjositos.tech and GitHub Pages

**Device asymmetry note**: gpu_direct uses SN530 (Gen3, ~3.4 GB/s max),
CPU baselines use 980 PRO (Gen4, ~7 GB/s max). Despite this 2x advantage
for CPU methods, gpu_direct achieves higher throughput due to pipeline parallelism.

SN740 data (3,350 MB/s sustained at 512K/QD=32, 8.6GB transfer) is from
manual `test_large_read` runs, not in the CSV sweep.

## Key Technical Details

### Architecture (Tier 1 — Write-Only P2P)
```
GPU (CUDA kernel, 1 thread)
  1. Write 64-byte SQ entry to host pinned memory
  2. __threadfence_system()
  3. Write doorbell via PTX st.relaxed.mmio.sys to NVMe BAR0
  4. __threadfence_system()
  5. NVMe DMAs data to host pinned buffer (autonomous)
  6. GPU polls CQ phase bit from host pinned memory
  7. GPU writes CQ head doorbell to NVMe BAR0
  CPU is NOT involved in any step.
```

### Why Writes Work on AMD Consumer
AMD B450/Vermeer data fabric forwards **posted** Memory Write TLPs between
root ports but drops **non-posted** Memory Read TLPs (returns CmpltTO).
NVMe doorbells only need writes. CQ polling reads from host DRAM, not BAR0.

### Critical Bugs (7 resolved)
1. `cudaHostRegisterIoMemory` fails — `follow_pfn()` removed in kernel 6.12
2. GPU reads return 0xFFFFFFFF — AMD drops P2P read TLPs
3. Admin commands hang — CQ not page-aligned
4. Pipeline depth >=4 timeouts — out-of-order completions
5. Post-warmup timeouts — queue state reset mismatch
6. GPU reads crash NVMe link — accumulated CmpltTO errors
7. 64-bit MMIO write corruption — non-atomic BAR0 write

## How to Reproduce

### Regenerate figures
```bash
python3 figures/generate_paper_figures.py
```

### Compile paper
```bash
cd drafts && pdflatex gpu-nvme-direct-paper.tex && pdflatex gpu-nvme-direct-paper.tex
```

### Run benchmarks
```bash
cd ../gpu-nvme-direct/build-hw

# VFIO setup
sudo modprobe vfio enable_unsafe_noiommu_mode=1
sudo modprobe vfio-pci
sudo bash scripts/setup_vfio.sh 0000:01:00.0
sudo sh -c 'echo on > /sys/bus/pci/devices/0000:01:00.0/power/control'
sudo setpci -s 0000:01:00.0 0x84.W=0x0008
sudo setpci -s 0000:01:00.0 COMMAND=0x0006

# GPU direct benchmark
sudo ./bench/bench_gpu_direct --device 0000:01:00.0 \
  --block-size 512K --queue-depth 32 --num-ops 200 --pattern seq

# Full sweep
python3 bench/bench_sweep.py --bdf 0000:01:00.0 --device /dev/nvme0n1 \
  --block-sizes 4K 16K 64K 256K 512K --queue-depths 1 4 16 32 \
  --patterns seq --runs 3 --output-dir bench_results
```
