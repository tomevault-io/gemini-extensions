# Working on roost

`roost` is a single Go binary that serves a web UI: terminal + file tree + AI
dashboard + push notifications. It's designed for SSH-tunnel-first deployment
on a personal dev box. Single-user; not multi-tenant.

## Layout

```
cmd/roost/main.go             # CLI entry: serve / setup / hook-info / notify-stop / version
internal/
  auth/        # bcrypt + cookie session middleware
  config/      # TOML loader (~/.config/roost/config.toml)
  session/     # tmux-backed PTY sessions
    log.go        # append-only disk log per session (unbounded scrollback)
    session.go    # Session: owns PTY+log, broadcasts to attached clients
    client.go     # Client: WS conn + buffered out queue (drops slow consumers)
    manager.go    # Manager: GetOrCreate / Rename / Delete / Cwd / GC loop
    handler.go    # GET /ws/terminal[/{id}] WebSocket handler
  fs/          # rooted filesystem API
    fs.go         # path-traversal-safe operations
    handler.go    # /api/fs/* routes (list, download, upload, mkdir, rename, remove)
  ai/          # Claude Code state reader
    ai.go         # JSONL parser, per-model pricing, token aggregation
    handler.go    # /api/ai/usage and /api/ai/sessions
  notify/      # in-process pub/sub
    notify.go     # broadcast + slow-consumer drop policy
  server/      # HTTP wiring
    server.go     # routes + auth middleware + handlers for sessions/notify/SSE
    web/          # embedded SPA (//go:embed all:web)
      *.html      # home / login / index pages
      *.js        # app.js (terminal), fs.js (tree), home.js, cost.js, notify.js
      vendor/     # xterm.js bundle (v5.5, addon-fit, addon-webgl, css)
dist/                          # generated by `make cross` (gitignored)
  systemd/roost.service
  launchd/com.liamsysmind.roost.plist
Makefile                       # build / cross / dev / install
```

## Build & run

```
make           # builds ./roost with version baked in via ldflags
make cross     # linux+darwin × amd64+arm64 → dist/
make dev       # kill old, build, run in foreground
```

Setup once:

```
echo 'PASSWORD' | ./roost setup --password-stdin
```

Generates `~/.config/roost/config.toml` with bcrypt hash, session secret, and
**hook secret** (used by Claude Code Stop hooks to POST notifications).

Run:

```
./roost serve
# Listens on 127.0.0.1:8080. Access via SSH tunnel.
```

## Design philosophy — non-negotiable

**roost is single-user per instance, forever.** If multiple people share a
machine, each runs their own `roost serve` on a different port with their
own `~/.config/roost/` and `~/.local/share/roost/`. UNIX UIDs handle the
isolation; we do not.

Do not add user databases, role-based auth, per-user data partitions, or
"workspace" sharing within a single roost process. Anything that smells
like "let's make roost serve multiple identities" should instead push that
concern up to the deployment layer (a separate hub component, reverse
proxy, etc.) rather than into this codebase.

## Key design decisions

- **tmux is required.** Each session is `tmux new-session -A -s {id}`. Shell
  survives WS disconnect, server restart, idle GC. roost refuses to start if
  tmux isn't in PATH.
- **Tmux is invisible to the user.** `roost-tmux.conf` (written to a temp
  file at startup) sets `status off`, `unbind-key -a`, `history-limit 0`,
  `destroy-unattached off`. No prefix key, no status bar, no nesting warnings.
- **Disk log is the source of truth for scrollback.** Per-session
  `~/.local/share/roost/sessions/{id}.log` is append-only and effectively
  unbounded. `replay_kb` only caps how much of the tail is replayed on attach.
- **Multi-tab broadcast.** Two browser tabs on the same `/s/{id}` see the
  same screen; bytes are fanned out to every attached client.
- **Path traversal is rejected at API boundary.** `fs.API.resolve()` keeps
  every operation under `Root` (default $HOME). Session IDs are validated
  against `[A-Za-z0-9._-]{1,128}` before touching disk.
- **File tree follows cwd via tmux.** The frontend polls
  `GET /api/sessions/{id}/cwd` every 2s, which shells out to
  `tmux display-message -p '#{pane_current_path}'`. No shell integration
  required. Paths outside `[fs] root` are ignored silently.
- **Hook secret is the trapdoor.** `POST /api/notify` accepts the
  `X-Roost-Hook-Secret` header instead of a cookie so local hooks can push
  without a UI session. The hook is a one-line `roost notify-stop`
  invocation (printed by `roost hook-info`); the binary reads Claude
  Code's Stop JSON from stdin, extracts cwd, and POSTs.
- **Stop-hook gating is client-side.** Server fans out every notification;
  notify.js only fires the OS toast when the tab matches all of: same
  cwd as the event, hidden/unfocused, `currentApp === 'claude'`, idle
  longer than `[notify] idle_alert_after` (default 30s). Events without
  a `cwd` (legacy snippets, manual UI POSTs) bypass the gate.

## Code conventions

- **Frontend is vanilla JS** under `internal/server/web/`. No build pipeline.
  Each page loads only the scripts it needs (`app.js`, `fs.js`, `home.js`,
  `cost.js`, `notify.js`). Stay this way until a feature genuinely justifies
  introducing Svelte / a bundler.
- **HTTP methods use Go 1.22 patterns**: `mux.HandleFunc("GET /path", h)` and
  `r.PathValue("id")`. Don't reach for chi unless we need middleware grouping.
- **Errors flow up with `%w` wrapping** and become user-readable HTTP error
  bodies. Don't log-and-continue silently in API handlers.
- **Each new HTTP handler must respect the auth middleware.** Public routes
  go into `isPublicPath()`. Hook-secret-bypass paths are explicit in
  `authMiddleware` and only for `POST /api/notify` so far.
- **No comments stating the obvious.** Comment WHY a non-obvious choice was
  made (e.g. why `=ID` doesn't work with `tmux display-message`).

## Testing

Smoke tests are end-to-end through HTTP:

- Curl + cookie jar (`/tmp/cj`) for HTTP routes
- Tiny Go WebSocket client in `/tmp/wstest*/` for `/ws/terminal/{id}`
- `tmux send-keys` + `display-message` for verifying tmux backend
  independently of WS framing races

No unit-test suite yet. If logic gets gnarly (e.g. ring buffer, OSC parsing
when added), add table-driven tests in the relevant package.

## What's intentionally not done

- **No editor.** roost surfaces files for transfer; users edit inside the
  terminal with whatever editor they brought.
- **No multi-user.** One password, one user. Reverse-proxy if you need more.
- **No Svelte / no build step.** Vanilla JS until it stops fitting.
- **No public-internet listener.** 127.0.0.1 only; SSH tunnel or Tailscale.
- **No tmux status bar / prefix key.** Tmux is plumbing, not UI surface.

## When extending

- Adding an API endpoint → wire it in `internal/server/server.go`'s
  `setupRoutes`; if it needs to bypass auth, add to `isPublicPath` or the
  hook-secret branch in `authMiddleware`. Re-confirm path-traversal containment
  if filesystem-adjacent.
- Adding a frontend page → new HTML in `internal/server/web/`, register a
  `GET /your-page` handler that serves it. Add to the static handler if it
  needs JS/CSS siblings.
- Adding a config knob → extend `internal/config/config.go`'s struct + the
  matching default in `Load()` / `runSetup()`. Document in README's config
  section. Don't add CLI flags for the same setting unless overriding per-run
  makes sense.
