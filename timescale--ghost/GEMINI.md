# Ghost CLI Development Guide

## Repository Structure

- **`cmd/`** - Binary entry points. Contains `ghost/main.go` (the main CLI binary, which sets up context/signal handling and delegates to the internal command infrastructure), `npm-publisher/` (a CI tool that generates and publishes npm packages for each platform), and `generate-docs/` (generates Markdown CLI reference docs to `docs/cli/`).
- **`internal/`** - All core application logic (non-public Go packages).
  - **`internal/cmd/`** - Cobra command implementations for all CLI commands (create, fork, list, delete, pause, resume, connect, psql, sql, schema, logs, password, rename, status, feedback, api-key, login, logout, config, mcp, version, upgrade, completion, payment). Each command lives in its own file, named to match the command in snake_case (e.g. `ghost payment list` → `payment_list.go`). Helper files like `completion.go`, `errors.go`, and `logger.go` contain shared utilities. Commands that are not yet ready for public release can be gated behind the `GHOST_EXPERIMENTAL` env var (see `internal/common/app.go`'s `App.Experimental` field).
  - **`internal/api/`** - API client layer. Includes an OpenAPI-generated REST client (`client.go`, `types.go`), shared HTTP client singleton, and request/response types. **Do not edit `client.go` or `types.go` by hand** — they are generated from `openapi.yaml` (see [Code Generation](#code-generation)). The `mock/` subdirectory contains a generated mock of `ClientWithResponsesInterface` for use in tests.
  - **`internal/config/`** - Configuration management. Handles config file loading (via Viper), credential storage (keyring with file fallback), and version checking.
  - **`internal/common/`** - Shared business logic used across commands and MCP tools. Includes API client initialization, database connection/schema/query utilities, error handling with exit codes, and version update checks.
  - **`internal/mcp/`** - Model Context Protocol (MCP) server. Exposes Ghost database operations as MCP tools for AI/LLM integration, plus a documentation search proxy. Each MCP tool lives in its own file, named to match the tool (e.g. `ghost_status` → `status.go`). Helper files like `util.go`, `errors.go`, and `proxy.go` contain shared utilities.
  - **`internal/analytics/`** - Analytics event tracking with sensitive data redaction for flags, positional arguments, and MCP inputs.
  - **`internal/util/`** - General utilities: type conversion, duration formatting, path helpers, context-aware stdin reading, JSON/YAML serialization, and terminal detection.
- **`docs/`** - Documentation. `docs/cli/` contains generated Markdown CLI reference docs (produced by `cmd/generate-docs`).
- **`scripts/`** - Build and installation scripts (install.sh, install.ps1, completions generation).
- **`openapi.yaml`** - OpenAPI spec used to generate the API client. Should be kept in sync with the canonical spec in the `ghost-api` repo (see [Code Generation](#code-generation)).
- **`.github/`** - GitHub Actions CI/CD workflows for testing and releases.

## Build & Test

Always run the following after editing Go code:

- Build:
  - `go install ./...`
- Format:
  - `go fmt ./...`
- Fix:
  - `go mod tidy`
  - `go fix -omitzero=false ./...`
- Lint:
  - `go vet ./...`
  - `go tool staticcheck ./...`
- Test:
  - `go test ./...`

## Testing

CLI command tests live in `internal/cmd/` alongside the command files (e.g. `list_test.go` tests `list.go`). Shared test infrastructure lives in `main_test.go`.

### How it works

Tests execute commands through Cobra's `Execute()` with a mock API client injected via `App.SetClientFactory`. The `runCommand` helper in `main_test.go` handles all the wiring: it builds the root command, injects the mock, sets up a temp config directory, disables analytics/version checks, stubs `common.OpenBrowser` (to prevent real browser opens), captures stdout/stderr, and returns the result.

The mock is generated by `mockgen` from the `ClientWithResponsesInterface` in `internal/api/client.go`. The generated mock lives at `internal/api/mock/mock_client.go` and is regenerated by `go generate ./...`. All production code uses `api.ClientWithResponsesInterface` (not the concrete `*api.ClientWithResponses`) so that the mock can be substituted.

### Writing a new command test

- One `Test<Command>` function per command file, using table-driven subtests. Test files must be 1:1 with source files (e.g. `config_set.go` → `config_set_test.go`).
- Use the shared `cmdTest` struct for test cases and `runCmdTests(t, tests)` to execute them. These are defined in `main_test.go` and handle the standard assertion pattern (check `wantErr`, then `wantStdout`, then `wantStderr`).
- Cover all code paths: success with each output format (text, JSON, YAML), not-logged-in, API errors, network errors, nil response bodies, flag variations, and aliases.
- Order test cases to follow the code execution flow of the function under test. Error paths that occur early in the function (e.g. auth failure, network error, bad status code, nil body) should come before success-path tests. For commands with multi-step flows (e.g. fetch then delete), error paths on the second step should come before success-path tests. This makes the test table read like a walkthrough of the function's logic.
- Use `withClientError` to simulate authentication failures, `withStdin` for commands that read from stdin, `withEnv` for environment variable overrides, and `withOpenBrowser` to override the browser stub (e.g. to simulate a successful browser open).
- Extract helper functions for mock setup that is repeated across test cases with minor variations. The helper should accept the varying parameter. For example, if several test cases mock `ForkDatabaseWithResponse` with different fork names, create a `setupFork(name string)` helper rather than inlining the full mock setup in each case.

### Assertions

Use exact output matching via `assertOutput`, which uses `cmp.Diff` for clear diffs on failure. Do not use substring matching — it's too weak to catch formatting regressions. Use backtick strings for multi-line expected output. **Trailing whitespace caveat:** The Write tool strips trailing spaces from lines, but some expected output (e.g. table-rendered text) has significant trailing whitespace. After writing or editing test files with backtick strings that need trailing spaces, use `sed` to add them back (e.g. `sed -i '48s/SIZE$/SIZE  /' file_test.go`).

### Mock expectations

Set up mock expectations using `gomock`'s `EXPECT()` API. Use `validCtx` for the context parameter. Return the appropriate response struct with `HTTPResponse` set to `httpResponse(statusCode)` and the relevant `JSON200`/`JSON201`/`JSONDefault` field populated. For network errors, return `(nil, error)`.

Validate all parameters passed to mock expectations — do not use `gomock.Any()` as a shortcut when the expected value is known. For `context.Context` parameters, use `validCtx` (defined in `main_test.go`) which verifies the context is non-nil. For request body parameters (`CreateDatabaseRequest`, `ForkDatabaseRequest`, etc.), always specify the exact expected struct. If the parameter value is truly non-deterministic (e.g. a randomly generated password with `--generate`), `gomock.Any()` is acceptable — add a comment explaining why.

### What's testable

- **API-only commands** (list, create, delete, fork, resume, rename, logs, status, etc.) — fully testable with mock.
- **Login command** — testable using a fake `httptest.Server` for the OAuth token exchange, `withOpenBrowser` to simulate the browser callback, and `withNewGhostClient` to inject a mock for post-login API calls. See `login_test.go` for the pattern.
- **Database-connecting commands** (sql, schema, psql) — test error paths only (auth errors, DB not ready, missing password). Success paths require a real database.
- **Config commands** — testable without a mock (pass `nil` for `setupMock`).

### Keeping the mock up to date

When the OpenAPI spec changes and the client is regenerated, the mock must also be regenerated. Running `go generate ./...` handles both. If the mock is stale, `go vet` will report that the mock doesn't implement the interface.

## Code Generation

The API client and types in `internal/api/` are generated from `openapi.yaml` using `oapi-codegen`. The mock client in `internal/api/mock/` is generated by `mockgen`. **Never edit `internal/api/client.go`, `internal/api/types.go`, or `internal/api/mock/mock_client.go` by hand.** To regenerate after editing the OpenAPI spec:

```bash
go generate ./...
```

This regenerates the types, client, and mock in the correct order (the `//go:generate` directives in `generate.go` are ordered so the mock is always built from the freshly generated client).

The typical workflow for API changes is:

1. Edit the canonical `openapi.yaml` in the `ghost-api` repo first.
2. Copy the updated spec into this repo's root `openapi.yaml`.
3. Run `go generate ./...` to regenerate the client, types, and mock.

## Database References (`ref` vs `id`)

Database-scoped API endpoints accept either an ID or a name, so the path parameter is `database_ref` (generated as `api.DatabaseRef`). Use `databaseRef` for variables holding user input that could be either form; reserve `databaseID` for values resolved from `database.Id` on an API response.

## Command Patterns

All cobra commands must:

1. Set `SilenceUsage: true`
2. Set `ValidArgsFunction` or `ValidArgs` for shell completions. Use `ValidArgsFunction` for dynamic completions (e.g. database IDs) and `ValidArgs` for static lists. Use `cobra.NoFileCompletions` if no completions are needed. The important thing is that one of these is always set.
3. Use `RunE` (not `Run`) for error handling
4. Use `cmd.Print*` functions for output (not `fmt.Print*` or `fmt.Fprint*`)
5. Use `cmd.InOrStdin()`, `cmd.OutOrStdout()`, `cmd.ErrOrStderr()` for any direct references to stdin/stdout/stderr (do not use os.Stdin/os.Stdout/os.Stderr directly)

Consider adding common aliases for commands (e.g. `ls` for `list`, `rm` for `delete`) to match user expectations from other CLI tools.

## Case Insensitivity

Commands, aliases, and flags are matched case-insensitively (configured in `root.go`). Always declare them in lowercase — that's the canonical form. Two names that differ only in case are a collision and must be avoided.

## Logging

The `slog` logger (configured in `internal/cmd/logger.go`) is reserved for the MCP server only, since that's the only context where the CLI acts as a long-running backend process that needs structured logging. Normal CLI commands should never emit log output — it looks bad and interferes with the user-facing output.

## Output Streams

CLI commands should write their "primary output" (i.e. the "result" of the command) to **stdout** (`cmd.Print*`) by default. This is true whether the output is structured (e.g. an ASCII table, JSON, or YAML), or simple text indicating what happened. On the other hand, the following should go to **stderr** (`cmd.PrintErr*`):

1. **Errors and warnings** — e.g. `cmd.PrintErrf("Warning: ...")`.
2. **Interactive terminal UI** — confirmation prompts (`[y/N]`), password prompts, stdin-reading prompts (`Enter your query (press Ctrl+D when done):`), and live-updating elements like spinners and progress bars. These must go to stderr so they remain visible when stdout is redirected and don't pollute piped output.
3. **Anything that isn't the data payload itself.** When a command writes structured output (an ASCII table, JSON, YAML, or data that a user might pipe/redirect), any informational or progress messages must go to stderr to avoid polluting the data stream.

In the **text-output path** (the default, when `--json`/`--yaml` are not used), the entire output is considered "primary output" and goes to stdout. For example, `ghost resume` prints `"Resuming 'mydb'...\nConnection: postgres://..."` — even though "Resuming..." looks like a progress message, it's the primary output of the command and belongs on stdout. There's no need to split informational text from data in this path.

## Color Output

Use the `lipgloss` package (`charm.land/lipgloss/v2`) for colored/styled terminal output. The `--color` config option globally enables or disables color by configuring the command's stdout/stderr writers in `root.go` to strip ANSI sequences when color is off. For this to work, all output must go through the command's writers (`cmd.Print*`, `cmd.OutOrStdout()`, `cmd.ErrOrStderr()`) — never write directly to `os.Stdout` or `os.Stderr`.

**TermWriter gotcha:** The command's stdout/stderr writers are wrapped in `util.TermWriter`, which implements `io.Writer` but is **not** an `*os.File`. This means `cmd.OutOrStdout()` and `cmd.ErrOrStderr()` cannot be used where an `*os.File` is required — for example, passing them to `exec.Cmd.Stdout`/`Stderr` causes `os/exec` to create a pipe instead of passing the file descriptor directly, which breaks TTY detection in child processes (see `psql.go` for a concrete example). Use `util.TryUnwrapFile()` to extract the underlying `*os.File` when needed. Note that this bypasses color stripping, which is acceptable when the caller (e.g. a child process) manages its own terminal output.

## Interactive Terminal UIs

Use BubbleTea (`charm.land/bubbletea/v2`) for interactive terminal menus and prompts. When creating a `tea.NewProgram`, always pass:

- `tea.WithInput(cmd.InOrStdin())` — or `tea.WithInput(nil)` if the program doesn't read input (e.g. a spinner). Without this, BubbleTea puts the terminal in raw mode where Ctrl+C is sent as a key press instead of triggering a signal.
- `tea.WithOutput(cmd.OutOrStdout())` (or the appropriate writer) — so that color stripping applies.
- `tea.WithoutSignalHandler()` — primarily useful when the BubbleTea program is **not reading input** (e.g. a spinner or progress indicator). In this case, BubbleTea can't handle Ctrl+C via key presses since it's not in raw mode, so signals should propagate normally and be handled via `context.Context` cancellation as usual.

## Configuration

Configuration is managed via Viper. The config file is stored in the config directory (default `~/.config/ghost/`). There are two categories of config values:

- **Public**: User-facing options (e.g. `analytics`, `color`, `read_only`, `version_check`). Shown by `ghost config` and included in shell completion for `ghost config set`.
- **Private/Internal**: Internal settings (e.g. `api_url`, `docs_mcp_url`, `releases_url`). Hidden from `ghost config` by default (use `--all` to see them) and not included in shell completions, but still settable via `ghost config set`.

Many public config options have corresponding global CLI flags. All config values can be provided as `GHOST_*` environment variables (e.g. `GHOST_COLOR=false`). Precedence is: **flags > env vars > config file > defaults**.

The config directory can be overridden via `--config-dir` flag or `GHOST_CONFIG_DIR` env var. This controls where the config file (and credentials file, if the system keyring is unavailable) are stored.

## Environment Variables

- **`GHOST_CONFIG_DIR`** — Override the config directory (default `~/.config/ghost/`).
- **`GHOST_EXPERIMENTAL`** — Set to `true` to enable experimental CLI commands and features that haven't been publicly released yet.
- **`GHOST_*`** — Any public or private config value can be set via `GHOST_<KEY>` env vars (e.g. `GHOST_COLOR`, `GHOST_API_URL`).

## Config & API Client

All CLI commands and MCP tool handlers receive a `*common.App` which holds the config and API client. The `App` is loaded once in `PersistentPreRunE` (see `root.go`) and shared across all command handlers. MCP tool handlers call `s.app.Load(ctx)` on each invocation to refresh credentials.

Use the appropriate getter depending on what you need:

```go
// Most commands: get the API client and project ID
client, projectID, err := app.GetClient()

// When you also need config (e.g. for connection strings)
cfg, client, projectID, err := app.GetAll()

// When you only need config (e.g. config commands, logout)
cfg := app.GetConfig()

// Best-effort access (doesn't error if client is unavailable).
// Only for rare cases where the API call is optional (e.g. analytics tracking).
cfg, client, projectID := app.TryGetAll()
```

## Error Handling

- Handle ALL errors (even if just by outputting a warning to the console). Never ignore errors or assign them to `_`
- Return errors from `RunE`; don't call `os.Exit()` directly
- Use `common.ExitWithErrorFromStatusCode()` for API errors
- Use `common.ExitWithCode()` for specific exit codes (defined in `internal/common/errors.go`)
- Error messages should start with a lowercase letter
- Use `errors.New()` for static error strings; only use `fmt.Errorf()` when you need format verbs (`%w`, `%v`, `%s`, etc.)
- **Handle `Close()` errors (and other deferred cleanup errors) explicitly.** Use `defer x.Close()` for best-effort cleanup on error/panic paths, *and also* call `x.Close()` explicitly at the end of the happy path and return its error. That way, if an earlier error occurred it's returned first; if the only error is from close, it still gets returned. This relies on `Close()` being idempotent, which is typical.

## Password Errors

Commands and MCP tools that need a database password should call `common.GetPassword` and handle errors properly rather than silently falling back to an empty password. Check for `common.ErrPasswordNotFound` to provide actionable guidance:

- **CLI commands**: Recommend `ghost password <id>` to reset the password, or adding the entry to `~/.pgpass` manually.
- **MCP tools**: Recommend `ghost_password` to reset the password, or adding the entry to `~/.pgpass` manually.

Commands that only display connection strings (e.g. `ghost connect`) can treat missing passwords as non-fatal warnings since the user may not need the password in the output.

## Database Readiness Checks

Any CLI command or MCP tool that requires the database to be up and running must call `common.CheckReady(database)` before proceeding. This returns `common.ErrPaused` if the database is paused/pausing, or `common.ErrNotReady` if it is queued, configuring, or resuming. This applies to operations that connect to the database (sql, schema, psql), fork it, or change server-side settings like the password.

Use the `handleDatabaseError` helpers in `internal/cmd/errors.go` and `internal/mcp/errors.go` to translate these errors into user-friendly messages. These helpers also handle `common.ErrPasswordNotFound`.

Operations that only call the Ghost API (create, delete, list, pause, resume, logs, connect) do not need this check.

## Output Formatting

Commands with structured output should support `--json` and `--yaml` flags, made mutually exclusive:

```go
cmd.Flags().BoolVar(&jsonOutput, "json", false, "Output in JSON format")
cmd.Flags().BoolVar(&yamlOutput, "yaml", false, "Output in YAML format")
cmd.MarkFlagsMutuallyExclusive("json", "yaml")
```

Use `util.SerializeToJSON(cmd.OutOrStdout(), output)` and `util.SerializeToYAML(cmd.OutOrStdout(), output)` for serialization, with a default text output path (typically using `tablewriter` for lists or `cmd.Printf` for single items).

**Do not add `yaml:` struct tags to output types.** `SerializeToYAML` works by encoding to JSON first, then converting to YAML. This ensures consistent serialization even for types outside our control that only have `json:` tags. Only `json:` tags are needed.

## Waiting for Long-Running Operations

Commands should never block waiting for long-running operations (like a database starting up) by default. Provide an optional `--wait` flag (or `wait` MCP tool parameter) that enables polling until the operation completes. The pattern:

1. Perform the API call and output results immediately.
2. If `--wait` is not set, return.
3. If `--wait` is set, call `common.WaitForDatabaseWithProgress()` (shows a spinner in TTY, falls back to plain polling otherwise). Progress is written to stderr so it doesn't interfere with JSON/YAML output on stdout.

## Confirmation Prompts

Destructive CLI commands (e.g. `delete`) should prompt for confirmation unless a `--confirm` flag is provided. Use `util.ReadLine()` for the prompt (it respects context cancellation). Default to cancelling if the user doesn't type `y` or `yes`.

All interactive prompts (confirmation prompts, password prompts, interactive menus, etc.) must be gated with `util.IsTerminal(cmd.InOrStdin())`. If stdin is not a terminal, return an error directing the user to the non-interactive alternative (e.g. `--confirm` flag, positional argument). This prevents commands from hanging or reading garbage when stdin is piped or redirected. `util.IsTerminal` is a function variable that can be overridden in tests via `withIsTerminal(true)` to simulate a terminal stdin.

## MCP Tool Patterns

Each MCP tool in `internal/mcp/` follows a consistent structure:

- **Tool definition**: `newXxxTool() *mcp.Tool` — sets name (`ghost_xxx`), description, input/output schemas, and annotations (`ReadOnlyHint`, `DestructiveHint`, `IdempotentHint`).
- **Handler**: `(s *Server) handleXxx(ctx context.Context, req *mcp.CallToolRequest, input XxxInput) (*mcp.CallToolResult, XxxOutput, error)` — returns `nil, output, nil` on success.
- **Input/Output types**: Structs with `json` tags and a `Schema() *jsonschema.Schema` method. Use the schema property helpers in `internal/mcp/util.go` (e.g. `databaseIDInputProperties`, `waitInputProperties`) for consistent descriptions and validation. Input and output helpers are separated — input helpers may set `Pattern`, `Enum`, `Default`, etc., while output helpers typically only set `Description`. Don't put example values inline in `Description` strings (e.g. `"Database status (running, paused, etc.)"`). Instead, use the `Examples` field for illustrative values (e.g. `Examples = []any{"running", "paused"}`) or the `Enum` field if the set of values is fixed and exhaustive.
- **Registration**: `mcp.AddTool(s.mcpServer, newXxxTool(), s.handleXxx)` in `server.go`'s `registerTools()` method.
- **Read-only guard**: Destructive/write tools must call `checkReadOnly(cfg)` after loading config, to respect the read-only config option.
- **JSON Schema defaults are applied by the SDK**: The Go MCP SDK (`go-sdk`) calls `resolved.ApplyDefaults()` on the raw JSON input *before* unmarshaling into the typed input struct. This means if a schema sets `Default` on a field (e.g. `schema.Properties["size"].Default = json.RawMessage("\"1x\"")`), the handler is guaranteed to receive that value when the caller omits the field. Handlers can rely on this and do not need to check for zero values or apply defaults themselves.

## Context Handling

ALWAYS pass the command context (`cmd.Context()`) down into any functions that perform long-running operations that may need to be cancelled or time out. This is standard Go best practice and an important rule we follow throughout this codebase. It ensures that commands exit promptly upon receiving a Ctrl+C, SIGINT, or SIGTERM.

## Reading from Stdin

When reading user input from stdin, always use `util.ReadLine` or `util.ReadPassword` instead of direct reads (e.g., `bufio.NewReader`). These functions respect context cancellation, which is important for handling Ctrl+C gracefully. `ReadPassword` also saves and restores terminal state, preventing the terminal from being left in an unusable state if the user cancels during password entry.

```go
// For regular line input
line, err := util.ReadLine(cmd.Context(), cmd.InOrStdin())

// For password input (hides typed characters)
password, err := util.ReadPassword(cmd.Context(), cmd.InOrStdin())
```

## HTTP Requests

All outgoing HTTP requests must use the singleton `api.HTTPClient` defined in `internal/api/client_util.go`. This client has a built-in 30-second request timeout. The API client created by `api.NewGhostClient` uses it automatically. If you need to make HTTP requests outside of the API client, use `api.HTTPClient` directly rather than `http.DefaultClient` or creating a new `http.Client`.

If a request needs a shorter timeout, set one via the context:
```go
ctx, cancel := context.WithTimeout(cmd.Context(), 5*time.Second)
defer cancel()
```

## Go Style

- **Pre-allocate slices when the length is known.** Use `make([]T, len(source))` with index assignment rather than `make([]T, 0)` with `append`, which may resize the backing array multiple times.
- **Construct output/result types at the end.** Build intermediate data (slices, computed values) first, then construct and return the struct as the final step. Avoid creating a struct early and mutating it over the course of the function.
- **Use `new()` for pointer-to-expression values.** As of Go 1.26, `new()` accepts an expression (not just a type) and returns a pointer to a new variable initialized with that expression's value. Use `new("value")`, `new(42)`, or `new(someFunc())` instead of declaring a variable first. This is cleaner and avoids polluting the scope with single-use variables. For example: `api.CreateDatabaseRequest{Name: new("mydb")}` instead of `name := "mydb"; api.CreateDatabaseRequest{Name: &name}`. This also works with type conversions: `new(api.DatabaseSize("2x"))`.

## API Response Handling

When working with API responses from the generated client, always nil-check response bodies. After checking the status code, check that the parsed response body (e.g. `resp.JSON200`, `resp.JSON202`) is not nil before accessing it. A nil body indicates a bug and should return `errors.New("empty response from API")`.

## Analytics & Sensitive Data

Sensitive values (passwords, keys, queries, etc.) must never be sent in analytics events. There are three mechanisms to be aware of:

- **Flags and MCP inputs**: The `ignore` list in `internal/analytics/analytics.go` filters out sensitive flag names and MCP tool parameter names. Add new entries there when introducing flags or MCP parameters that carry sensitive data.
- **Positional arguments**: The `analytics.Args` function in `internal/analytics/analytics.go` redacts sensitive positional arguments for specific commands (e.g. the password in `ghost password`). Update it when adding commands that accept sensitive positional arguments.
- **Call site**: Commands are tracked in `wrapCommands` in `internal/cmd/root.go`. That function uses `analytics.Args` (not `analytics.Property`) to record args so that redaction is applied.

When adding a new command or MCP tool that handles secrets, credentials, or user-supplied queries, make sure it is covered by one of these mechanisms.

## Deployment

Creating a release automatically triggers a build and publish via GoReleaser. To create a new release, follow these steps:

1. Make sure you are on `main` and up to date (`git pull`).
2. Find the latest existing tag to determine the current version:
   ```bash
   git tag --sort=-v:refname | head -5
   ```
3. Ask the user whether this is a **major**, **minor**, or **patch** bump. Do not assume.
4. Compute the next semver version based on the current latest tag and the bump type.
5. Create the release:
   ```bash
   gh release create v<NEXT_VERSION> --generate-notes --latest
   ```
6. Get the workflow run URL and open it in the browser:
   ```bash
   open $(gh run list -w Release -b v<NEXT_VERSION> -L 1 --json url -q '.[0].url')
   ```
7. If the user asks to watch/wait for the workflow to complete:
   ```bash
   gh run watch $(gh run list -w Release -b v<NEXT_VERSION> -L 1 --json databaseId -q '.[0].databaseId')
   ```

The tag push triggers the `Release` workflow, which runs GoReleaser to build binaries, Docker images, and Linux packages, then publishes them to GitHub Releases, S3, GHCR, PackageCloud, and the [Homebrew tap](https://github.com/timescale/homebrew-tap). No manual deploy step is needed.

### CLI Reference Docs

The `docs/cli/` directory contains generated Markdown reference documentation for every CLI command. These are produced by `cmd/generate-docs`, which uses Cobra's `doc` package to walk the command tree and emit one file per command.

Regenerate the docs whenever a CLI command is added, updated, or removed. Run from the repository root, or pass `-out` explicitly — the default output path is `./docs/cli` relative to the working directory, and `-clean` will delete that directory:

```bash
go run ./cmd/generate-docs -clean
```

Flags:

- `-out` — Output directory (default `./docs/cli`).
- `-format` — Output format: `markdown`, `man`, `rest`, or `yaml` (default `markdown`).
- `-frontmatter` — Prepend YAML frontmatter to Markdown files (default `true`).
- `-clean` — Remove the output directory before generating (default `false`).
- `-experimental` — Include experimental commands in the output (default `false`). This sets `GHOST_EXPERIMENTAL` for the run, overriding any inherited value, so released-only docs are produced by default.
