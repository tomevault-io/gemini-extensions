# 全局开发规范 — 字节范（ByteStyle）

> 以字节范核心价值观驱动高效协作，适用于所有项目。项目级 CLAUDE.md 可覆盖本文件中的规则。

### 最高原则：做好比做完重要

> **此原则不可被任何项目级规则覆盖。**

- **质量优先于速度**：做对一次，好过返工三次。"赶紧出结果"不是跳过思考的理由
- **先想后做**：收到需求后的第一反应不是动手写代码，而是确认自己理解了问题。动手前花 1 分钟想清楚，省下 10 分钟返工
- **慢就是快**：该走的流程不跳过、该读的代码不跳过、该问的问题不跳过。每一次"省事"的跳过，都在积累后面更大的返工成本

## 一、工作原则

### 始终创业 (Always Day 1)
- **最简方案优先**：用最少的代码解决当前问题，不为假想需求增加复杂度
- **MVP 驱动**：新功能先跑通最小可用版本，验证方向后再迭代完善
- **快速反馈**：每次改动后立即验证，保持短反馈循环
- **敢于丢弃**：如果发现方向错了，果断推翻重来，沉没成本不是继续的理由

### 坦诚清晰 (Be Candid and Clear)
- **commit message**：`<type>(<scope>): <中文简要描述>`
  - type: feat / fix / refactor / docs / test / chore / perf
- **代码即文档**：命名自解释，注释说明"为什么"而非"做了什么"
- **主动暴露问题**：遇到不确定的方案直接提问，不猜测不硬上
- **有意义的错误信息**：错误提示要能帮助定位问题，不吞异常

### 求真务实 (Seek Truth and Be Pragmatic)
> 回归第一性原理：从问题的基本事实出发推导方案，不从"上次怎么做的"或"别人怎么做的"出发类比。每一个"理所当然"都值得追问一句"为什么"。
- **先读后改**：修改代码前必须先理解现有实现，搞清楚上下游影响
- **根因驱动**：bug 修复先复现、定位根因，再写修复代码，不做表面补丁
- **数据说话**：性能优化基于实际瓶颈，不做无依据的"优化"
- **保持整洁**：删除无用代码，不留注释掉的废弃代码

### 多元兼容 (Be Open and Inclusive)
- **尊重现有风格**：修改代码时保持与项目已有模式一致，不擅自按个人偏好重构
- **兼容性意识**：考虑不同运行环境和用户场景，不只为自己的环境写代码
- **依赖克制**：引入新依赖前先问"项目里有没有现成的"

### 敢为极致 (Be Courageous and Aim for the Highest)
- **安全第一**：防范 OWASP Top 10，边界输入必须校验，外部数据不可信
- **敏感信息零容忍**：密钥、token、密码禁止硬编码或提交
- **测试护航**：核心业务逻辑必须有测试覆盖，验证脚本必须沉淀为可回归的自动化用例
- **验证才算完成**：功能开发完成后必须运行自动化验证，通过后才能进入收尾流程
- **不将就**：发现代码有隐患时当场提出，不留"以后再说"的隐患

### 共同成长 (Grow Together)
- **知识沉淀**：关键决策和经验**必须记录到 `docs/` 目录并随 git 提交**，memory 仅作为跨会话辅助索引。文档是团队共享的持久化真相源，memory 是个人的临时记忆
- **小步提交**：大型改动拆分为小的、可独立审查的 PR
- **复盘驱动成长**：每次大任务完成后复盘，从实战中沉淀经验
- **测试即资产**：开发过程中编写的验证脚本必须沉淀为回归测试用例，不允许用完即弃

## 二、通用技术规范

### 语言与风格
- 代码注释和文档使用中文，变量名和代码使用英文
- 遵循项目已有的 lint / formatter 配置，无配置时遵循语言社区最佳实践
- 修改代码时保持与项目已有风格一致，不擅自重构
- 文件末尾保留空行，UTF-8 编码

### Git
- 分支命名：`feat/xxx`、`fix/xxx`、`refactor/xxx`
- 主分支通过 PR 合入，不直接推送
- 合并前确保 CI 通过

### 依赖与环境
- `.env` 文件不入版本控制
- 锁文件（lock file）必须提交
- 使用项目指定的包管理器
- 优先复用项目已有的库，引入新依赖前说明必要性

### 项目文档规范
项目使用 `docs/` 目录统一管理文档，流转关系：

```
backlog（想法）→ prd（产品需求）→ designs（UI 交互）→ spec（技术方案）→ plan（拆步骤）→ tests（验证 + 回归）→ decisions（复盘）
```

> 不是每个环节都必须有产出，没有产物的环节跳过即可。比如纯后端任务无 designs，小需求无 prd。

| 目录 | 级别 | 用途 | 命名格式 |
|------|------|------|---------|
| `docs/backlog/` | **必须** | 需求池 & 待办事项（想法、需求、待办，按优先级排列） | `YYYY-MM-DD-<主题>.md` |
| `docs/prds/` | **推荐** | 产品需求文档（功能描述、用户故事、验收标准），有产品需求时创建 | `YYYY-MM-DD-<功能名>.md` |
| `docs/designs/` | **推荐** | UI 交互（原型图、交互说明、设计标注、截图），有 UI 产出时创建 | `YYYY-MM-DD-<功能名>/` |
| `docs/specs/` | **必须** | 方案设计（背景 + 调研 + 决策 + 技术方案） | `YYYY-MM-DD-<模块名>-<主题>.md` |
| `docs/plans/` | **必须** | 开发计划（任务拆分 + 进度跟踪） | `YYYY-MM-DD-<模块名>-plan.md` |
| `docs/tests/` | **必须** | 测试用例（功能验证 + 回归测试用例，开发验证脚本沉淀于此） | `<模块名>-<功能>.md` |
| `docs/decisions/` | **必须** | 复盘与经验教训（踩坑记录、事后复盘、可复用的经验沉淀） | `YYYY-MM-DD-<主题>.md` |
| `docs/architecture/` | **推荐** | 系统架构（按模块维护，INDEX.md 为全局导航），模块 ≥3 个时应创建 | `<模块名>.md` + `INDEX.md` |
| `docs/user-guide/` | **推荐** | 用户手册（按模块/功能维护，INDEX.md 为目录导航），有外部用户时应创建 | `<功能名>.md` + `INDEX.md` |
| `docs/guides/` | **推荐** | 开发指南（环境搭建、上手文档、开发约定），多人协作项目应创建 | `<主题>.md` |
| `docs/changelog/` | 建议 | 版本变更记录（重大发布、breaking changes），有外部用户时创建 | `YYYY-MM-DD-v<版本号>.md` |
| `docs/runbooks/` | 建议 | 运维手册（部署流程、故障处理 SOP、监控告警），有线上服务时创建 | `<场景>.md` |
| `docs/assets/` | **必须** | 文档资源文件（图表 PNG、截图、示意图等），文档中引用的图片统一存放 | `<图表类型>-<描述>.png` |

> **级别说明**：**必须** = 所有项目必建；**推荐** = 多人协作项目应建；建议 = 按项目性质自行判断

文档与代码同步更新，中文编写，文件名英文小写+连字符。文档必须随代码一起 git 提交，不允许只存 memory 不入库。

**backlog 与 plan 的关系**：backlog 是需求的全生命周期管理（收集 → 排优先级 → 跟踪状态），plan 是单个需求的执行进度管理（拆子任务 → 跟踪每步完成情况）。backlog 管"做不做"，plan 管"做到哪了"。

**backlog 状态管理**：
```yaml
---
priority: P1        # P0（紧急）| P1（重要）| P2（普通）| P3（低优）
status: open        # open | in-progress | done | dropped
spec: docs/specs/xxx.md   # 关联方案
plan: docs/plans/xxx.md   # 关联计划
---
```

**backlog 索引（INDEX.md）**：

`docs/backlog/INDEX.md` 是 backlog 的快速索引文件，AI 读取此文件即可掌握全部需求概况，无需逐个打开 backlog 文件。

- **更新时机**：仅在**新建需求**和**需求状态变更**（open → done / dropped）时更新
- **格式**：按优先级分组，每条一行

```markdown
# Backlog Index

## P0 — 紧急
<!-- 当前无 P0 -->

## P1 — 重要
- [ ] [Session 改造](2026-03-23-auth-session.md) — 合规要求，Q1 deadline
- [x] [日志规范化](2026-03-10-logging.md) — 统一日志格式

## P2 — 普通
- [ ] [首页性能优化](2026-03-15-homepage-perf.md) — LCP > 3s

## P3 — 低优
- [ ] [管理后台暗色模式](2026-03-20-admin-dark-mode.md) — 用户反馈
```

- `[ ]` = open，`[x]` = done，`~~删除线~~` = dropped
- 每条格式：`- [ ] [标题](文件名) — 一句话描述`

**开发计划（plan）规范**：

plan 是 spec 的执行层：spec 定义"做什么、怎么做"，plan 定义"拆成几步、每步具体干什么、当前做到哪了"。进度跟踪在 plan 中完成，不在 backlog 中重复跟踪。

**plan 必备内容**：

| 内容 | 说明 |
|------|------|
| **关联方案** | 指向对应的 spec 文件 |
| **子任务列表** | 用 checklist 拆分，每个子任务有明确的交付物 |
| **子任务描述** | 每个子任务说清楚：具体做什么、涉及哪些文件/模块、验收标准 |
| **进度状态** | 实时更新 checklist，反映当前进展 |

```markdown
# 示例：docs/plans/2026-03-23-auth-session-plan.md

## 关联
- 方案：docs/specs/2026-03-23-auth-session-design.md

## 子任务

- [x] 1. Redis 连接模块
  - 做什么：封装 Redis 客户端，支持连接池和健康检查
  - 涉及：`src/lib/redis.ts`（新建）
  - 验收：连接成功、健康检查通过

- [ ] 2. Session 中间件改造
  - 做什么：替换 Express 默认 session 为 Redis-backed session
  - 涉及：`src/middleware/session.ts`（改造）
  - 验收：登录后 session 存入 Redis，注销后 key 删除

- [ ] 3. 回归测试
  - 做什么：编写登录/注销/过期场景的自动化测试
  - 涉及：`docs/tests/auth-session.md`，`tests/auth.test.ts`
  - 验收：全部用例通过
```

**方案设计（spec）规范**：

每份方案必须完整交代**背景 → 调研 → 验证 → 决策**全链路，不允许只出方案不写来龙去脉。

**spec 必备章节**：

| 章节 | 必须回答的问题 |
|------|--------------|
| **背景与动机** | 为什么要做这件事？解决什么问题？谁提出的？有什么约束条件？ |
| **现状分析** | 当前是怎么做的？痛点在哪？不改会怎样？ |
| **调研与备选方案** | 调研了哪些方案？每个方案的验证过程和结果？ |
| **决策与取舍** | 选了哪个？为什么选它不选别的？放弃了什么？遗留什么风险？ |
| **技术方案** | 具体怎么实现？（这才是"方案"本身） |

```markdown
# 示例：docs/specs/2026-03-23-auth-session-design.md

## 背景与动机
法务团队在 2026-03 合规审查中指出：当前 cookie-based session 无法满足
"用户注销后 token 立即失效"的合规要求。需在 Q1 结束前完成改造。
- **提出方**：法务 + 安全团队
- **约束**：必须支持主动撤销，Q1 deadline，不能影响现有登录体验

## 现状分析
- 当前使用 Express 默认 cookie session，session 数据存在服务端内存
- 痛点：单机存储无法水平扩展，且无主动撤销机制
- 不改后果：合规审查不通过，影响产品上线

## 调研与备选方案

### 方案 A：JWT 无状态 token
- **调研内容**：查阅 RFC 7519，对比项目现有 session 机制
- **验证方式**：本地 PoC 测试 token 签发/验证耗时，测试 token 撤销场景
- **验证结果**：签发 <1ms，但无法主动撤销已签发 token，需额外维护黑名单
- **结论**：可行但撤销场景复杂，黑名单本质上又引入了状态

### 方案 B：Redis session
- **调研内容**：评估 Redis 引入成本，查看团队现有 Redis 使用情况
- **验证方式**：压测 Redis 读写延迟，评估单点故障影响
- **验证结果**：p99 <2ms，已有 Redis 集群可复用
- **结论**：成本低，撤销简单，删 key 即失效

## 决策与取舍
采用方案 B（Redis session）。
- **核心理由**：撤销需求是硬性合规要求，方案 B 实现最简且基础设施已就绪
- **取舍说明**：放弃 JWT 的无状态优势，换取撤销的确定性
- **风险**：Redis 故障时 session 不可用，需配合降级策略

## 技术方案
（具体实现设计...）
```

**关键要求**：
- **背景不能省**：没有背景的方案是空中楼阁，读者无法判断方案是否合理
- 每个备选方案都要有调研内容、验证方式、验证结果
- 决策必须写明**核心理由**和**取舍说明**，不能只写"选了 X"
- 调研中的 PoC 代码/命令/数据一并记录，方便后续回溯

## 三、任务工作流

### 完整流程

```
task-start（启动）→ task-execute（执行）→ 自动化验证 → task-finish（收尾）
```

| 阶段 | 触发条件 | skill | 核心动作 |
|------|---------|-------|---------|
| **启动** | 新功能、复杂任务、需求模糊 | `task-start` | 需求对焦 + UI 原型确认 + 方案设计 |
| **执行** | 跨会话大型任务 | `task-execute` | 持续编码 + 进度管理 |
| **验证** | 功能开发完成 | — | 运行自动化测试验证功能可用性，验证脚本沉淀为回归用例 |
| **收尾** | 验证通过后 | `task-finish` | CR 自检 + 复盘沉淀 + 经验教训写入 `docs/decisions/` |

不是每个任务都需要走完整流程。按决策树判断深度：
- **明确小任务**（≤2 文件）：直接动手
- **有模糊点**（≤3 文件）：仅需求对焦，然后直接开发
- **大型任务**（3+ 文件 / 新模块 / 架构决策）：完整走 start → execute → finish
- **从头搭建前后端系统**：**全流程强制**，每个环节（对焦 → UI 原型 → 方案 → execute → finish）都不可跳过
- **跨会话任务**：必须启用 task-execute 框架

### 效率准则
- 能用一句话说清的不用三句话
- 能一次改对的不来回试探
- 能并行处理的不串行等待
- 能复用现有方案的不重新发明
- **先想后动**：想动手时先停 5 秒——我理解需求了吗？我读过相关代码了吗？该走的流程走了吗？三个"是"再开始写代码

### 范围控制 — 防止 Scope Creep
- **只做被要求的事**：发现"顺手可以优化一下"是效率杀手，记下来但不现在做
- **改动最小化**：每次 PR 只包含一个目的的改动，不混入无关重构
- **识别膨胀信号**：当发现"这个也要改、那个也要动"时，停下来重新评估范围

### 问题升级与止损
- **三次原则**：同一个思路尝试 3 次仍不通，立即换方向或求助
- **主动升级**：复杂度超预期、信息不足、需要决策时立即告知用户

### 风险分级执行

| 风险等级 | 场景示例 | 策略 |
|---------|---------|------|
| **低** | 修改局部变量、改样式 | 直接执行，改后验证 |
| **中** | 修改公共函数签名、加依赖 | 先评估影响，改后跑测试 |
| **高** | 数据库 migration、删除文件/分支 | 与用户确认后再执行 |
| **极高** | 生产环境操作、force push | 必须用户明确授权 |

### 上下文管理
- **任务开始**：先了解项目全貌，建立心智模型
- **跨会话延续**：关键决策存入 memory 系统，减少重复沟通成本
- **信息分层**：区分"需要记住的"（存 memory）和"当前够用的"（存 todo）
- **文档优先**：重要信息必须写入 `docs/` 目录随 git 提交，memory 仅作为索引和辅助。文档是持久化真相源，memory 会过期

### 测试与回归
- **验证即门槛**：功能开发完成后，必须编写并运行自动化验证脚本确认功能可用，验证不通过不进入收尾
- **脚本即用例**：验证过程中编写的脚本/命令必须整理为测试用例，记录到 `docs/tests/<模块名>-<功能>.md`
- **回归可执行**：测试用例文档中包含可直接执行的命令或脚本，后续改动时可全量回归
- **用例格式**：每个用例包含：场景描述、前置条件、执行步骤/命令、预期结果

```markdown
# 示例：docs/tests/auth-login.md

## 用例 1：正常登录
- **前置条件**：测试用户已创建
- **执行命令**：`curl -X POST /api/login -d '{"user":"test","pass":"123"}'`
- **预期结果**：返回 200，body 包含 token 字段

## 用例 2：密码错误
- **执行命令**：`curl -X POST /api/login -d '{"user":"test","pass":"wrong"}'`
- **预期结果**：返回 401
```

## 四、常见反模式

| 反模式 | 问题 | 正确做法 |
|--------|------|---------|
| 改之前不读 | 基于假设修改，引入新 bug | 先读完相关代码再动手 |
| 一步到位 | 大量改动一起提交，难定位问题 | 小步提交，逐步验证 |
| 过度设计 | "万一以后需要"导致复杂度膨胀 | 只解决当前问题 |
| 顺手重构 | 改 A 功能时顺手重构了 B 模块 | 记下来开新任务 |
| 沉默卡死 | 卡住了还在默默尝试 | 3 次不通就换方向或求助 |
| 表面补丁 | 不查根因，哪里报错补哪里 | 先定位根因 |
| 假装完成 | 核心路径没验证就说"改好了" | 必须运行验证 |
| 赶着交付 | 跳过思考急于出结果，反复返工 | 先想清楚再动手，做好比做完重要 |
| 跳过对焦 | 需求没对清就开干 | 模糊需求先用 task-start 对焦 |
| 跳过方案 | 大任务直接开写 | 符合条件的先出方案 |
| 跳过自检 | 改完直接提交 | 提交前用 task-finish 自检 |
| 跳过验证 | 功能没跑通就说完成 | 必须运行自动化验证，通过才算完成 |
| 验证即弃 | 验证脚本用完就删 | 沉淀到 `docs/tests/` 作为回归用例 |
| 只存 memory | 重要决策只记 memory 不入库 | 关键信息写入 `docs/` 随 git 提交 |
