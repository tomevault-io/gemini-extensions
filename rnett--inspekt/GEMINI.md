# Inspekt Project Memory

Foundational mandates for AI agents working on `inspekt`, a Kotlin Multiplatform compile-time reflection library.

## Primacy Zone (Mandatory Invariants)

- **Skill Mandate:** ALWAYS activate and use the `compiler_plugin_dev` skill when modifying the compiler plugin. ALWAYS activate and use the `compiler_testing` skill when modifying or running compiler plugin tests. This is a mandatory
  protocol.
- **Security:** NEVER log, print, or commit secrets or `.env` files.
- **Source Control:** NEVER stage, commit, or push changes without explicit user instruction.
- **Tooling:** ALWAYS prefer Gradle MCP for Gradle tasks and JVM runtime exploration.
- **Compiler Version:** The project tracks the latest Kotlin version (currently 2.3.10). ALWAYS use the latest K2/FIR compiler APIs. Do NOT use K1/Legacy compiler APIs.
- **Symbol Access:** Use the generated `Symbols` object (via `dev.rnett.symbol-export`) to access runtime declarations. Do NOT manually look up runtime class IDs or callable IDs if they are available in `Symbols`.
- **Internal Model Backward Compatibility:** Classes in `dev.rnett.inspekt.internal` (like `InspektionResultV1` and its nested classes) are instantiated by code generated by the compiler plugin. Changing their constructor signatures breaks
  backward compatibility for users who update the runtime library but use older compiled code. When adding new fields, ALWAYS provide a secondary constructor with the old signature that supplies defaults, or introduce a new version (e.g.
  `InspektionResultV2`). Validate this requirement every time these classes are edited.

## Architecture & Terminology

- **Modules:**
    - `inspekt`: Runtime library containing the model and annotations.
    - `compiler-plugin`: K2 compiler plugin (FIR and IR).
    - `gradle-plugin`: Gradle integration for the compiler plugin.
    - `test-helpers`: Shared testing utilities.
- **Spekt**: The internal "reflection" model generated by the compiler plugin.
- **Proxy**: Generated JVM-like proxies for interfaces.
- **Symbol Export**: A mechanism that generates a `Symbols` object containing `ClassId` and `CallableId` constants for all exported runtime declarations.

## Technical Rationale (The "Why")

- **K2/FIR**: We target K2 specifically. Frontend checks happen in `InspektChecker` (FIR), while generation happens in `SpektGenerator` and `ProxyGenerator` (IR).
- **KCP Development Utils**: We use `dev.rnett.kcp-development.utils` to reduce boilerplate. It provides `WithIrContext`, `withBuilder`, and extension functions for IR construction.
- **Context Parameters**: We use Kotlin context parameters (e.g., `context(IrBuilderWithScope)`) extensively to pass the builder and context implicitly.

## Operational Execution (The "How")

### Gradle Commands

- **Build & Check:** `./gradlew check`
- **Compiler Plugin Tests:** `./gradlew :compiler-plugin:test`
- **Clean Dumps:** `./gradlew :compiler-plugin:clearDumps` (removes generated test outputs)

### Compiler Testing Protocol

Tests are located in `compiler-plugin/src/testData/auto`.

- **FIR Tests:** Verify frontend extensions (`.fir.txt`).
- **IR Tests:** Verify backend transformations (`.ir.txt` or `.fir.ir.txt`).
- **Box Tests:** Verify runtime execution (`box()` returns `"OK"`).
- **Adding Tests:**
    1. Create a `.kt` file in `compiler-plugin/src/testData/auto/<level>/`.
    2. Run `./gradlew :compiler-plugin:test`.
    3. Review and commit the generated actual output files if they are correct.

### Coding Standards

- **Assertions:** Use `kotlin.test` assertions. Prefer `kotlin.assert` via power-assert.
- **Suspending Tests:** ALWAYS use `runTest`, NEVER `runBlocking`.
- **KDocs:** Use multi-line KDocs (`/** ... */`). Use short names for links and import the target.
- **Dependencies:** ALWAYS use the version catalog (`gradle/libs.versions.toml`).

## Key Compiler API Entry Points

- **FIR (Frontend):**
    - `FirSession`: Access to types, symbols, and session state.
    - `FirSymbol`: Represents a declaration in FIR.
    - `FirChecker`: Validation logic for declarations or expressions.
- **IR (Backend):**
    - `IrPluginContext`: Primary entry point for IR transformations and symbol lookup.
    - `IrBuilderWithScope`: Base class for generating IR expressions and declarations.
    - `IrClass`, `IrFunction`, `IrProperty`: Core IR declaration types.
- **Project Helpers:**
    - `SpektGenerator`: Generates the reflection model (Spekt).
    - `ProxyGenerator`: Generates proxy implementations.
    - `Utils.kt`: Common IR and FIR utilities.

## Agent-Specific Guidance (Kotlin Compiler)

- **Search First:** If you are unsure about a specific compiler API (FIR or IR), search for similar usage in the existing `compiler-plugin/src/main/kotlin` codebase before making assumptions.
- **Symbol Lookup:** Check `compiler-plugin/src/main/kotlin/Names.kt` and the generated `Symbols` object first.
- **IR Building:** Use the extension functions from `kcp-development` (e.g., `irCall`, `irString`, `irArrayOf`).
- **Visibility:** Be careful with visibility when generating calls; use `visibility.isPublicAPI` check where necessary.
- **Type Parameters:** When generating IR for generic functions, ensure type arguments are correctly mapped or erased (`eraseTypeParameters()`).
- **Fake Overrides:** Handle fake overrides explicitly (e.g., `realOverrideTarget`) when inspecting properties or functions.
