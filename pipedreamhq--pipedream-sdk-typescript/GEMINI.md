# CLAUDE.md — Pipedream SDK (TypeScript)

This is a Fern-generated TypeScript SDK with substantial team customizations.
Most files in `src/` are auto-regenerated by the Fern CLI from the Pipedream API
spec; a small set of files in `.fernignore` are hand-maintained and **must not
be replaced by regen output**. Generated and custom code coexist in the same
trees.

## The `.fernignore` contract

`.fernignore` is the only thing standing between custom code and a regen wiping
it out. Treat it as a load-bearing file.

- **Never comment out entries** before running a regen "to see what happens."
  That has happened, and it deleted the wrapper, the browser entrypoint, the
  workflows client, the auth providers, the prettier config, the license, and
  the migration guide. Recovering took a full restoration pass — see `git log`
  around `f831a8c`.
- Before adding new custom files, add their paths to `.fernignore` first.
- When the Fern API spec adds a resource that needs custom client behavior (like
  proxy or workflows), the custom files go on `.fernignore` immediately.

## Public API: the wrapper is `PipedreamClient`

There are two classes named `PipedreamClient`. They are not the same thing:

- `src/Client.ts` — the **base class**, customized but still considered
  "generator-adjacent" (it's listed in `.fernignore` because we override the
  auth init logic). Not exported publicly.
- `src/wrapper/Pipedream.ts` — the `Pipedream` class, which `extends
  PipedreamClient` (the base). `src/index.ts` re-exports it as
  `PipedreamClient`. **This is what consumers see.**

The wrapper adds: env-var resolution (`PIPEDREAM_PROJECT_ID`,
`PIPEDREAM_CLIENT_ID`, `PIPEDREAM_CLIENT_SECRET`,
`PIPEDREAM_PROJECT_ENVIRONMENT`, `PIPEDREAM_BASE_URL`,
`PIPEDREAM_WORKFLOW_DOMAIN`), input validation, the `workflows` getter with
custom domain support, and the `rawAccessToken` accessor.

If a future regen tries to add `export { PipedreamClient } from "./Client.js"`
to `src/index.ts`, that's a name collision — keep the wrapper re-export and drop
the generated one.

## Auth model: `TokenProvider` bridged to `AuthProvider`

Pipedream-side abstraction (`core.TokenProvider`, `getToken(): Promise<string>`)
is older than the generator's auth model (`core.AuthProvider`,
`getAuthRequest()`). The base `Client.ts` keeps a `_tokenProvider` field and
bridges it into the generator's pipeline:

```ts
const auth: core.AuthProvider = {
    getAuthRequest: async () => ({
        headers: { Authorization: `Bearer ${await this._tokenProvider.getToken()}` },
    }),
};
this._options = normalizeClientOptionsWithAuth({ ..._options, auth, headers });
```

`this._options` is the **normalized** shape
(`NormalizedClientOptionsWithAuth<...>`) and is passed verbatim to all
sub-clients. The generated `OAuthAuthProvider` is intentionally bypassed — we
use our `core.OAuthTokenProvider` instead because it has a smart 2-minute expiry
buffer (and a half-lifetime fallback for short-lived tokens) that
`OAuthAuthProvider` doesn't replicate.

The wrapper and browser entrypoint reach through `_tokenProvider` directly.
Don't make it private.

## Custom resources

**Workflows** (`src/api/resources/workflows/`) — entirely custom. Calls a
different domain (`m.pipedream.net` by default, overridable via
`workflowDomain`), not `api.pipedream.com`. Surfaces `invoke()` and
`invokeForExternalUser()`. The Fern API spec doesn't model this resource; if a
regen ever adds a generated `workflows` directory, delete it and keep the custom
one.

**Proxy** (`src/api/resources/proxy/client/`) — single-argument request shape,
**not** the generator's `(url_64, request, requestOptions)` shape. Callers pass
`{ url, externalUserId, accountId, params, headers }`; the client base64-encodes
the URL internally, prefixes all caller headers with `x-pd-proxy-`, and switches
between JSON parsing and binary passthrough based on the `Accept` header. The
generated reference docs occasionally drift back to the `url_64` signature on
regen — check `reference.md` Proxy section after every regen.

**Browser** (`src/browser/index.ts`) — separate entrypoint that must stay free
of Node-only deps. Uses `ConnectTokenProvider` (token-callback flow with
`externalUserId`), exposes `startConnect()` for the Pipedream Connect iframe
flow, and shims `globalThis.process` for the browser. Imports `PipedreamClient
as BackendClient` from `../index.js` — that resolves to the wrapper.

## Generator-managed files (don't hand-edit)

- `src/BaseClient.ts` — added by Fern v3.69.x. Exports `BaseClientOptions`,
  `NormalizedClientOptionsWithAuth`, `normalizeClientOptionsWithAuth`.
  Sub-clients now expect the normalized shape, not the older `BaseClientOptions
  & { token? }`.
- `src/version.ts` — generator bumps `SDK_VERSION` on each regen. The wrapper
  and `Client.ts` import this to populate `X-Fern-SDK-Version` and `User-Agent`
  headers.
- `tests/wire/*.test.ts` — auto-regenerated wire tests. They must keep passing
  after any change to `Client.ts` or sub-client construction.
- All `src/api/types/*.ts`, `src/serialization/types/*.ts`, and resource clients
  **except** those listed in `.fernignore`.

## Regen playbook

When a Fern regen runs (`fern-bot` PR or local `fern generate`):

1. Confirm `.fernignore` has every custom file listed and is **not** commented
   out.
2. After regen, run `npx tsc --noEmit`. If it fails, errors usually fall into
   three buckets:
   - Removed types referenced by `src/api/types/index.ts` (e.g., a
     `ConfigurableProp*` variant the spec dropped) — edit `index.ts` to drop the
     dead re-export.
   - Sub-client option shape changes (e.g., generator adopts a new normalization
     helper) — adapt `src/Client.ts`'s sub-client construction calls. This
     happened with the `NormalizedClientOptionsWithAuth` introduction; it's the
     most likely place a regen breaks things.
   - Custom code referencing a renamed/removed generator symbol — see what the
     regen produced and adjust the bridge.
3. Run `npx vitest run`. Wire tests must still pass.
4. Spot-check the public API: `PipedreamClient` from `@pipedream/sdk` is the
   wrapper, `.workflows` is reachable, `rawAccessToken` returns a Promise,
   browser entrypoint imports cleanly.
5. If the generator is leaving the SDK version unchanged but you've made
   customization changes, bump it manually in `package.json`.

## Verification commands

- Typecheck: `npx tsc --noEmit` (≈1s, must be 0 errors).
- Tests: `npx vitest run` (≈3s; unit + wire, expect ~620+ passing).
- Format / lint: `pnpm format`, `pnpm lint`, `pnpm check` (all biome).

`biome.json` is **broken** under biome 2.4.10 due to glob patterns like
`"!!**/Thumbs.db"` — this is unrelated to any regen and was pre-existing. Fix it
separately when convenient (likely needs `**/Thumbs.db` rewritten as `Thumbs.db`
plus path-separator adjustments).

## Tooling stack

- **Formatter / linter**: Biome 2.4.10 (config in `biome.json`). Prettier was
  previously used for IDE format-on-save but has been removed;
  `.vscode/settings.json` uses `biomejs.biome`. Don't reintroduce
  `.prettierrc.yml` / `.prettierignore`.
- **Test runner**: Vitest 4.x (split into `unit` and `wire` projects).
- **Package manager**: pnpm 10.33.0. CI uses `pnpm install --frozen-lockfile`.
- **Build**: `pnpm build` produces both CJS (`dist/cjs`) and ESM (`dist/esm`)
  outputs, with a separate `dist/{cjs,esm}/browser/` build for the browser
  entrypoint. The `package.json` `"exports"` field is hand-maintained to expose
  `.`, `./browser`, `./server`, `./serialization`, and per-resource paths.

## CI / publishing

`.github/workflows/ci.yml` has three jobs: `compile`, `test`, and `publish`. The
publish job is custom (Fern-generated CI does not include it) and runs only on
`refs/tags/*` pushes — it does `pnpm build` then `pnpm publish --access public
--no-git-checks` using `NPM_TOKEN`. Don't let a regen drop it.

`.github/workflows/auto-close-empty-prs.yml` is also custom; the regen does not
produce it.
