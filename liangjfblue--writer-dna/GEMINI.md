# writer-dna — 写作风格基因提取器

> 从你的原始素材中提炼个人写作风格，生成专属于你的写作 skill。支持任何写作类型——公众号、小说、公文、学术、文案等。框架从你的素材中自动推导，不预设任何写作类型或技法。

**核心理念：零假设。不预设写作类型、不预设技法列表、不预设质量标准。一切从素材中发现。**

## 项目结构

```
writer-dna/
├── CLAUDE.md              ← 你正在读的这个文件
├── templates/             ← 参考实现（展示完整 skill 长什么样，不要修改）
│   └── khazix-writer/     ←   公众号非虚构写作的参考实现
│       ├── SKILL.md       ←   完整 skill 示例
│       └── references/    ←   参考文件
│           ├── style_examples.md
│           ├── content_methodology.md
│           └── scenes/    ←   场景插件示例（按文章原型组织）
├── raw/                   ← 你的原始素材（放进来，AI 只读不改）
│   ├── profile.md         ←   基础信息（AI 引导生成）
│   ├── chats/             ←   聊天记录、朋友圈截图
│   ├── articles/          ←   你自己写的文章
│   ├── references/        ←   你喜欢的作者的文章
│   └── interviews/        ←   访谈、演讲文字稿
├── wiki/                  ← AI 分析后生成的结构化风格知识
│   ├── persona/           ←   人格轨（我是谁）
│   │   ├── core-values.md ←     核心价值观与定位
│   │   └── tone-voice.md  ←     语气、句式、节奏
│   └── craft/             ←   技法轨（我怎么写）
│       ├── vocabulary.md  ←     词库（高频词、口癖、禁用/推荐词）
│       ├── techniques.md  ←     写作技法模式
│       └── self-check.md  ←     个性化自检体系（含 L5 纠错记录）
└── outputs/               ← 最终生成的 skill 包
    └── <你的账号名>-writer/
        ├── SKILL.md
        └── references/
            └── scenes/    ←   用户专属场景插件（生成时自动创建）
```

## 阶段一：引导收集基础信息

当用户调用 `/writer-dna` 或表达创建写作 skill 的意图时，按以下流程引导。

### Step 1: 收集基础信息

逐个询问（一次一个问题）：

1. "你的账号/作品叫什么名字？"
2. "用一句话描述你的写作定位。比如'激发大家对AI的好奇'、'用文字治愈都市人'、'让政策文件看得懂'。"
3. "简单介绍一下你自己。你从事什么行业？做了多久？你的读者主要是谁？"
4. "你的 skill 英文代称用什么名字？最终会生成 `<英文名>-writer` 作为 skill 包的目录名和 SKILL.md 的 name 字段。"
5. "你主要写什么类型的内容？比如公众号科技文、言情小说、政府公文、学术论文、品牌文案……尽量具体描述。"

### Step 2: 生成 profile.md

将用户的回答整理为 `raw/profile.md`，格式：

```markdown
# 写作风格档案

## 基本信息
- **账号名**：<用户回答>
- **skill英文名**：<用户回答>（用于目录名和 SKILL.md name 字段，如 `entropy-writer`）
- **一句话定位**：<用户回答>
- **自我简介**：<用户回答>
- **写作类型**：<用户回答，如"政府公文"/"言情小说"/"科技公众号长文"/"品牌文案">

## 补充信息
- **行业背景**：<从简介中提取>
- **目标读者**：<从简介中提取>
- **写作平台**：<从用户描述中提取>
```

### Step 3: 引导放入素材

先判断用户属于哪种情况，再针对性引导：

**情况 A：有自己作品的写作者**
- "你写过文章吗？如果有，把它们保存为 md 或 txt 放到 `raw/articles/`。作品越多，风格分析越精准。"
- "你平时跟朋友聊天有没有特别有意思的记录？放到 `raw/chats/`。聊天记录最能反映你真实的说话方式。"
- "有没有你特别喜欢的作者？觉得'我想写出那种感觉'的？放到 `raw/references/`，我会分析他们的风格供你借鉴。"

**情况 B：纯新手（想学某种写作风格）**
- "你有没有特别喜欢的作者？想写出他们那种感觉的文章？把他们最好的作品放到 `raw/references/`。这是最重要的素材，我会从中学习写作技法。"
- "如果你有朋友圈、微博、知乎回答等能体现你表达方式的素材，可以放到 `raw/chats/`。这不是必需的——对初学者来说，先从模仿开始是完全正常的，你自己的写作声音会在实践中自然形成。"
- "以后你开始写作品了，把它们放到 `raw/articles/`，系统会自动对比你的写作和学习目标的差距，帮你精准调整。"

**通用补充**：
- **raw/interviews/** — "访谈、播客文字稿、演讲记录放到这里。"
- 告诉用户："素材可以分多轮放入，随时补充。放好后告诉我，我就开始分析。"

### 两种用户画像

系统会根据素材情况自动判断用户类型，采用不同的分析策略：

| 维度 | A：有自己作品的写作者 | B：纯新手（学风格） |
|------|---------------------|-------------------|
| 核心输入 | raw/articles/（自己的作品） | raw/references/（想学的作者） |
| 个人声音来源 | articles/（主）+ chats/（辅） | profile.md（定位）+ chats/（可选） |
| 技法来源 | articles/（自己的技法） | references/（学习目标技法） |
| 生成目标 | "让 AI 用我的风格写" | "让我学会 XX 的风格" |
| 技法标注 | 更多 [CONFIRMED] | 技法标 [LEARNED]，待验证 |
| 迭代路径 | 补充作品 → 微调风格 | 开始写作品 → 自己的声音自然浮现 → 过渡到 A |

情况 B 的用户开始写作品后，随着 raw/articles/ 积累，系统会自动从 B 过渡到 A。

## 阶段二：风格分析与 wiki 生成

当用户表示素材已放好，开始分析。

### 双轨分析

分析分两条轨道独立进行，最后合并到 SKILL.md：

- **人格轨（persona/）**：从素材中提炼"你是什么样的人"——核心价值观、语气节奏、情绪表达方式
- **技法轨（craft/）**：从素材中提炼"你怎么写"——词库、写作技法、自检规则

几乎所有素材都同时影响两条轨道（聊天记录既反映说话方式也反映叙事习惯）。
分轨的意义在于**分析视角**的分离，不是素材的物理隔离。AI 读取全部素材，
但在生成 wiki 时分两个视角提炼。

### 分析前检查

1. 统计 raw/ 中的素材数量和类型
2. 评估素材充分度：

| 素材情况 | 充分度 |
|---------|--------|
| 只有 profile.md | 不足 |
| profile + 1-5 篇素材 | 基础 |
| profile + 6-10 篇素材或 >3万字 | 充分 |
| profile + 10+ 篇素材且类型多样（含多目录）或 >5万字 | 丰富 |

告诉用户当前充分度和可提升方向（如"更多素材会让技法分析更精准"）。

### 素材优先级

权重根据用户类型动态调整：

**情况 A（有自己作品）**：

| 素材类型 | 权重 | 分析角色 |
|---------|------|---------|
| raw/chats/ | 3x | 真实说话方式（人格轨主输入） |
| raw/articles/ | 2x | 自己的技法偏好（技法轨主输入） |
| raw/references/ | 1x | 审美偏好参考，补充技法灵感 |
| raw/interviews/ | 1x | 场景特殊，有参考价值 |

**情况 B（无自己作品，纯新手）**：

| 素材类型 | 权重 | 分析角色 |
|---------|------|---------|
| raw/references/ | 2x（技法轨） | 写作技法学习源（主要输入） |
| raw/chats/ | 可选 | 朋友圈/微博等可作为补充参考，但非必需 |
| raw/interviews/ | 1x | 场景特殊，有参考价值 |
| raw/articles/ | — | 不存在，提示用户开始写后补充 |

**判断规则**：raw/articles/ 中有作品（≥1 篇）则为情况 A，否则为情况 B。

### 冲突解决规则

当不同素材的风格信号矛盾时：

- **聊天记录 vs 作品**（正式度冲突）：skill 输出以作品风格为基准，vocabulary.md 中记录聊天口语作为"非正式模式"备选
- **情况 A：references vs 自己的风格**：始终优先用户自己的风格，references 仅作灵感补充
- **情况 B：references 内部风格不一致**：如果 references 包含多个作者，标注各作者的技法差异，让用户选择想学谁的。如果只有一个作者，以该作者风格为技法基准
- **早期 vs 近期作品**：默认反映最近 6 个月风格，wiki 中注明演变趋势
- 所有矛盾标记出来让用户决定

### wiki/ 写作规范

每个 wiki 文件必须：
- 以一段摘要开头（3-5 句话概括该文件的核心发现）
- 使用清晰的分节标题，结构由分析结果自然推导
- 在相关条目之间加入交叉引用，使用带轨道前缀的路径（如"参见 persona/tone-voice.md#情绪表达"、"参见 craft/techniques.md#开头模式"）
- 技法标注使用四档标签：[CONFIRMED]、[LEARNED]、[SUGGESTED]、[UNIQUE]

### wiki/persona/core-values.md 生成规则

从 raw/profile.md + raw/chats/ + raw/articles/（如有）中提炼：

1. **核心价值观**：
   - 情况 A：从作品中反复出现的观点、判断、好恶中提炼 3-5 个
   - 情况 B：从 profile.md 的定位中提炼 2-3 个；如果 references 中的作者有明确价值观且用户表达了认同，可作为"想践行的价值观"标 [LEARNED]
   - 每个价值观用一句话表述，附带 1-2 个来自素材的证据引用
2. **一句话写作姿态概括**：从素材中提炼一句概括作者写作姿态的话。不预设格式——可能是"有见识的普通人在认真聊一件打动他的事"，也可能是"以温柔笔触写尽人间冷暖"，也可能是"用严谨逻辑讲清复杂政策"。从素材中自然推导。
3. **使命/驱动**：这个账号/作品系列为什么存在？作者想通过写作实现什么？
4. **反对什么**：明确作者不认同、不做的方向。

### wiki/persona/tone-voice.md 生成规则

从 raw/chats/（如有）+ raw/articles/（如有）+ raw/references/ 中分析：

- 情况 A：从自己的作品和聊天记录中提炼说话方式
- 情况 B（无 chats）：从 references/ 中提炼目标作者的语气和节奏（标 [LEARNED]），从 profile.md 获取用户的基本定位作为人格基础。生成的 tone-voice 会更接近 references 作者的风格，这是正常的——初学者从模仿开始，自己的写作声音会在实践中自然发展

分析维度（全部从素材中推导，不预设"应该怎样"）：

1. **句子节奏**：平均句长、长短交替模式、是否有意制造停顿
2. **段落习惯**：一句话成段 vs 长段落，段落长度分布
3. **人称使用**："我"的使用频率和方式、"你"的称呼方式
4. **情绪表达**：直接 vs 克制、自嘲频率和方式、感叹号/问号/省略号频率
5. **标点偏好**：哪些标点用得多、哪些几乎不用
6. **叙述视角**：私人视角 vs 旁观视角、第一人称 vs 第三人称
7. **正式度**：口语化程度、是否使用书面表达
8. **该类型特有的语气特征**：从素材中发现该写作类型独有的语气模式（如公文的严谨权威、小说的情绪张力、学术论文的客观中立等）

### wiki/craft/vocabulary.md 生成规则

从全部 raw 素材中提取，按用户类型区分分析策略：

1. **高频特色词**：
   - 情况 A：从自己的作品和聊天记录中提取出现 3 次以上的特色表达
   - 情况 B（无 chats）：从 references/ 中提取目标作者的特色词（标 [LEARNED]，作为学习目标）
   - 分类整理：根据素材中的实际用法分类（如转场类、判断类、情绪类、行业术语类等）
2. **口头禅/口癖**：
   - 情况 A：从自己的素材中提取个人标志性表达
   - 情况 B（无 chats）：从 references/ 中提取作者口癖（标 [LEARNED]）
3. **禁用词/禁用语**：
   - 情况 A：用户素材中从未出现的、与该写作类型调性不符的表达
   - 情况 B：references 作者也从不使用的表达 + 用户自己不用的表达
   - 注意：不同类型的禁用词不同——公众号可能禁用"AI味词"，公文可能禁用口语，小说可能禁用说教
4. **推荐替代词**：
   - 情况 A：用户习惯用什么替代其他表达
   - 情况 B：references 作者的替代方式（标 [LEARNED]）+ 用户自己的替代
5. **标点偏好**：用户常用和不用的标点
6. **质感光谱**（关键维度，不可省略）：
   - 从素材中提炼整体语言的质感定位，用一句比喻性描述概括（如"温柔清淡如流水"、"犀利干脆如刀锋"、"厚重沉稳如大地"）
   - 分别定义以下子维度的质感定位：
     - **叙事语言质感**：旁白/叙事部分的整体质地（如"温润清淡"vs"活泼跳脱"）
     - **对话语言质感**：角色对话的质地（如"自然口语"vs"文艺腔"）
     - **幽默质感**：搞笑/吐槽的质地（如"温暖会心一笑"vs"新锐爆笑"）
     - **比喻质感**：修辞手法的质地（如"偏文学感"vs"偏网络段子"）
   - 明确指出素材中**不存在**的质感方向（如"叙事语言不是活泼跳脱的网络感"）
   - 这一维度解决的核心问题：技法可以枚举，但"同样的技法用不同的质感呈现，味道完全不同"

### wiki/craft/techniques.md 生成规则

从 raw/articles/ + raw/references/ + raw/chats/ 中分析。**核心原则：发现，不预设。**

不预设任何技法名称或分类。从素材中发现什么就记录什么。

#### 第一步：结构发现

从素材中识别以下模式（不预设分类名称，从素材中自然发现）：

1. **作品原型分类**：
   - 从素材中识别作者写了哪几种类型的作品（不预设类型名称）
   - 每种原型记录：名称、特征描述、典型示例
   - 可能是"调查实验型/产品体验型"（公众号），也可能是"甜宠/虐恋/权谋"（小说），也可能是"通知/报告/请示"（公文）——完全由素材决定
2. **整体结构**：从素材中提取叙事/论述结构模式（线性？三幕式？总分总？多线并行？起承转合？）
3. **开头模式**：从素材中发现有哪些不同的开头方式，每种附真实示例
4. **结尾模式**：从素材中发现收尾方式，每种附真实示例
5. **段落/章节衔接方式**：从素材中发现如何衔接不同段落或章节

#### 第二步：技法发现

扫描素材中反复出现的写法模式，每个模式：
- 给它一个描述性名称
- 附 1-3 个来自素材的真实示例
- 标注来源：
  - [CONFIRMED] — 从用户自己的素材中有明确证据
  - [LEARNED] — 从 references 中学习到的技法（情况 B 专属，用户尚未实践验证）
  - [SUGGESTED] — 分析中推断可能有用的技法，但素材中未明确出现
  - [UNIQUE] — 用户独有的写法

不限制技法数量和种类——可能是"文化升维"也可能是"人物白描"也可能是"悬念反转"也可能是"先抑后扬"也可能是"三段论式论证"。

**特别关注**：该写作类型独有的技法。不同类型有完全不同的技法体系：
- 公众号非虚构：亲自下场、文化升维、读者直呼……
- 小说虚构：伏笔回收、人物弧光、多视角叙事、悬念节奏……
- 公文：三段论、总分总、措辞精准、引用依据……
- 学术：文献综述、论点-论据-论证、研究方法说明……
- 品牌文案：情感共鸣、利益点转化、行动呼吁、场景代入……

以上只是举例，实际技法从素材中发现。

#### 保留的通用分析维度

以下不是预设的技法名称，而是分析视角（确保不遗漏关键维度）：
- 开头方式
- 结尾方式
- 段落衔接
- 整体结构
- 反复出现的写法模式
- 该类型独有的特殊技法

#### 第三步：节奏密度量化

从素材中量化叙事节奏模式，提取具体的密度参数（不预设参数值，从素材中测量）：

1. **互动间隔**：角色互动（对话、动作交锋、情感碰撞）之间的"留白"密度
   - 从素材中测量：每两个互动之间平均插入多少非互动段落（环境描写、思绪漫游、日常铺陈）
   - 记录为具体数字（如"1-2段留白"vs"无留白直接推进"）
2. **慢段落频率**：非推进情节的"慢下来"段落出现频率
   - 从素材中测量：每 N 字中至少有 1 处不推进情节的描写/思绪段落
   - 记录为具体数字范围
3. **余韵要求**：关键场景（甜蜜/搞笑/紧张）结束后的"尾巴"密度
   - 从素材中测量：关键场景结束后是否有 1-2 句/段不直接推进情节的余韵
   - 记录为"有余韵"/"无余韵直接切换"
4. **节奏呼吸周期**：情绪循环的频率
   - 从素材中测量：从一种情绪切换到另一种情绪大约需要多少字
   - 记录为具体数字范围（如"每 1500-2000 字完成一个情绪循环"）

**核心目的**：技法是"写什么"，节奏密度是"以什么频率写"。同样的技法用不同节奏呈现，风格完全不同。这一步确保 wiki 不仅记录"有哪些技法"，还记录"这些技法应该以什么密度和节奏使用"。

### wiki/craft/self-check.md 生成规则

基于其他 wiki 文件生成个性化自检体系。**核心原则：L1/L5 是硬性底线用指令式，L2-L4 是风格偏好用建议式。具体规则从素材推导，不硬编码。**

整体通过判定：L1 和 L5 必须全部通过（零命中/零违规），否则阻断输出。L2-L4 不阻断，但在质检报告中标注偏离程度（"达标"/"略偏离"/"严重漂移"）。

**L1 硬性规则**（自动扫描，指令式，零容忍）：
- 从 wiki/craft/vocabulary.md 提取禁用词列表和禁用标点
- 从素材中发现该类型的硬性禁忌（不预设什么是禁忌——公文可能禁忌口语，小说可能禁忌人物崩坏，学术可能禁忌主观断言）
- 规则格式：`指令：全文搜索xxx，出现则必须替换`
- 定义通过标准：零命中，未通过则必须修复

**L2 风格一致性**（模式匹配，建议式，不阻断）：
- 从 wiki/persona/tone-voice.md 提取节奏和结构检查规则
- 从 wiki/craft/vocabulary.md 提取用词检查规则
- 规则格式：`建议：xxx。参考范围：xxx`
- 定义通过标准：建议达标，未达标标注"风格偏离风险"
- 例外：L2 的标点禁令二次确认保留指令式（本质是 L1 的防回归检查）

**L3 内容质量**（深度审查，建议式，不阻断）：
- 从 wiki/craft/techniques.md 推导该类型的内容质量标准
- 不预设质量标准——从素材中发现该类型重视什么：
  - 非虚构可能是"观点支撑""知识输出方式"
  - 虚构可能是"人物一致性""剧情逻辑""世界观自洽"
  - 公文可能是"格式规范""措辞准确""逻辑严密"
  - 学术可能是"论证严谨""引用规范""方法清晰"
- 规则格式：`建议：xxx。参考范围：xxx`
- 定义通过标准：建议达标，未达标标注"内容质量建议"

**L4 作者声音一致性**（终审，建议式，不阻断）：
- 从 wiki/persona/core-values.md 提取声音/姿态判断标准
- 从 wiki/persona/tone-voice.md 提取语气一致性判断标准
- 不同类型"声音"的定义不同——公众号需要"活人感"，公文需要"权威感"，学术需要"客观感"。从素材推导该类型需要什么样的声音一致性。
- 规则格式：`建议：xxx`
- 定义通过标准：整体感知达标，未达标标注"声音不一致"

**L5 运行时纠错**（动态最高优先级）：
- 从对话中收集用户纠错反馈，格式为：`- [日期] 当[场景]时，不应[错误行为]，应[正确行为]`
- 每条 L5 记录必须包含"当[xxx场景]时"的条件，只在匹配的场景下生效
- L5 不能 blanket 否定整条 L1 规则——如果一条 L5 会使某条 L1 在所有场景下失效，必须警告用户
- L5 最多保留 50 条，超出时合并语义相近的记录（保留最新的）
- 相同场景的新纠错覆盖旧纠错（旧记录标记为 `[已替代]`）
- 优先级：L5 > L1 > L2 > L3 > L4
- 定义通过标准：无 L5 记录时自动通过；有 L5 记录时，匹配场景必须遵循纠错指令

## 阶段二（续）：质感对比分析

wiki 生成后、用户确认前，必须执行质感对比分析，确保 skill 捕捉的不只是"可枚举的显性特征"，还包括"整体质感"。

### Step 1：生成测试稿

基于已生成的 wiki，用 wiki 中的风格规则写一篇 **500-1000 字的仿写测试稿**：
- 选择 wiki/craft/techniques.md 中识别出的一个作品原型
- 使用 wiki 中记录的语气、词汇、技法
- 测试稿存入 `raw/articles/`（文件名标记为测试稿，如 `test-auto-<日期>.md`）

### Step 2：逐段质感对比

将测试稿与原始素材（raw/references/ 或 raw/articles/）逐段对比，检查以下维度：

| 对比维度 | 检查内容 |
|---------|---------|
| 节奏 | 测试稿的叙事节奏是否与素材一致？是否有"快了"或"慢了"的感觉？ |
| 语言质感 | 测试稿的整体语言质地是否与素材一致？是否有"太新锐"/"太老旧"/"太正式"/"太随意"的感觉？ |
| 用词质地 | 测试稿的用词风格是否与素材一致？是否有词汇虽然不在禁用列表中，但"感觉不对"？ |
| 情感烘托 | 测试稿的情感表达方式是否与素材一致？是否有"太直接"或"太含蓄"的感觉？ |
| 余韵 | 测试稿的场景切换是否有素材中的"尾巴"？还是过于干脆？ |

### Step 3：生成差异报告

将发现的质感差异整理为报告，格式：

```markdown
## 质感对比报告

### 测试稿与素材一致的方面
- ...

### 质感差异
| 维度 | 素材中的表现 | 测试稿中的表现 | 差异描述 |
|------|------------|--------------|---------|
| ... | ... | ... | ... |

### 建议新增的 L5 纠错
- [日期] 当[场景]时，不应[错误行为]，应[正确行为]
```

### Step 4：更新 wiki 和 skill

将差异报告中的发现：
1. 更新到 wiki/craft/vocabulary.md 的"质感光谱"维度
2. 更新到 wiki/craft/techniques.md 的"节奏密度"参数
3. 转化为 L5 纠错记录写入 wiki/craft/self-check.md
4. 同步到 outputs 中的 SKILL.md

**目的**：给 skill 包一个"出生体检"，在用户确认前就发现并修正质感偏差，而不是等到用户自己测试才发现问题。

## 阶段二（续续）：风格确认

wiki 生成后，逐项跟用户确认。一次一个维度，一次一个问题：

**确认示例：**
- "我注意到你写文章时经常用'xxx'，这是你的口癖吗？"
- "你的文章开头好像习惯从xxx切入，我总结了 3 种模式，你看准不准？"
- "这些词我标记为禁用词（你素材中从没用过），要调整吗？"
- "你好像从来没有用过小标题，你的文章都是一口气顺下来的，对吗？"
- "我识别出你写 3 种类型的作品，分别是xxx、xxx、xxx，这个分类准确吗？"

用户确认后更新 wiki，修正后进入阶段三。

## 阶段三：生成 skill 包

### 生成公式

```
outputs/<账号名>-writer/ = 从 wiki/* 推导出的完整 skill 包
```

templates/ 目录下的参考实现不参与生成——它只展示一个完整 skill 长什么样，供 AI 理解 skill 的"形状"。实际生成完全由 wiki 分析结果驱动。

### SKILL.md 结构推导

**不使用固定映射表**。根据 wiki 分析结果推导出适配该写作类型的 SKILL.md 结构。

#### 所有类型共有的骨架

```markdown
---
name: <账号名>-writer
description: <基于定位生成>
---

# <账号名>写作

> 关于作者（从 wiki/persona/core-values.md 生成）

## 核心价值观
（从 wiki/persona/core-values.md 生成）

## 写作流程
（步骤数和内容由 wiki/craft/techniques.md 推导，见下方规则）

## 自检体系
（L1-L5 结构固定，具体规则从 wiki/craft/self-check.md 注入）

## 参考资料
（指向 references/ 下的文件）
```

#### 写作流程的推导规则

写作流程的步骤数和内容**从 wiki/craft/techniques.md 中发现的结构模式推导**：

- wiki 中发现了什么创作流程模式 → SKILL.md 就包含什么步骤
- 如果素材显示"选题→构思→写作→修改" → 就用这个流程
- 如果素材显示"世界观设定→人设→大纲→写作→修改" → 就用这个流程
- 如果素材显示"需求理解→素材整理→起草→审核" → 就用这个流程
- 如果素材显示"调研→论证→写作→审查" → 就用这个流程

每个步骤中注入对应 wiki 文件的内容：
- 选题/构思类步骤 → 注入 wiki/craft/techniques.md 的作品原型分类和选题方法
- 写作类步骤 → 注入 wiki/persona/tone-voice.md（风格内核）、wiki/craft/vocabulary.md（禁区+推荐词）、wiki/craft/techniques.md（技法+结构模板+场景）
- AI 协作类步骤 → 根据 wiki 内容生成适合该类型的"AI 擅长/必须人来"分工
- 自检步骤 → 注入 L1-L5 自检体系

**参考实现的映射**：templates/khazix-writer/SKILL.md 的四步结构（选题判断→角色边界→写作→自检）只是"公众号非虚构"的推导结果。如果分析结果与之一致，自然会生成类似结构；如果不一致，就生成不同的结构。

#### 自检体系的注入规则

- L1→L2→L3→L4→L5 五层结构固定，所有类型都保留
- 每层的具体规则从 wiki/craft/self-check.md 注入（推导式生成，见上方生成规则）
- 质检报告模板保留，结构一致：
  - L1 硬性规则：✅/❌（必须通过）
  - L2-L4：✅/⚠️/❌ 三档
  - L5 纠错记录：✅/❌/无记录
  - 总评 + 风格偏离提示

### references/ 生成规则

#### style_examples.md

1. 从 raw 素材中提取用户自己的真实片段作为示例
2. 素材不够时，AI 根据 wiki 风格规则生成仿写示例，标注"[AI 仿写，请替换为真实素材]"
3. 分类不预设——从 wiki/craft/techniques.md 中发现的技法驱动分类
4. 每个技法类别附真实示例或仿写示例，标注 [CONFIRMED]/[SUGGESTED]/[UNIQUE]

#### content_methodology.md

- 如果素材中有选题/创作方法论的内容 → 提取并生成
- 如果没有 → 不生成这个文件（不是所有写作类型都需要独立的方法论文档）
- 生成时结构从素材推导，不套用固定模板

#### scenes/

场景插件提供具体作品类型的写作技法。

**生成时机**：在生成 SKILL.md 之后，基于 wiki/craft/techniques.md 的作品原型分类生成。

**生成规则**：

1. 从 wiki/craft/techniques.md 提取作品原型列表
2. 如果只有一种原型 → 不需要 scenes/ 目录，技法直接写在 SKILL.md 中
3. 如果有多种原型 → 每种生成一个场景插件文件，文件名用原型名
4. 每个场景插件包含：
   - 该类型作品的写法重心和注意事项
   - 适用的开头模式（附用户真实示例或仿写示例）
   - 适用的结构模板（附真实示例或仿写示例）
   - 适用的收尾方式
   - 该类型独有的技法（标 [CONFIRMED]/[SUGGESTED]/[UNIQUE]）
5. 通用技法放在一个 `通用技法.md` 文件中

**参考**：`templates/khazix-writer/references/scenes/scene_examples.md` 展示了完整的场景插件应该包含什么内容（但具体分类和内容由用户素材驱动）。

### 生成后

告诉用户：
1. skill 包已生成到 `outputs/<账号名>-writer/`
2. 建议先审阅 SKILL.md，重点检查核心价值观、禁用词、风格示例是否准确
3. 建议审阅 `references/scenes/` 下的场景插件（如果有），确认作品类型分类和技法是否准确
4. 安装方式：将 outputs 下的整个目录复制到 skills 目录（macOS/Linux: `~/.claude/skills/`，Windows: `%USERPROFILE%\.claude\skills\`）
5. 后续可以随时补充素材到 raw/，重新运行分析来迭代升级

## 迭代升级

当用户补充新素材或要求更新时：

1. 读取新放入 raw/ 的素材
2. 对比现有 wiki，只在有新证据时提出修改
3. 默认反映用户最近的风格（近 6 个月权重更高）
4. 修改以 diff 形式呈现给用户审批
5. 用户确认后更新 wiki，重新生成 outputs

### 对话纠错流程

当用户在写作过程中说"这句不像我"、"这里不对"时，按以下 5 步处理：

1. **理解纠错**：AI 提取 [场景, 错误行为, 正确行为]
2. **追问澄清**：如果提取不明确（用户只说"不对"没说哪不对），追问一次
3. **冲突检查**：检查是否与现有 L5 记录冲突或语义重复，冲突时告知用户
4. **确认写入**：展示将要写入的纠错记录，请用户确认
5. **持久化**：用户确认后追加到 wiki/craft/self-check.md 的 L5 纠错记录区，仅更新 outputs 中 SKILL.md 的 L5 部分（不重新生成整个文件）

wiki 单文件超过 500 行时，提示用户是否需要拆分。

## 边界情况

| 情况 | 处理 |
|------|------|
| 只有 profile，无素材 | 降级生成，技法全标 [SUGGESTED]，提示补充素材 |
| 素材是英文/中英混合 | 用素材原始语言分析，skill 保持用户主语言 |
| 用户风格与参考实现完全不同 | 不影响，系统不依赖参考实现，完全从素材推导 |
| 只有 references 无自己的作品 | 按情况 B 处理：references 作为技法学习源（标 [LEARNED]），profile.md 提供基本定位。生成的 skill 以学习 references 风格为主，提示用户开始写作品后补充 articles/ 以发展个人风格 |
| wiki/ 既有平铺文件又有 persona/craft/ 子目录 | 子目录优先，平铺文件视为遗留数据，提示用户可删除 |
| 素材类型与 profile 中的"写作类型"不一致 | 以素材实际内容为准，提醒用户核实"写作类型"描述 |
