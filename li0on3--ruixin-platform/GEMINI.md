1. 系统概述
1.1 项目背景
瑞幸咖啡分销商自动化系统旨在通过中间平台技术解决方案，实现瑞幸咖啡产品的高效分销和订单管理。系统核心创新点在于利用特殊优惠链接（如https://lkcoffe.cn/?card=4H8SB644）实现无需在线支付的自动化下单流程，并为分销商和管理员提供全方位的业务管理和数据分析能力。
平台采用Go语言后端和Vue前端技术栈，以确保系统高性能、稳定性和良好的用户体验。通过模拟HTTP请求方式与瑞幸咖啡官方平台交互，系统实现了完整的自动化下单流程，显著提升了分销业务的效率和规模化能力。
1.2 业务价值
本系统为分销业务带来的核心价值包括：

效率提升：通过API自动化取代手动下单，显著提高订单处理速度，降低人为错误
成本优化：集中管理特殊优惠链接资源，最大化链接使用效率，优化资源配置
规模扩展：批量处理能力支持业务规模快速扩张，降低运营压力与人力成本
数据驱动：全面的数据分析工具提供业务洞察，支持精准决策
风险控制：完整的资金管理机制和安全审计体系，降低业务风险

1.3 用户群体定义
系统服务两类核心用户：

管理员：平台运营方管理人员，负责链接资源管理、分销商管理、订单监控和系统配置

平台运营管理员
财务管理人员
系统维护人员


分销商：使用平台服务的合作伙伴，通过API接口发起下单请求

企业级分销合作伙伴
渠道代理商
中小型分销商户



1.4 系统定位与边界
本系统定位为瑞幸咖啡官方平台与分销商之间的中间层服务平台，具有以下特点：

业务边界：核心聚焦于链接资源管理、订单自动化处理和分销商关系管理
技术边界：通过模拟HTTP请求方式与瑞幸咖啡平台交互，独立于官方系统
用户边界：仅服务平台管理员和授权分销商，不直接面向终端消费者
扩展预留：架构设计预留多渠道接入和功能扩展可能性

3. 角色与权限设计
3.1 角色定义
系统定义两个主要角色，具有不同的权限范围和操作能力：
3.1.1 管理员角色
管理员作为系统运营方代表，拥有最高级别的系统访问权限：

管理权限界定：可访问所有管理功能，执行所有管理操作
安全责任：负责系统配置安全、资源管理和敏感操作控制
业务洞察：可查看所有业务数据和分析报告
账号管理权限：可创建和管理分销商账户

3.1.2 分销商角色
分销商作为系统服务对象，通过API接口使用系统服务：

账户权限范围：仅能访问自己的账户信息和订单数据
API操作能力：通过API执行下单和查询操作
资金管理权限：可查询自身余额和申请提现
数据查看范围：仅能查看与自身相关的业务数据

5. 核心业务流程
5.1 链接资源全生命周期流程
链接资源是系统的核心业务资产，其完整生命周期包含从获取到失效的全过程管理。
5.1.1 链接获取与入库流程
1. 管理员从外部渠道获取优惠链接资源
   ↓
2. 准备链接入库数据（分类、批次、价格）
   ↓
3. 选择入库方式（单个/批量）
   │
   ├─> 3.1 单个链接录入
   │     ↓
   │    3.1.1 填写链接信息
   │     ↓
   │    3.1.2 系统验证链接有效性
   │     ↓
   │    3.1.3 确认并保存
   │
   └─> 3.2 批量链接导入
         ↓
        3.2.1 准备导入文件(Excel/CSV)
         ↓
        3.2.2 上传并验证文件格式
         ↓
        3.2.3 映射字段并预览数据
         ↓
        3.2.4 系统批量验证链接
         ↓
        3.2.5 确认并执行批量导入
           ↓
4. 系统生成入库记录
   ↓
5. 链接状态设置为"未使用"
   ↓
6. 系统检查库存预警条件
   ↓
7. 链接入库完成，可用于分销商下单
关键节点与处理逻辑：

链接验证：

验证链接格式是否符合要求(https://lkcoffe.cn/?card=XXXXXXXX)
检查链接是否已存在于系统中
验证链接的有效性（可选的外部API调用）


批量导入处理：

支持Excel/CSV格式导入
提供字段映射配置界面
导入过程的数据验证和错误处理
导入结果统计和报告生成


库存预警检查：

计算种类当前库存量
对比预警阈值设置
触发条件满足时发送预警通知



5.1.2 链接使用与状态流转
1. 分销商通过API发起下单请求
   ↓
2. 系统根据订单信息选择合适链接
   ↓
3. 链接状态变更为"使用中"（临时状态）
   ↓
4. 系统使用链接构造瑞幸咖啡下单请求
   ↓
5. 发送模拟HTTP请求到瑞幸咖啡平台
   ↓
6. 接收并解析瑞幸咖啡平台响应
   │
   ├─> 6.1 下单成功
   │     ↓
   │    6.1.1 获取取餐码和二维码
   │     ↓
   │    6.1.2 链接状态变更为"已使用"
   │     ↓
   │    6.1.3 记录成功使用记录
   │
   └─> 6.2 下单失败
         ↓
        6.2.1 链接状态变更为"使用异常"
         ↓
        6.2.2 记录失败原因
         ↓
        6.2.3 生成异常报告
           ↓
7. 更新链接状态历史记录
   ↓
8. 系统检查库存预警条件
   ↓
9. 链接生命周期结束（不可再次使用）
关键节点与处理逻辑：

链接选择算法：

优先选择近期入库的链接（FIFO原则）
考虑链接种类与订单商品的匹配度
考虑有效期因素，优先使用接近过期的链接


链接状态管理：

使用状态锁定机制防止并发问题
完整记录状态变更历史和原因
异常状态需记录详细失败原因


过期处理：

定时任务检查链接有效期
自动将过期链接标记为失效
生成过期统计报告



5.2 分销商服务流程
5.2.1 分销商账户创建与管理
1. 分销商线下联系平台运营方
   ↓

    message := distributorId + strconv.FormatInt(timestamp, 10) + body
    h := hmac.New(sha256.New, []byte(secretKey))
    h.Write([]byte(message))
    return hex.EncodeToString(h.Sum(nil))
}

func assignLinkWithOptimisticLock(categoryID int64) (*Link, error) {
    tx := db.Begin()
    defer func() {
        if r := recover(); r != nil {
            tx.Rollback()
        }
    }()

    var link Link
    // 使用FOR UPDATE SKIP LOCKED获取一个未使用的链接
    if err := tx.Raw(`
        SELECT * FROM links 
        WHERE category_id = ? AND status = 'unused' AND expires_at > NOW()
        ORDER BY expires_at ASC
        LIMIT 1 FOR UPDATE SKIP LOCKED`, categoryID).Scan(&link).Error; err != nil {
        tx.Rollback()
        return nil, err
    }

    // 更新链接状态
    if err := tx.Model(&link).Update("status", "using").Error; err != nil {
        tx.Rollback()
        return nil, err
    }

    // 创建状态历史记录
    history := LinkStatusHistory{
        LinkID: link.ID,
        Status: "using",
        Reason: "Assigned for order",
    }
    if err := tx.Create(&history).Error; err != nil {
        tx.Rollback()
        return nil, err
    }

    return &link, tx.Commit().Error
}

go// Go实现示例 - 请求重试机制
func doWithRetry(fn func() error, maxRetries int, delay time.Duration) error {
    var err error
    for i := 0; i < maxRetries; i++ {
        err = fn()
        if err == nil {
            return nil
        }
        
        // 判断是否是可重试的错误
        if !isRetryableError(err) {
            return err
        }
        
        // 指数退避策略
        sleep := delay * time.Duration(math.Pow(2, float64(i)))
        time.Sleep(sleep)
    }
    return fmt.Errorf("after %d retries: %w", maxRetries, err)
}

func isRetryableError(err error) bool {
    // 网络超时、连接重置等错误可重试
    if netErr, ok := err.(net.Error); ok && netErr.Timeout() {
        return true
    }
    
    // HTTP 5xx 错误可重试
    if respErr, ok := err.(*httpError); ok && respErr.StatusCode >= 500 {
        return true
    }
    
    return false
}

func updateDistributorBalance(id int64, amount float64, txType string, relatedID string, remark string) error {
    return db.Transaction(func(tx *gorm.DB) error {
        var distributor Distributor
        
        // 使用行锁获取最新账户信息
        if err := tx.Set("gorm:query_option", "FOR UPDATE").First(&distributor, id).Error; err != nil {
            return err
        }
        
        // 检查余额是否充足（扣减操作）
        if amount < 0 && distributor.Balance+amount < 0 {
            return errors.New("insufficient balance")
        }
        
        // 保存操作前的余额
        beforeBalance := distributor.Balance
        
        // 更新余额
        if err := tx.Model(&distributor).Update("balance", gorm.Expr("balance + ?", amount)).Error; err != nil {
            return err
        }
        
        // 创建交易记录
        transaction := Transaction{
            DistributorID: id,
            Type:          txType,
            Amount:        amount,
            BalanceBefore: beforeBalance,
            BalanceAfter:  beforeBalance + amount,
            RelatedID:     relatedID,
            Remark:        remark,
        }
        
        if err := tx.Create(&transaction).Error; err != nil {
            return err
        }
        
        return nil
    })
}

9.2.2 缓存策略

多级缓存：
go// Go实现示例 - 多级缓存
type CacheService struct {
    localCache *sync.Map      // 本地内存缓存
    redisCache *redis.Client  // Redis分布式缓存
}

func (s *CacheService) Get(key string, data interface{}) error {
    // 先查本地缓存
    if val, ok := s.localCache.Load(key); ok {
        // 类型断言和反序列化
        bytes, ok := val.([]byte)
        if ok {
            return json.Unmarshal(bytes, data)
        }
    }
    
    // 查Redis缓存
    val, err := s.redisCache.Get(context.Background(), key).Bytes()
    if err == nil {
        // 缓存命中，更新本地缓存
        s.localCache.Store(key, val)
        return json.Unmarshal(val, data)
    }
    
    // 缓存未命中
    if err != redis.Nil {
        return err
    }
    
    return errors.New("cache miss")
}

func (s *CacheService) Set(key string, data interface{}, expiration time.Duration) error {
    // 序列化数据
    bytes, err := json.Marshal(data)
    if err != nil {
        return err
    }
    
    // 更新本地缓存
    s.localCache.Store(key, bytes)
    
    // 更新Redis缓存
    return s.redisCache.Set(context.Background(), key, bytes, expiration).Err()
}

4.6 分销商API子系统
功能描述： 为分销商提供标准化的API接口，实现程序化订单创建和查询。

详细功能规格：

认证接口
Token获取：
接口路径：/api/v1/auth/token
请求方法：POST
参数：分销商ID、密码、时间戳
返回：访问Token、有效期
错误处理：认证失败、请求超时
Token刷新：
接口路径：/api/v1/auth/refresh
请求方法：POST
参数：刷新Token
返回：新访问Token、有效期
错误处理：Token无效、超过刷新期限
请求签名验证：
签名算法：HMAC-SHA256
签名内容：分销商ID + 时间戳 + 请求体
签名位置：请求头X-Signature字段
验证规则：时间戳5分钟内有效
订单接口
创建订单：
接口路径：/api/v1/distributor/order/create
请求方法：POST
参数：订单信息（门店、商品、规格等）
处理：验证余额、分配链接、执行下单
返回：订单ID、状态、取餐码、二维码
异常处理：余额不足、商品无效、下单失败
批量创建订单：
接口路径：/api/v1/distributor/order/batch-create
请求方法：POST
参数：订单列表（最多10个）
处理：批量验证、并行处理
返回：订单结果列表、成功/失败统计
异常处理：部分成功处理策略
查询订单：
接口路径：/api/v1/distributor/order/query
请求方法：POST
参数：订单ID或自定义订单号
返回：订单详细信息、状态、商品明细
异常处理：订单不存在、权限验证
订单列表：
接口路径：/api/v1/distributor/order/list
请求方法：POST
参数：筛选条件、分页信息
返回：订单列表、分页元数据
异常处理：参数错误、权限验证
门店与商品接口
门店查询：
接口路径：/api/v1/distributor/store/search
请求方法：POST
参数：城市ID、关键词、产品ID等
返回：门店列表、位置信息、营业时间
异常处理：参数错误、无匹配门店
菜单查询：
接口路径：/api/v1/distributor/menu/query
请求方法：POST
参数：门店代码、其他筛选条件
返回：分类列表、商品信息、价格信息
异常处理：门店代码无效、菜单获取失败
商品详情：
接口路径：/api/v1/distributor/goods/detail
请求方法：POST
参数：门店代码、商品ID
返回：商品详情、规格选项、价格信息
异常处理：商品ID无效、详情获取失败
账户接口
余额查询：
接口路径：/api/v1/distributor/balance/query
请求方法：POST
参数：分销商ID
返回：当前余额、冻结金额、可用余额
异常处理：权限验证
流水查询：
接口路径：/api/v1/distributor/transaction/list
请求方法：POST
参数：时间范围、类型筛选、分页信息
返回：流水记录列表、分页元数据
异常处理：参数错误、权限验证
预警设置：
接口路径：/api/v1/distributor/warning/config
请求方法：POST
参数：预警阈值、通知配置
返回：配置结果、生效时间
异常处理：参数错误、权限验证
业务规则：

API访问需有效Token认证和请求签名
Token默认有效期24小时，刷新Token有效期7天
订单创建前验证分销商余额充足性
批量下单单次请求最多支持10个订单
所有金额相关接口都精确到小数点后2位
分销商只能访问自己的订单和账户数据
无效的API请求将返回标准格式的错误响应

   │     ↓
   │    3.1.2 系统验证链接有效性
   │     ↓
   │    3.1.3 确认并保存
   │

链接验证：
验证链接格式是否符合要求(https://lkcoffe.cn/?card=XXXXXXXX)

支持Excel/CSV格式导入

计算种类当前库存量
对比预警阈值设置
触发条件满足时发送预警通知
5.1.2 链接使用与状态流转
1. 分销商通过API发起下单请求
   ↓
2. 系统根据订单信息选择合适链接
   ↓
3. 链接状态变更为"使用中"（临时状态）
   ↓
4. 系统使用链接构造瑞幸咖啡下单请求
   ↓
5. 发送模拟HTTP请求到瑞幸咖啡平台
   ↓
6. 接收并解析瑞幸咖啡平台响应
   │
   ├─> 6.1 下单成功
   │     ↓
   │    6.1.1 获取取餐码和二维码
   │     ↓
   │    6.1.2 链接状态变更为"已使用"
   │     ↓
   │    6.1.3 记录成功使用记录
   │
   └─> 6.2 下单失败
         ↓
        6.2.1 链接状态变更为"使用异常"
         ↓
        6.2.2 记录失败原因
         ↓
        6.2.3 生成异常报告
           ↓
7. 更新链接状态历史记录
   ↓
8. 系统检查库存预警条件
   ↓
9. 链接生命周期结束（不可再次使用）
关键节点与处理逻辑：

链接选择算法：
优先选择近期入库的链接（FIFO原则）
考虑链接种类与订单商品的匹配度
考虑有效期因素，优先使用接近过期的链接
链接状态管理：
使用状态锁定机制防止并发问题
完整记录状态变更历史和原因
异常状态需记录详细失败原因
过期处理：
定时任务检查链接有效期
自动将过期链接标记为失效
生成过期统计报告

9.1.1 链接资源管理挑战
并发控制： 在高并发场景下确保链接不被重复分配是关键挑战。 解决方案：
go
// Go实现示例 - 使用乐观锁控制链接分配
func assignLinkWithOptimisticLock(categoryID int64) (*Link, error) {
    tx := db.Begin()
    defer func() {
        if r := recover(); r != nil {
            tx.Rollback()
        }
    }()

    var link Link
    // 使用FOR UPDATE SKIP LOCKED获取一个未使用的链接
    if err := tx.Raw(`
        SELECT * FROM links 
        WHERE category_id = ? AND status = 'unused' AND expires_at > NOW()
        ORDER BY expires_at ASC
        LIMIT 1 FOR UPDATE SKIP LOCKED`, categoryID).Scan(&link).Error; err != nil {
        tx.Rollback()
        return nil, err
    }

    // 更新链接状态
    if err := tx.Model(&link).Update("status", "using").Error; err != nil {
        tx.Rollback()
        return nil, err
    }

    // 创建状态历史记录
    history := LinkStatusHistory{
        LinkID: link.ID,
        Status: "using",
        Reason: "Assigned for order",
    }
    if err := tx.Create(&history).Error; err != nil {
        tx.Rollback()
        return nil, err
    }

    return &link, tx.Commit().Error
}
批量导入性能： 大量链接批量导入时的性能优化。 解决方案：
go
// Go实现示例 - 批量导入优化
func batchImportLinks(links []Link, batchSize int) error {
    total := len(links)
    for i := 0; i < total; i += batchSize {
        end := i + batchSize
        if end > total {
            end = total
        }
        
        batch := links[i:end]
        if err := db.CreateInBatches(batch, len(batch)).Error; err != nil {
            return err
        }
    }
    return nil
}

// Go实现示例 - 请求重试机制
func doWithRetry(fn func() error, maxRetries int, delay time.Duration) error {
    var err error
    for i := 0; i < maxRetries; i++ {
        err = fn()
        if err == nil {
            return nil
        }
        
        // 判断是否是可重试的错误
        if !isRetryableError(err) {
            return err
        }
        
        // 指数退避策略
        sleep := delay * time.Duration(math.Pow(2, float64(i)))
        time.Sleep(sleep)
    }
    return fmt.Errorf("after %d retries: %w", maxRetries, err)
}

func isRetryableError(err error) bool {
    // 网络超时、连接重置等错误可重试
    if netErr, ok := err.(net.Error); ok && netErr.Timeout() {
        return true
    }
    
    // HTTP 5xx 错误可重试
    if respErr, ok := err.(*httpError); ok && respErr.StatusCode >= 500 {
        return true
    }
    
    return false
}

type CacheService struct {
    localCache *sync.Map      // 本地内存缓存
    redisCache *redis.Client  // Redis分布式缓存
}

func (s *CacheService) Get(key string, data interface{}) error {
    // 先查本地缓存
    if val, ok := s.localCache.Load(key); ok {
        // 类型断言和反序列化
        bytes, ok := val.([]byte)
        if ok {
            return json.Unmarshal(bytes, data)
        }
    }
    
    // 查Redis缓存
    val, err := s.redisCache.Get(context.Background(), key).Bytes()
    if err == nil {
        // 缓存命中，更新本地缓存
        s.localCache.Store(key, val)
        return json.Unmarshal(val, data)
    }
    
    // 缓存未命中
    if err != redis.Nil {
        return err
    }
    
    return errors.New("cache miss")
}

func (s *CacheService) Set(key string, data interface{}, expiration time.Duration) error {
    // 序列化数据
    bytes, err := json.Marshal(data)
    if err != nil {
        return err
    }
    
    // 更新本地缓存
    s.localCache.Store(key, bytes)
    
    // 更新Redis缓存
    return s.redisCache.Set(context.Background(), key, bytes, expiration).Err()
}
