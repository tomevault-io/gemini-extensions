## Project Overview

Business Charts is a Grafana panel plugin that wraps Apache ECharts to provide interactive visualizations.
Plugin ID: `volkovlabs-echarts-panel`. Requires Grafana 11 or 12.

## Commands

```bash
# Install
npm install                    # install dependencies (use npm ci in CI)

# Development
npm run dev                    # watch mode build
npm run build                  # production build

# Quality
npm run lint                   # ESLint check
npm run lint:fix               # auto-fix lint issues
npm run typecheck              # TypeScript check (no emit)
npm run markdownlint             # markdownlint-cli2 on AGENTS.md, CHANGELOG.md, README.md
npm run spellcheck               # cspell on all source files

# Tests
npm run test                   # Jest watch mode (only changed files)
npm run test:ci                # Jest with coverage (CI)
npm run test:e2e               # Playwright E2E tests (headless)
npm run test:e2e:docker        # Playwright tests via Docker Compose
npm run test:e2e:dev             # Playwright interactive UI

# Run a single Jest test file
npx jest src/utils/data-frame.test.ts
# Run tests matching a pattern
npx jest --testPathPattern="EchartsPanel"
# Run a single test by name
npx jest -t "Should render panel"

# Local Grafana instance
npm run start:dev    # docker compose (dev profile)
npm run stop         # docker compose down
```

## Architecture

### Plugin entry points

- `src/module.ts` — `PanelPlugin<PanelOptions>` registration; wires editor modes, field config suppression, and option builders
- `src/plugin.json` — plugin metadata and Grafana dependency declarations
- `src/migration.ts` — option migrations between plugin versions

### Two editor modes

The panel exposes two distinct UX paths controlled by `PanelOptions.editor`:

- **CODE** — Monaco editor where users write a JS function returning an ECharts option object (`src/components/EchartsEditor/`)
- **VISUAL** — Form-based builder that generates ECharts options without code
  (`src/components/VisualEditor/`, `src/components/SeriesEditor/`, etc.)

### Main panel rendering

`src/components/EchartsPanel/` is the runtime component. It initializes an ECharts instance (SVG or Canvas
renderer), applies Grafana theme tokens, handles streaming data updates via Grafana Live, and injects user
code with a parameter set built by `src/utils/code-parameter-builder.ts`.

### Types and constants

- `src/types/panel.ts` — top-level `PanelOptions`; other files in `types/` are chart- or map-specific
- `src/constants/default.ts` — default panel options applied on first load
- Path alias `@/*` resolves to `src/*`

### External maps

Three map providers (Baidu, Gaode/AMap, Google Maps) are loaded dynamically. API credentials live in
`PanelOptions` and map initialization is in `src/maps.ts`. Map type constants are in `src/constants/maps.ts`.

### ECharts extensions bundled

`echarts-gl` (3D/WebGL), `echarts-liquidfill`, `echarts-stat`, `echarts-wordcloud` — all imported at the panel
level so they are available in user code.

## Testing Notes

- Jest config (`jest.config.js`) uses `resetMocks: true` and randomizes test order — tests must be fully isolated
- Playwright snapshots are versioned by Grafana release (e.g. `test/panel.spec.ts-snapshots/v12.1/`);
  update with `npm run test:e2e:update`
- Screenshot comparisons allow 3% pixel diff (`maxDiffPixelRatio: 0.03`) to handle chart rendering variance
- Mock external modules at the top of test files: `jest.mock('@grafana/runtime')`
- Define a `getComponent` factory returning rendered components with sensible defaults
- Assert with `screen.getByTestId(TEST_IDS.xxx.yyy)`; centralized test IDs live in `src/constants/tests.ts`
- Use `act()` + `render()` for components with async side effects

## Configuration Tooling

The `.config/` directory is auto-generated by Grafana's plugin scaffolding — do not edit it directly.
Extend configs at the repo root (`jest.config.js`, `tsconfig.json`, `webpack.config.ts`, etc.).

## Code Style

### Formatting (Prettier)

- Print width: **120**, single quotes, trailing commas `es5`, semicolons always
- 2-space indentation, no tabs, end of line `auto`

### Imports

Three groups separated by blank lines, each alphabetized. Destructured members sorted alphabetically:

1. External packages (`@emotion/css`, `@grafana/*`, `react`, `echarts`, etc.)
2. Internal absolute imports (`../../constants`, `../../types`)
3. Sibling/local imports (`./EchartsPanel.styles`, `../EchartsEditor`)

### Naming Conventions

| Element | Convention | Example |
| --- | --- | --- |
| Component files | `PascalCase.tsx` | `EchartsPanel.tsx` |
| Style files | `ComponentName.styles.ts` | `EchartsPanel.styles.ts` |
| Test files | `ComponentName.test.tsx` | `EchartsPanel.test.tsx` |
| Hook files | `useName.ts` | `useDashboardRefresh.ts` |
| Utility files | `kebab-case.ts` | `code-parameters.ts` |
| Constants | `SCREAMING_SNAKE_CASE` | `TEST_IDS`, `DEFAULT_OPTIONS` |
| Enums | PascalCase name, SCREAMING_SNAKE values | `enum EditorMode { CODE = 'code' }` |
| Interfaces | PascalCase | `PanelOptions`, `Props` |
| Functions/variables | camelCase | `getStyles`, `onChangeOption` |
| Barrel exports | Every directory has `index.ts` | `export * from './EchartsPanel'` |

### TypeScript

- **Named exports only** — no default exports.
- Use **enums** (not string unions) for option types. Use `const enum` for internal-only; regular `enum` when
  values are iterated at runtime.
- Use **interfaces** for object shapes and props; use `type` for unions and intersections.
- Prefer explicit generics: `useState<SeriesItem[]>([])`.
- Avoid `as any` in production code; use `as never` for Grafana API type escapes.

### React Components

- **Functional components only** using `React.FC<Props>` with arrow functions.
- Props destructured in the function signature.
- Styles via `@emotion/css` + Grafana's `useStyles2(getStyles)` pattern.
- Style functions: `(theme: GrafanaTheme2) => ({ className: css\`...\` })`.
- Wrap callbacks in `useCallback` with explicit dependency arrays.
- All testable elements must use `data-testid={TEST_IDS.section.element}`.

### JSDoc Comments

Add `/** ... */` blocks above every interface and its properties (with `@type` tags), every function, and
logical sections within function bodies (state, theme, callbacks, return).

### Error Handling

- Use try/catch in async effects; store errors in state and display with Grafana's `<Alert severity="error">`.
- Format: `error instanceof Error ? error.message : \`${error}\``.
- Effects that subscribe must return cleanup functions calling `unsubscribe()`.

## Migration Pattern

When removing or renaming panel options, update `src/migration.ts`:

- Define deprecated fields in `OutdatedPanelOptions` with JSDoc noting the removal version.
- Use spread destructuring: `const { legacyField, ...actualOptions } = panel.options`.
- Use `semver.lt()` for version-gated migrations.
- Add corresponding tests in `migration.test.ts`.

  (e.g. `@ci-cd-workflows/v7.0`). All other GitHub Actions should be pinned to SHAs.

  before committing.

  `cspell.config.json` if they are legitimate.

## Changelog Policy

Add entries under the current `[Unreleased]` section in `CHANGELOG.md`. Categorize under `### Added`,
`### Changed`, `### Removed`, `### Fixed`, or `### Project Updates`.

## Branching Policy

- **Never commit directly to `main`.** Always create a new branch for changes.
- Use descriptive branch names (`feat/add-feature`, `fix/bug-description`).
- **Always create pull requests as drafts** (`gh pr create --draft`).
- When checking out a branch or `main`, always `git fetch` and `git pull` first.
- **Always run `git status`** before constructing `git add` commands. Only add files that are unstaged or untracked.

## PR Summary Policy

- **Wrap PR summary lines at 120 characters.**
- **Use categories** to organize changes: `### Added`, `### Fixed`, `### Changed`, `### Removed`,
  `### Dependencies`, `### CI/CD`, `### Documentation`, `### Tooling`.
- Always include a `## Test plan` section with a checklist of verification steps.
