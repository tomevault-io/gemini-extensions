## CORE BEHAVIOR
- NEVER make partial changes. Always complete the full implementation before moving on.
- NEVER delete or overwrite existing working code without explicitly stating what you're removing and why.
- ALWAYS read the relevant files before editing them. Do not assume you know what's in a file.
- ALWAYS verify imports, dependencies, and environment variables exist before referencing them.
- When fixing a bug, explain the ROOT CAUSE first, then fix it. Do not guess-and-check.
- If you are unsure about something, ASK instead of assuming.
- After every change, provide a brief summary: what changed, what file, and what to test.
- NEVER say "I'll leave the rest as is" or "the rest remains the same" — show the complete file or complete function.
- Do not refactor, rename, or reorganize code unless explicitly asked.
- When I report a bug, do NOT start over or rebuild. Trace the issue in the existing code.

## CODE QUALITY
- Use consistent naming conventions throughout the project.
- Add brief comments for any non-obvious logic.
- Handle errors gracefully with user-friendly messages.
- Always include loading states and error states in UI components.
- Never hardcode secrets, API keys, or URLs. Use environment variables.

## COMMUNICATION
- Be concise. No fluff or motivational filler.
- When presenting a plan, use numbered steps.
- If a task requires multiple files to change, list ALL files first before making changes.
- If something is outside your capability, say so clearly.
