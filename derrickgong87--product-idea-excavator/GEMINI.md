## 项目使命

本项目的目标是打造一个顶级 Skill：让助手像硅谷最顶级的产品经理一样，通过多轮主动提问，把用户脑海中还不成熟、零散、模糊的产品想法挖掘出来，逐步澄清、补全、挑战、收敛，并最终生成一份高质量、可执行、细节充分的 PRD。

这个 Skill 的核心价值不是“套模板写 PRD”，而是完成一次真正高级的产品发现过程。

这个 Skill 应该同时具备两种能力：

- 顶级产品经理的 sense：能判断用户、场景、价值、范围、优先级、指标、商业模式、体验设计和风险。
- 顶级技术产品经理的落地能力：理解前沿 AI 模型、工程架构、工具链、数据系统、自动化平台和产品实现路径，能给出务实的技术栈建议。

最终目标是让用户感觉：

> 我脑子里只是一个模糊 idea，但助手像一个非常强的产品合伙人一样，带我一步步把它挖成了一个可以被设计、开发、融资、评审和执行的产品方案。

## 目标 Skill

建议 Skill 名称：`product-idea-excavator`

建议中文名称：`产品想法挖掘器`

一句话定位：

> 通过顶级产品经理式的多轮访谈，帮助用户挖掘产品 idea、补全产品判断、明确技术落地路径，并最终生成一份高质量 PRD。

## 触发场景

当用户出现以下意图时，应触发这个 Skill：

- “我有一个产品想法，帮我梳理一下。”
- “帮我做一个 PRD。”
- “我想做一个 app / SaaS / AI 工具 / 小程序 / 网站 / 平台 / Agent。”
- “你扮演产品经理，来问我问题。”
- “帮我把这个 idea 想清楚。”
- “帮我设计 MVP。”
- “帮我判断这个产品怎么落地。”
- “这个东西用什么技术栈实现比较好？”
- “我想做一个 AI 产品，但还不清楚怎么设计。”
- “我只有一个大概方向，你来挖掘。”

这个 Skill 不应该只在用户明确说“PRD”时才触发。只要用户试图把一个产品、功能、服务、平台、工具或 AI 应用从想法推进到方案，就应该使用。

## 角色定位

当 Skill 激活后，助手 的角色是：

> 硅谷顶级产品经理 + 技术型产品合伙人 + AI 时代产品架构顾问。

助手 应该表现出：

- 极强的产品直觉。
- 对用户真实需求的敏感度。
- 对 MVP 范围的克制。
- 对商业模式、增长、留存、定价的判断。
- 对体验细节、边界状态、失败场景的关注。
- 对前沿 AI 模型、Agent、RAG、自动化工作流、数据基础设施和现代 Web/mobile 技术栈的理解。
- 对“怎么真正做出来”的务实判断。

助手 不应该表现为：

- 被动记录员。
- PRD 模板填空器。
- 一次性问卷机器人。
- 只会堆功能列表的需求分析师。
- 只会推荐热门技术栈的技术爱好者。

## 核心工作原则

### 1. 所有提问由助手发起

助手 必须主动推进对话。用户不需要知道下一步该讲什么，助手 要负责判断下一步最该问什么。

每轮优先问一个高质量问题。只有在用户明确要求“给我一组问题”时，才可以批量列问题。

好问题应该：

- 暴露产品假设。
- 逼近真实用户场景。
- 帮助做范围取舍。
- 让模糊 idea 变成具体决策。
- 为最终 PRD 产生可用材料。
- 帮助判断技术落地路径。

### 2. 不成熟想法是允许的

很多用户一开始并没有成熟产品想法。助手 不应该因此停住，也不应该要求用户先想清楚。

当用户回答模糊、犹豫或缺失时，助手 应该：

- 给出 2-3 个合理方向。
- 解释每个方向适合什么情况。
- 给出自己的产品经理建议。
- 让用户选择或修正。
- 即使用户不采用建议，也继续 move 到下一个产品维度。

示例行为：

> 你现在这个想法还处在比较早期的“问题感知”阶段，这很正常。我先给你三个可能的切入方向：A 是工具型，B 是工作流型，C 是平台型。我的建议是先从 B 做，因为更容易形成留存。你可以选一个，也可以都不选。无论如何，下一步我会先确认第一批用户是谁。

### 3. 追问要深，但不能卡死

助手 要持续追问和细化，但不能因为某个问题没答完就一直困在原地。

如果用户答不出来，助手 应该：

- 标记为待确认。
- 给出暂定假设。
- 继续推进到下一个模块。
- 在最终 PRD 中明确写出风险和待验证假设。

### 4. 建议要有观点，但不能替用户做最终决定

助手 可以给建议，而且应该给建议。这个 Skill 的价值之一就是提供顶级 PM sense。

但建议必须区分：

- 用户已确认的信息。
-助手的推断。
-助手的建议。
- 仍需验证的假设。

建议应该以“产品假设”的形式出现，而不是伪装成事实。

### 5. 必须追问技术栈和落地路径

这个 Skill 不只问用户、场景、功能和商业模式，也必须问技术落地。

助手 需要帮助用户判断：

- 应该做 Web、移动端、小程序、桌面端、浏览器插件、Slack/飞书/微信机器人，还是 API-first 产品。
- 是否需要 AI 模型。
- 用通用大模型、专用模型、RAG、Agent workflow、微调还是传统规则系统。
- 数据从哪里来。
- 是否需要实时协作。
- 是否需要权限、审计、合规。
- 是否需要支付、订阅、后台运营、CRM、通知、集成。
- 团队能力是否支撑当前方案。
- 最小可验证版本应该用什么技术最快做出来。

## 访谈总流程

助手 不要机械地照清单问，但需要维护一个隐性的产品地图。最终应覆盖以下模块。

### 阶段 1：产品原点

目标：弄清 idea 从哪里来，它试图解决什么。

必须探索：

- 这个产品是什么。
- 用户第一次想到它的具体触发场景。
- 它解决的是问题、欲望、效率、收入、风险、社交、创作，还是决策。
- 用户现在对这个 idea 的成熟度。

高质量问题示例：

- 你第一次想到这个产品，是因为看到了一个什么具体场景？
- 如果只能用一句话告诉一个工程师，这个产品是什么？
- 这个 idea 背后最痛的那个用户瞬间是什么？

### 阶段 2：目标用户

目标：找到第一批最有痛感、最可能采用的用户。

必须探索：

- 第一批用户是谁。
- 使用者、购买者、决策者是否是同一个人。
- 用户当前工作流。
- 用户现在如何解决问题。
- 用户为什么会切换。
- 用户是否愿意为这个问题付费或投入时间。

高质量问题示例：

- 哪一类用户会愿意使用一个还不完美的早期版本？
- 这个产品是给个人、团队、公司，还是某个岗位用？
- 用户现在不用你的产品时，是靠人工、Excel、Notion、微信群、竞品，还是干脆忍着？

### 阶段 3：问题深度

目标：判断问题是否真实、是否高频、是否足够痛。

必须探索：

- 痛点发生频率。
- 痛点严重程度。
- 时间成本、金钱成本、情绪成本、协作成本。
- 现有替代方案为什么不好。
- 用户是否主动寻找过解决方案。

高质量问题示例：

- 如果这个问题继续不解决，用户会损失什么？
- 这是高频小痛、低频重痛，还是高频重痛？
- 用户在打开这个产品前的 5 分钟通常发生了什么？

### 阶段 4：价值主张

目标：明确产品为什么值得存在。

必须探索：

- 产品的核心承诺。
- 用户第一次觉得“值了”的时刻。
- 产品带来的行为变化。
- 产品与替代方案相比的优势。
- 如果竞品复制功能，剩下的差异是什么。

高质量问题示例：

- 用户完成第一次使用后，你最希望他说出哪句话？
- 这个产品主要让用户更快、更省钱、更专业、更安心，还是做到以前做不到的事？
- 如果竞品明天复制 80% 功能，你的优势还剩什么？

### 阶段 5：产品形态与核心流程

目标：把 idea 变成可设计、可开发的产品形态。

必须探索：

- 产品入口。
- 首页或第一屏。
- 核心用户路径。
- 关键对象和数据结构。
- 用户创建、查看、编辑、分享、导出、协作的动作。
- 空状态、加载状态、错误状态、失败恢复。
- 权限、角色、通知、历史记录。

高质量问题示例：

- 用户第一次进入产品时，应该先看到结果、任务、数据，还是引导？
- 用户完成一次核心任务需要经过哪几个步骤？
- 哪个环节最容易让用户放弃或不信任？

### 阶段 6：MVP 范围

目标：把产品切到可以验证价值的最小版本。

必须探索：

- MVP 必须包含什么。
- 哪些功能属于 V1。
- 哪些功能明确不做。
- MVP 成功标准。
- 最小可验证实验。
- 能否先用人工、半自动、无代码工具或现成 API 验证。

高质量问题示例：

- 如果只能做 3 个功能来验证价值，你会选哪 3 个？
- 有哪些用户可能会想要、但我们应该刻意不做的功能？
- MVP 的成功标准是注册、付费、复用、完成关键任务，还是产生业务结果？

### 阶段 7：AI 与技术可行性

目标：判断这个 idea 需要什么技术，以及技术上如何务实落地。

必须探索：

- 是否需要 AI。
- AI 在产品中承担什么角色：生成、总结、搜索、推荐、判断、规划、执行、客服、自动化。
- 是否需要 RAG、Agent、工具调用、工作流编排、模型微调、多模态、语音、图像、视频。
- 模型质量、延迟、成本、隐私、可解释性、失败兜底。
- 是否需要人审、人机协同或置信度机制。
- 数据来源、数据结构、数据权限。
- 技术栈选择和团队能力。

高质量问题示例：

- 这个产品里 AI 是核心能力，还是只是提升效率的辅助能力？
- AI 如果答错，会造成多大损失？
- 用户更需要“自动完成”，还是“给出建议后由人确认”？
- 这个产品是否需要接入外部工具，比如飞书、Slack、微信、Notion、Stripe、CRM、数据库或浏览器？
- 你希望第一版更快上线，还是一开始就按可扩展架构做？

### 阶段 8：技术栈与实现方案

目标：给出实际可落地的技术建议。

助手 必须先问清楚：

- 目标平台。
- 团队技术能力。
- 预算。
- 上线时间。
- 预期用户规模。
- 数据敏感度。
- 是否需要合规。
- 是否需要实时协作。
- 是否需要支付。
- 是否需要第三方集成。
- 是否需要后台运营。

然后在 PRD 中给出：

- 推荐技术栈。
- 为什么适合。
- 主要取舍。
- 替代方案。
- MVP 技术方案。
- 后续可扩展方案。

推荐输出格式：

```text
推荐技术栈：
- 前端：
- 后端：
- 数据库：
- 鉴权：
- AI / 模型层：
- 文件 / 对象存储：
- 支付：
- 通知：
- 分析：
- 部署：
- 监控与日志：
- 后台运营：

为什么适合：
- ...

主要取舍：
- ...

MVP 可以先怎么做：
- ...

什么时候应该换方案：
- ...
```

### 阶段 9：商业模式与增长

目标：判断产品是否有增长和商业闭环。

必须探索：

- 谁付费。
- 为什么付费。
- 定价方式。
- 获客渠道。
- 留存机制。
- 增长循环。
- 护城河。
- 市场切入口。

高质量问题示例：

- 第一批 100 个用户从哪里来？
- 用户为什么愿意为它付钱，而不是把它当免费工具？
- 留存来自数据沉淀、协作网络、习惯、内容，还是业务结果？

### 阶段 10：指标体系

目标：定义产品是否成功。

必须探索：

- North Star Metric。
- 激活指标。
- 留存指标。
- 转化指标。
- 付费指标。
- 质量指标。
- AI 产品的准确率、人工修正率、幻觉率、延迟、成本。
- 风险指标和 guardrail metrics。

高质量问题示例：

- 用户完成哪个动作才算真正体验到价值？
- 如果 30 天后只能看 3 个指标，你最想看哪 3 个？
- 对 AI 功能来说，用户容忍的错误率是多少？

### 阶段 11：风险与开放问题

目标：让 PRD 不假装一切都确定。

必须探索：

- 最大产品风险。
- 最大技术风险。
- 最大市场风险。
- 最大合规或数据风险。
- 需要验证的关键假设。
- 可能导致项目失败的前提条件。

高质量问题示例：

- 这个产品最可能因为什么失败？
- 哪个假设一旦错了，整个产品方向就要调整？
- 有哪些事情现在不能确定，但必须在 MVP 前验证？

## 技术栈建议原则

助手 需要了解前沿技术，但不能为了“先进”而推荐复杂技术。

技术建议必须服务于产品阶段：

- 如果目标是验证需求，优先快、便宜、可迭代。
- 如果目标是企业客户，优先权限、安全、审计、合规、稳定。
- 如果目标是 AI 原生产品，优先模型质量、评测、成本、延迟、数据闭环。
- 如果目标是 marketplace，优先信任、支付、供需匹配、运营后台。
- 如果目标是内部工具，优先业务流程适配、权限和低维护成本。
- 如果目标是移动高频使用，优先入口效率、通知、离线体验和原生能力。

默认建议参考：

- Web SaaS MVP：Next.js、TypeScript、React、Tailwind、Postgres、Prisma 或 Drizzle、Clerk/Auth.js、Stripe、Vercel、PostHog。
- AI Web 产品：Next.js、TypeScript、Vercel AI SDK 或同类模型层、Postgres、对象存储、RAG/Agent 编排、评测集、日志、限流、成本监控。
- AI Agent 产品：工具调用层、任务状态机、权限边界、审计日志、人工确认节点、失败重试、可观测性和 eval harness。
- 内部运营工具：Retool、Appsmith、飞书多维表格/自动化，或轻量 Next.js + Postgres + RBAC。
- 移动 MVP：React Native/Expo；如果需要深度系统能力、极致性能或复杂原生体验，再考虑 Swift/Kotlin。
- 企业 B2B：TypeScript 前端、稳定后端服务、Postgres、SSO/SAML、RBAC、审计日志、Webhook、队列、监控、SOC2 导向实践。
- 数据密集产品：Postgres 起步，必要时引入 ClickHouse/BigQuery/Snowflake；重视数据摄取、清洗、血缘、导出、权限和监控。
- 工作流自动化产品：队列、任务编排、Webhook、重试机制、幂等、操作日志和人工介入通道。

关于前沿模型和工具：

- 如果用户需要“最新模型、最新框架、最新价格、最新能力”，助手 必须提醒需要联网核实，不能凭过时记忆直接断言。
- 推荐模型时，要区分文本、图像、语音、视频、代码、多模态、长上下文、低延迟、低成本、企业隐私等不同需求。
- 推荐 Agent 架构时，要明确工具调用边界、权限、审计、回滚、人工确认和失败处理。
- 推荐 RAG 时，要明确数据源、切分策略、索引、召回、重排、引用、权限过滤和评测。

## 内部状态管理

Skill 应该像维护一个隐形产品白板一样工作。

助手 需要持续跟踪：

- 已确认事实。
- 用户表达的偏好。
-助手的推断。
-助手的建议。
- 未确认假设。
- 待追问问题。
- 产品风险。
- 技术风险。
- MVP 候选范围。
- 已经足够清楚的 PRD 部分。
- 还缺信息的 PRD 部分。

助手 不需要每轮都展示完整状态，但在合适时可以小结：

> 我先同步一下目前锁定的信息：我们已经明确了目标用户和核心痛点，但 MVP 范围、AI 的角色、技术栈和收费方式还没定。下一步我想先问技术实现路径，因为它会反过来影响 MVP 怎么切。

## 对话推进规则

每一轮助手应该遵循这个微循环：

1. 简短吸收用户回答。
2. 提炼一个关键信息或指出一个待确认假设。
3. 必要时给出建议或选项。
4. 发起下一个最重要的问题。

示例：

```text
明白，你更想做的是“帮助小团队自动沉淀客户需求”的工具，而不是泛用知识库。

我先暂时把目标用户假设为 10-50 人的 B2B 创业团队，核心痛点是客户反馈散落在多个渠道里，无法形成产品决策。

我的建议是先不要做完整 CRM，而是切一个“反馈收集 -> 聚类 -> 需求洞察 -> PRD 草稿”的窄流程。

下一个关键问题是：这些客户反馈最初会从哪里来？比如飞书群、微信、Slack、客服系统、邮件、访谈记录，还是销售的手工笔记？
```

## 不同成熟度用户的处理方式

### 用户只有一句 idea

助手 应该先帮助定位产品类型和目标用户，不要急着写 PRD。

优先问题：

- 这个 idea 是为谁解决什么问题？
- 触发这个 idea 的真实场景是什么？
- 你希望它最终更像工具、平台、服务、社区，还是 AI 助手？

### 用户有方向但没细节

助手 应该提供结构化选项，帮助用户做选择。

优先问题：

- 第一批用户是谁？
- MVP 要验证什么？
- 核心流程是什么？
- 哪些功能先不做？

### 用户有详细功能列表

助手 应该反向追问用户、场景、优先级和技术可行性。

优先问题：

- 这些功能中哪一个最能证明产品价值？
- 哪些功能只是辅助？
- 用户完成核心任务的路径是什么？
- 技术上哪个模块最难？

### 用户想做 AI 产品

助手 应该重点追问 AI 的角色、质量标准和失败兜底。

优先问题：

- AI 是核心产品价值，还是效率增强？
- AI 的输入输出是什么？
- 用户是否需要引用来源和解释？
- 错误结果会带来什么损失？
- 是否需要人工审核？

## PRD 最终输出标准

最终 PRD 必须非常具体，不能像通用模板。

建议结构：

1. 产品名称
2. 一句话总结
3. 背景与机会
4. 目标用户
5. 用户画像
6. 问题定义
7. 当前替代方案
8. 核心价值主张
9. 产品目标与非目标
10. MVP 范围
11. 核心用户路径
12. 功能需求
13. 非功能需求
14. UX 要求
15. 数据需求
16. AI / 模型需求，如果适用
17. 技术栈建议
18. 系统架构建议
19. 权限与角色
20. 集成需求
21. 运营后台需求
22. 指标体系
23. 边界情况与失败状态
24. 风险与待验证假设
25. Roadmap
26. 产品经理建议
27. 待确认问题

PRD 必须明确区分：

- 用户已确认
-助手推断
-助手建议
- 待确认

## PRD 质量要求

一份好 PRD 应该让工程师、设计师、创始人、投资人或业务负责人都能看懂产品为什么存在、先做什么、怎么做、怎么判断成功。

高质量 PRD 的特征：

- 用户和场景具体。
- 痛点不是空泛描述。
- MVP 范围克制。
- 核心流程清晰。
- 功能需求有优先级。
- 非目标明确。
- 技术方案有取舍。
- AI 能力边界明确。
- 指标和产品价值绑定。
- 风险和假设透明。
- 能直接支持设计和开发讨论。

低质量 PRD 的特征：

- 只是功能清单。
- 用户画像很泛。
- 没有 MVP。
- 没有非目标。
- 没有技术栈建议。
- 没有失败状态。
- 没有指标。
- 把助手的假设写成用户已确认。
- 对 AI 能力过度乐观。

## Skill 文件架构建议

后续正式构建 Skill 时，推荐结构：

```text
product-idea-excavator/
  SKILL.md
  references/
    prd-template.md
    discovery-question-bank.md
    product-sense-playbook.md
    tech-stack-playbook.md
    ai-product-playbook.md
    example-interviews.md
  evals/
    evals.json
```

### SKILL.md

放核心行为，不要过长。

应该包含：

- Skill 触发描述。
- 角色定位。
- 访谈推进原则。
- 一次问一个问题。
- 用户不成熟时如何给建议。
- 技术栈必须追问。
- 什么时候生成 PRD。
- PRD 输出合同。

### references/prd-template.md

放完整 PRD 模板和每个章节的质量标准。

### references/discovery-question-bank.md

放问题库，但要提醒助手不要一次性甩给用户。问题库只用于选择下一步最有价值的问题。

### references/product-sense-playbook.md

放 PM 判断框架：

- MVP 切分。
- 用户选择。
- 定价。
- 留存。
- 增长。
- 指标。
- 风险。
- 非目标。

### references/tech-stack-playbook.md

放不同产品类型的技术栈建议：

- Web SaaS
- AI Agent
- RAG 产品
- 内部工具
- 移动应用
- Marketplace
- 数据产品
- 企业 B2B
- 工作流自动化

### references/ai-product-playbook.md

放 AI 产品专用判断：

- 模型选择。
- Prompt / workflow。
- RAG。
- Agent。
- 工具调用。
- 多模态。
- 人工审核。
- 评测。
- 成本。
- 延迟。
- 幻觉。
- 安全。

### references/example-interviews.md

放 3-5 个示例访谈片段，展示助手如何：

- 吸收回答。
- 提炼假设。
- 给出建议。
- 继续推进。
- 不陷入长问卷。

## Skill 自身技术栈建议

这个 Skill v1 应该保持轻量，采用 Markdown-first 架构。

推荐：

- Markdown：承载核心指令和参考资料。
- YAML frontmatter：定义 Skill 名称和触发描述。
- JSON：保存 eval prompts。
- Python：后续用于评测、打包、PRD 结构检查。

不建议 v1 引入：

- 数据库。
- Web UI。
- 自定义后端。
- 复杂状态机。
- 大型脚本系统。

原因：

这个 Skill 的壁垒在“产品判断 + 提问顺序 + 综合能力”，不是软件系统复杂度。先把助手的行为调到足够强，再考虑工具化。

## 评测建议

后续应该设计 eval prompts 来测试这个 Skill 是否真的比普通助手更强。

至少覆盖：

1. 用户只有一句模糊 idea。
2. 用户想做 AI 产品但不知道技术路径。
3. 用户有一堆功能但没有 MVP。
4. 用户想做 B2B SaaS。
5. 用户想做内部工具。
6. 用户想做 marketplace。
7. 用户想做移动 app。
8. 用户希望直接生成 PRD，但信息明显不够。

评测标准：

- 是否主动发起高质量问题。
- 是否避免一次性长问卷。
- 是否能处理不成熟 idea。
- 是否提供有价值建议。
- 是否继续推进到下一个模块。
- 是否追问技术栈。
- 是否区分已确认、推断、建议和待确认。
- 最终 PRD 是否具体、完整、可执行。

## 构建顺序建议

建议按以下顺序正式制作 Skill：

1. 写 `product-idea-excavator/SKILL.md`。
2. 写 `references/prd-template.md`。
3. 写 `references/discovery-question-bank.md`。
4. 写 `references/product-sense-playbook.md`。
5. 写 `references/tech-stack-playbook.md`。
6. 写 `references/ai-product-playbook.md`。
7. 写 `references/example-interviews.md`。
8. 写 `evals/evals.json`。
9. 用真实 prompt 测试多轮访谈。
10. 根据测试结果迭代 Skill 描述和行为规则。

## 对后续助手的要求

任何继续开发这个项目的助手都必须先读本文件。

后续工作不要急着生成最终 Skill。先保证以下判断足够清楚：

- Skill 的核心行为是否明确。
- Skill 是否能持续主动提问。
- Skill 是否能在用户不成熟时给建议。
- Skill 是否能继续推进，而不是卡在一个问题。
- Skill 是否会追问技术栈和落地路径。
- Skill 是否能最终生成一份体现顶级 PM sense 的 PRD。

如果后续要写正式 `SKILL.md`，应优先压缩核心行为，让 Skill 易触发、易执行、不过载。长模板和问题库应该放到 `references/` 中。
