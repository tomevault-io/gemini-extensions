# CLAUDE.md

本文件约束 Claude Code 在本仓库中的默认工作方式。除非用户明确要求，否则后续所有修改都按本文件执行。

## 1. 工作原则

- 先读后改：修改任何文件前，先阅读相关实现，理解当前逻辑和节点注册方式。
- 小改优先：只做用户明确要求的改动，不顺手重构、不额外加功能、不改无关代码。
- 尽量复用现有模式：优先沿用仓库里已有节点、配置、请求、注册、命名和返回结构。
- 保持中文语境：新增用户可见文案、日志、tooltip、错误提示时，默认保持与现有中文风格一致。
- 改完自检：检查是否破坏节点注册、输入输出类型、配置读取、图像张量格式。

## 2. ComfyUI 节点开发规范

新增或修改节点时，优先遵循仓库现有实现模式，例如 [banana2_aabao_node.py](banana2_aabao_node.py) 与 [__init__.py](__init__.py)。

### 2.1 节点类基本结构

每个节点类应包含：

- `INPUT_TYPES`
- `RETURN_TYPES`
- `RETURN_NAMES`
- `FUNCTION`
- `CATEGORY`

如该文件使用 `NODE_CLASS_MAPPINGS` / `NODE_DISPLAY_NAME_MAPPINGS` 自注册，则保持该模式；
如该节点由仓库根 [__init__.py](__init__.py) 统一汇总注册，则同步更新对应映射，不能只写类不注册。

### 2.2 分类与显示名

- `CATEGORY` 必须使用 dapaoAPI 体系，保持与现有节点分组一致。
- 不强行统一成单一分类；应参考相邻节点现有分类路径，例如 `🤖dapaoAPI/Nano Banana 2`。
- 显示名称延续仓库当前风格：中文名称 + 渠道/模型说明，必要时保留 `@炮老师的小课堂` 风格。

### 2.3 输入输出约定

- 图像输入张量默认按仓库现有约定处理：`[B, H, W, C]`。
- 遮罩输入张量默认使用：`[B, H, W]`。
- 图像值范围保持 `0-1`、`float32`。
- 图像与 PIL 转换优先复用现有写法，例如 `pil2tensor()`、`tensor2pil()`。
- 新增输出时，必须核对 `RETURN_TYPES`、`RETURN_NAMES` 与函数真实返回值一一对应。

## 3. 仓库结构与注册规范

- 根目录 [__init__.py](__init__.py) 是节点聚合与注册核心，新增节点若需要对外暴露，必须检查这里是否要同步导入和合并映射。
- 优先保持“单文件实现节点 + 根 `__init__.py` 汇总”的当前结构，不随意引入新的目录层级。
- 不创建无必要的新文件；能在现有文件内完成的，不额外拆分。

## 4. 配置、路径与资源规范

### 4.1 密钥与配置

- 严禁把真实 API Key、Token、Cookie、账号信息写入任何 Git 跟踪文件。
- 只提交示例配置，如 `*.example.json`；本地真实配置应继续放在已忽略的 `*_config.json` / `config.json` 中。
- 读取配置时，优先兼容仓库现有模式；不要为了“更通用”重写整套配置系统。

### 4.2 路径处理

- 严禁硬编码用户机器绝对路径。
- 引用仓库内资源时使用相对路径或基于当前文件位置构造路径。
- 涉及 ComfyUI 模型/资源目录时，优先使用 ComfyUI 提供的路径机制；不要假设固定安装位置。

### 4.3 下载与文件生成

- 下载前先检查目标是否已存在，以及当前逻辑是否允许覆盖。
- 不默认新增持久化缓存、日志文件、临时文件清理逻辑，除非任务明确要求。

## 5. API 请求与安全规范

- 沿用现有请求库和模式；若当前文件已使用 `requests`，不要无故改成别的客户端。
- 不输出敏感信息到日志，尤其是完整密钥、Authorization 头、Cookie、原始凭证。
- 调试日志可以保留必要的接口、状态码、响应摘要，但应避免泄露凭证和用户隐私内容。
- 修改请求参数时，优先最小化改动，避免连带改变已有渠道兼容性。

## 6. 代码风格规范

- 优先跟随仓库现有风格，而不是套用外部统一模板。
- 不额外补充大段注释、类型标注、抽象层、工具函数，除非当前修改确实需要。
- 三五行能说清的逻辑，不强行提炼 helper。
- 不因为“看起来更优雅”就改变量名、文件名、排序或格式。
- 新增 import 必须与实际改动直接相关，避免顺手整理全文件 import。

## 7. 测试与验证要求

- 若改动涉及节点输入输出、接口参数、图像处理或注册逻辑，至少做静态自检：
  - 节点是否已注册
  - 返回值数量和类型是否匹配
  - 可选输入是否安全读取
  - 失败分支是否仍返回 ComfyUI 可接受的数据结构
- 若仓库已有可直接运行的相关测试，优先运行与改动直接相关的测试，不扩大验证范围。
- 如果无法实际运行 UI 或 ComfyUI 流程，必须明确说明“已做代码级检查，但未实际在 ComfyUI 中点测”。

## 8. Git 与文件安全

- 不主动提交、不主动建分支、不主动推送，除非用户明确要求。
- 不覆盖用户未说明的现有改动。
- 发现疑似敏感文件未被忽略时，先提醒用户，再决定是否修改 `.gitignore`。

## 9. 本仓库的默认执行指令

当用户要求“改一下”“修一下”“按这个做”但没有补充细节时，Claude Code 默认按以下方式执行：

1. 先查找相关节点或注册入口。
2. 阅读将被修改的文件与相邻实现。
3. 仅做满足需求所需的最小修改。
4. 检查 `__init__.py` 注册、输入输出定义、配置和图像处理是否一致。
5. 向用户汇报具体改动位置，并明确是否完成了实际验证。

## 10. 本文件优先级

- 用户当前消息中的明确要求，优先级高于本文件。
- 若本文件与代码现状冲突，以代码现状和用户目标为准，并在修改时尽量延续仓库真实模式。
- 后续若用户继续强调新的协作规则，应直接更新本文件，使其成为最新准则。
