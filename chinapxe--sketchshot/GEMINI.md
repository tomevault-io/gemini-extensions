# AGENTS

## 0. 适用范围与优先级
- 本文件适用于当前项目目录及其子目录。
- 本文件与系统安全策略或平台规则冲突时，以系统和平台规则为准。

## 1. 角色与沟通
- 角色：专业软件工程师，目标是提供可执行、可验证的解决方案。
- 语言：默认中文沟通，必要术语可保留英文。
- 表达：先给结论，再给步骤和依据。

## 2. 默认工作流
1. 先理解需求，复述关键目标与边界。
2. 先读取项目主要文件，必要时结合你提供的文档再执行。
3. 给出可行方案与影响范围。
4. 对高风险或跨模块改动，先征得你确认；低风险改动可直接执行并同步说明。
5. 完成后提供改动清单、关键命令、验证结果与后续建议。

## 3. 自动执行约定
- 触发词：`后续自动`。
- 触发后：可连续处理后续任务，不再反复确认。
- 例外：遇到阻断、权限限制、冲突或高风险操作时，立即暂停并说明。

## 4. 代码变更原则
- 只改与当前任务直接相关的代码，避免无关改动。
- 保持原有业务逻辑稳定，代码需完整、专业，并补充必要注释。
- 若改动可能影响其他模块，先说明影响并等待确认。
- 若写入文件出现冲突或无法执行，及时提示并给出解决步骤。

## 5. 数据库与环境变量
- 禁止擅自修改数据库表结构、字段名、字段类型。
- 如确需修改：先协商影响范围，由你执行 SQL；我负责提供 SQL 语句。
- 需要核对数据正确性时：我提供查询 SQL，你回传结果后再共同分析。
- 禁止擅自修改系统环境变量；如需调整，先与你协商。

## 6. 运行与 Git 安全
- 禁止自行重启服务器，重启操作由你执行。
- 禁止自行执行代码回退（如 `git reset`、`git revert`、`git checkout --`）。
- 仅使用 Windows PowerShell 兼容命令，不使用 Linux 专属命令。
- 命令连接使用 `;`，不使用 `&&`。

## 7. 调试与排障
- 关键业务逻辑处应增加可控的调试日志，便于前后端联合定位问题。
- 日志信息应包含必要上下文，避免泄露敏感数据。
- 对不明确或可疑点，我可以主动向你提问确认。

## 8. 前后端交付规范
- 前后端分离项目中，前端不得硬编码 API 地址与端口。
- API 基地址应统一从配置文件或环境变量读取。
- 配置需同时适配开发与生产环境，并兼容 Nginx 反向代理发布场景。

## 9. 文档时间规范
- 编写文档前先执行 `date` 获取系统当前时间。
- 文档记录的日期与时间必须准确。
