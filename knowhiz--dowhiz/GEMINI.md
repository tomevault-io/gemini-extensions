# Repository Guidelines

## Operating Rules

- Correct weak or unsupported user requests and assumptions directly and politely; do not agree by default when the evidence points elsewhere.
- Do not skip core steps for debugging, implementation, refactors, or research: inspect the relevant files first, form an explicit plan, execute it, and verify the result.
- Do not claim completion without validation. If validation was not possible, say exactly what remains unverified and why.
- Always surface material assumptions, risks, and edge cases.
- Prefer thoroughness over speed when correctness or production impact matters.
- When changing code, run the most relevant available checks/tests, or state exactly why they were not run.
- When needed information is missing, state exactly what is missing instead of guessing. Only make clearly labeled, low-risk assumptions when necessary to proceed.
- Keep outputs concise, but never omit load-bearing reasoning, validation steps, or unresolved risks.

## Overall Guidance

- `external/` is reference-only. Do not modify files under it.
- For challenging tasks, create a branch from latest `dev`, commit in clear steps, and push step by step with clear status notes for traceability.
- Keep docs synchronized with code changes in the same PR.

## Runtime Configuration Policy (Source of Truth)

- Runtime services load **unprefixed** keys from `DoWhiz_service/.env`.
- VM `.env` is generated by merging secret sets:
  - staging: `ENV_COMMON + ENV_STAGING`
  - production: `ENV_COMMON + ENV_PROD`
- I will also point you the latest local path for `ENV_COMMON`; `ENV_STAGING`; and `ENV_PROD` (in local files named as `.env.common`, `.env.staging`, and `.env.prod` accordingly). For loading keys in prod VM we follow same logic as in `.github/workflows/CICD-production.yml`; for loading keys in staging VM we follow same logic as in `.github/workflows/CICD-staging.yml`.
- Runtime `.env` must not contain `STAGING_*` or `PROD_*` keys.
- `DEPLOY_TARGET` is optional and used for runtime policy decisions (for example `run_task` backend auto behavior), not for shell-side key remapping.

Notes on legacy aliases:
- Some ingestion/raw-payload paths also read `SCALE_OLIVER_*` aliases via code fallback logic.
- Do not rely on alias-only keys in new deployments; keep unprefixed keys authoritative.

## Project Structure

- `DoWhiz_service/`: Rust backend (inbound gateway, worker service, scheduler, task runner, adapters).
- `website/`: React 19 + Vite site.
- `assets/`, `reference_documentation/`, `example_files/`: docs/assets/support files.

## Build / Run

Backend (repo root):
```bash
./DoWhiz_service/scripts/run_employee.sh little_bear 9001 --skip-hook --skip-ngrok
# or
EMPLOYEE_ID=little_bear RUST_SERVICE_PORT=9001 \
  cargo run -p scheduler_module --bin rust_service -- --host 0.0.0.0 --port 9001
```

Gateway (separate process):
```bash
./DoWhiz_service/scripts/run_gateway_local.sh
```

Frontend:
```bash
cd website
npm install
npm run dev
```

## Coding Style

- Rust: `cargo fmt`, `cargo clippy --all-targets --all-features`.
- Web: `npm run lint`.
- Naming: Rust `snake_case`/`CamelCase`; React components `PascalCase`.

## Testing Expectations

- After DoWhiz_service changes, run relevant AUTO tests in:
  `reference_documentation/test_plans/DoWhiz_service_tests.md`.
- LIVE/MANUAL/PLANNED entries should be marked SKIP with reason unless explicitly executed.
- Summarize executed tests and outcomes in the PR/final report.

## PR Guidelines

- Use concise, imperative commit messages.
- Include PR summary, test evidence, and config/env changes.
- Include screenshots for UI changes.

## Secrets

- Keep service secrets in `DoWhiz_service/.env` (from `.env.example`).
- Never commit tokens or credentials.

Contributor principle:
- Keep code modular and maintainable; split oversized files when needed.
