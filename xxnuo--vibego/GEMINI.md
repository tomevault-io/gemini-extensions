## vibego

> VibeGo 是一个基于 Web 的 IDE 框架，旨在在浏览器中提供类似本地 IDE 的开发体验。项目采用前后端分离架构，通过 REST API 和 WebSocket 进行通信。

# VibeGo 技术报告

## 1. 项目概述

VibeGo 是一个基于 Web 的 IDE 框架，旨在在浏览器中提供类似本地 IDE 的开发体验。项目采用前后端分离架构，通过 REST API 和 WebSocket 进行通信。

核心目标：让用户能够在浏览器中完成文件管理、Git 版本控制、终端操作等开发任务，并且支持会话持久化，切换设备或刷新页面后能够恢复之前的工作状态。

---

## 2. 架构概览

### 2.1 整体架构

项目分为前端和后端两大部分：

**后端（Go 语言）**
- 提供 HTTP REST API 服务
- 处理文件系统的读写操作
- 管理 Git 仓库操作
- 创建和管理终端伪进程
- 存储会话数据到 SQLite 数据库

**前端（React + TypeScript）**
- 提供 IDE 风格的用户界面
- 管理页面状态和用户交互
- 与后端 API 通信获取数据
- 实时显示终端输出

### 2.2 通信方式

- **REST API**：用于文件操作、Git 操作、会话管理等同步请求
- **WebSocket**：用于终端的实时输入输出通信

---

## 3. 前端架构

### 3.1 IDE 框架设计

前端采用类似 IDE 的三段式布局结构：

**顶部栏（TopBar）**
- 左侧区域：显示当前页面的图标标识
- 中间区域：显示可自定义的内容，如打开的 Tab 列表
- 右侧区域：显示功能按钮，如刷新、新建等操作

**内容区域**
- 根据当前激活的页面组类型，渲染对应的页面内容
- 支持文件浏览器、Git 管理、终端、设置等多种页面类型

**底部栏（BottomBar）**
- 左侧：主菜单按钮，点击打开项目菜单
- 中间：Mac Dock 风格的任务栏，显示所有打开的页面和页面组图标，点击可快速切换
- 右侧：全屏切换按钮

### 3.2 页面组系统

框架支持多种类型的页面组，每种类型有不同的行为：

**主页面组（HomeGroup）**
- 应用程序的首页/欢迎页
- 显示最近打开的会话列表
- 提供打开新文件夹的入口

**文件夹页面组（GenericGroup）**
- 当用户打开一个文件夹时创建
- 内部包含三个子页面：文件浏览器、Git 管理、终端管理
- 这三个子页面共享同一个根目录路径
- 用户可以在同一个窗口内切换这三个视图

**终端页面组（TerminalGroup）**
- 独立的终端页面
- 不与特定文件夹关联
- 可以创建多个终端实例

**插件页面组（PluginGroup）**
- 用于加载第三方插件
- 每个插件有独立的页面视图

**设置页面组（SettingsGroup）**
- 应用程序的设置页面
- 主题、语言、字体等配置

### 3.3 Tab 管理机制

每个页面内部可以打开多个 Tab（标签页）：

- **文件 Tab**：打开的文件预览
- **Diff Tab**：Git 文件差异显示
- **终端 Tab**：终端实例

Tab 的特性：
- 可以关闭
- 可以固定（不被新 Tab 替换）
- 双击可固定
- 激活状态切换

### 3.4 状态管理

前端使用 Zustand 进行状态管理，分为多个独立的 Store：

**框架状态（frameStore）**
- 管理所有页面组
- 跟踪当前激活的页面组
- 管理 Tab 的添加、删除、激活
- 配置顶部栏和底部栏的显示内容

**会话状态（sessionStore）**
- 管理用户会话的持久化
- 自动保存当前工作状态
- 支持从会话恢复工作环境
- 包含：打开的文件夹列表、终端实例、激活的页面等信息

**应用状态（appStore）**
- 存储全局配置
- 主题切换（浅色、深色、 hacker 风格、终端风格）
- 语言切换（中英文）
- 菜单打开状态

**文件管理器状态（fileManagerStore）**
- 当前浏览路径
- 文件列表
- 选中状态
- 排序方式
- 视图模式（列表/网格）

**终端状态（terminalStore）**
- 每个页面组的终端实例
- 当前激活的终端
- 终端列表管理器的显示状态

**Git 状态（gitStore）**
- 当前仓库路径
- 变更文件列表
- 选中的文件
- 分支信息
- 提交历史
- 储藏列表
- 冲突文件列表

### 3.5 页面注册机制

框架使用注册表模式管理可用的页面类型：

- 预定义一组内置页面（文件管理、Git、终端、插件）
- 每个页面注册时提供：唯一标识、视图类型、图标、标签、对应组件
- 运行时可根据需要动态注册新页面

### 3.6 主应用渲染流程

应用启动时：

1. 初始化设置和终端清理
2. 尝试恢复上次会话
3. 如果有保存的会话，加载所有页面组和状态
4. 如果没有会话，显示欢迎页
5. 根据当前激活的页面组类型，渲染对应的页面内容

页面切换流程：

1. 用户点击底部栏的图标
2. 更新 frameStore 中的 activeGroupId
3. 根据新的页面组类型，渲染不同的页面组件

## 4. 后端架构

### 4.1 服务启动流程

1. 加载配置文件和环境变量
2. 初始化日志系统
3. 创建 Web 框架引擎
4. 注册各类中间件
5. 初始化数据库连接
6. 注册 API 路由处理器
7. 配置静态文件服务（前端构建产物）
8. 启动 HTTP 服务器

### 4.2 中间件系统

后端使用 Gin 框架的中间件机制：

**恢复中间件**
- 捕获程序 panic
- 防止单个请求错误导致服务崩溃

**日志中间件**
- 记录每个请求的处理时间、状态码等信息

**限流中间件**
- 防止恶意请求攻击
- 默认每分钟允许 1000 次请求

**WAN 访问控制**
- 控制是否允许局域网外部访问
- 如果未设置访问令牌，则强制禁用 WAN 访问

**CORS 中间件**
- 处理跨域资源共享
- 允许配置的域名访问 API

**认证中间件**
- 验证请求是否携带有效的访问令牌
- 保护需要认证的 API 接口

### 4.3 数据库设计

使用 SQLite 存储以下数据：

**用户表**
- 存储用户基本信息

**会话表**
- 存储用户的工作会话
- 关键字段：会话名称、状态 JSON 数据、最后活跃时间、过期时间
- 状态数据以 JSON 格式存储，包含：打开的文件夹列表、各文件夹的页面状态、终端实例等

**终端会话表**
- 存储终端会话信息
- 关键字段：终端名称、使用的 Shell、工作目录、窗口大小、运行状态

**终端历史表**
- 存储终端命令历史
- 关键字段：关联的会话 ID、执行的命令、退出码、执行时间

**设置表**
- 存储用户配置
- 键值对形式存储

**键值表**
- 通用键值存储

### 4.4 文件操作 API

后端提供完整的文件系统操作能力：

**目录浏览**
- 获取指定目录的文件列表
- 支持递归获取目录树结构

**文件搜索**
- 根据关键词搜索文件

**文件 CRUD**
- 创建新文件或目录
- 删除文件或目录
- 批量删除
- 重命名
- 修改文件权限
- 修改文件所有者

**文件内容**
- 读取文件内容
- 保存文件内容

**压缩解压**
- 压缩文件或目录为 zip/tar.gz
- 解压 zip/tar.gz 文件

### 4.5 Git 操作 API

后端封装了丰富的 Git 操作：

**仓库管理**
- 初始化新仓库
- 克隆远程仓库

**状态查询**
- 获取工作区文件状态（已修改、已暂存、未跟踪等）
- 获取分支状态（领先/落后远程分支数量）

**历史查看**
- 获取提交历史
- 获取特定提交的详细信息
- 获取文件差异

**分支操作**
- 获取本地分支列表
- 获取远程分支列表
- 创建新分支
- 删除分支
- 切换分支
- 智能切换分支（自动 stash 局部修改）

**远程操作**
- 获取远程仓库列表
- 执行 git fetch
- 执行 git pull
- 执行 git push

**变更操作**
- 暂存文件
- 取消暂存
- 检出文件（放弃修改）
- 提交选定文件

**储藏操作**
- 创建 stash
- 应用 stash
- 删除 stash

**冲突处理**
- 获取冲突文件列表
- 解决冲突并标记为已解决

### 4.6 终端服务

终端服务是后端的核心功能之一：

**终端管理器**
- 负责创建和管理所有终端会话
- 维护会话与用户的关联
- 处理终端的生命周期

**伪终端机制**
- 使用操作系统提供的伪终端功能
- 创建子进程运行用户的默认 Shell
- 实现输入输出转发

**WebSocket 通信**
- 前端通过 WebSocket 连接到终端会话
- 实时传输用户输入到后端进程
- 实时传输进程输出到前端显示

**会话持久化**
- 终端会话信息存储到数据库
- 记录终端名称、工作目录、Shell 类型等
- 支持终端历史记录

### 4.7 会话管理 API

**会话 CRUD**
- 创建新会话
- 获取会话详情
- 更新会话状态
- 删除会话
- 列出会话

**会话状态结构**
会话状态以 JSON 格式存储，包含：
- 打开的文件夹列表（每个文件夹包含文件浏览器、Git、终端三个视图的状态）
- 打开的插件列表
- 各页面组的终端实例
- 当前激活的页面组 ID

## 5. 前端状态与后端数据同步

### 5.1 会话持久化原理

**自动保存机制**
- 前端使用 Zustand 的订阅功能
- 当 frameStore（框架状态）或 terminalStore（终端状态）发生变化时
- 延迟 1 秒后自动触发保存
- 将当前状态序列化为 JSON，发送到后端保存

**会话恢复流程**
1. 应用启动时从本地存储读取上次会话 ID
2. 调用后端 API 获取会话数据
3. 解析 JSON 状态数据
4. 重建所有页面组和视图
5. 恢复终端实例
6. 设置当前激活的页面组

### 5.2 跨设备同步

由于会话数据存储在服务器端（SQLite），用户在不同设备上访问时：
- 登录后可以获取到保存的会话
- 会自动恢复到上次的工作状态

## 6. 核心交互流程

### 6.1 打开文件夹流程

1. 用户点击"打开文件夹"按钮
2. 弹出目录选择器
3. 选择目标目录
4. 前端创建新会话，发送到后端
5. 后端创建会话记录，返回会话 ID
6. 前端创建文件夹页面组
7. 加载该目录的文件列表
8. 显示文件浏览器页面

### 6.2 切换视图流程

1. 用户点击底部栏的页面组图标
2. 更新框架状态的 activeGroupId
3. 框架自动切换到对应的页面
4. 如果是文件夹组，还会切换到该组的当前活动页面（文件浏览器/Git/终端）

### 6.3 打开文件流程

1. 用户在文件浏览器中双击文件
2. 创建新的 Tab，存储文件路径信息
3. 切换到 TabBar 视图
4. 根据文件类型选择对应的预览组件
5. 调用后端 API 获取文件内容
6. 渲染文件预览

### 6.4 Git 操作流程

1. 用户打开文件夹时，自动初始化 Git 状态
2. 并行调用多个 API 获取：文件状态、分支信息、提交历史、远程仓库、stash 等
3. 通过 WebSocket 监听仓库变化
4. 用户执行操作（commit、push、pull 等）时，调用对应 API
5. 操作完成后刷新相关状态

### 6.5 终端连接流程

1. 用户点击"新建终端"
2. 后端创建新的终端会话和伪终端进程
3. 返回终端 ID
4. 前端通过 WebSocket 连接到该终端
5. 用户输入命令，通过 WebSocket 发送到后端
6. 后端将输入写入伪终端
7. 进程输出通过 WebSocket 实时返回前端
8. 前端使用 xterm.js 渲染输出

## 7. 配置系统

### 7.1 后端配置

通过命令行参数或环境变量配置：

- **日志级别**：控制日志输出详细程度
- **配置目录**：存放数据库文件的位置
- **日志目录**：日志文件存放位置
- **服务地址和端口**：HTTP 服务监听地址
- **访问令牌**：API 访问验证
- **WAN 访问**：是否允许外部网络访问
- **CORS 策略**：允许的跨域来源
- **默认 Shell**：终端使用的 Shell 程序

### 7.2 前端配置

- **主题**：浅色、深色、 hacker 风格、终端风格
- **语言**：中文、英文
- **字体**：选择代码字体

## 9. 技术决策要点

### 9.1 前后端分离

- 前端专注 UI 交互，使用现代化的 React 技术栈
- 后端专注业务逻辑，使用 Go 语言的高性能特性
- 通过 JSON API 进行数据交换

### 9.2 状态管理选择

- 使用 Zustand 而非 Redux：更轻量、更简洁的 API
- 每个功能模块独立的状态 Store，便于管理和调试

### 9.3 会话持久化

- 将会话状态存储在后端数据库
- 支持跨设备同步和刷新恢复
- 自动保存机制减少数据丢失风险

### 9.4 终端实现

- 使用伪终端技术实现真正的 Shell 环境
- WebSocket 保证实时性和双向通信

### 9.5 SQLite 使用

- 轻量级数据库，无需单独部署
- WAL 模式提供良好的并发性能
- 适合单用户本地应用场景

## 10. 数据流总览

**用户操作 → 组件事件 → Zustand Store 状态更新 → UI 自动渲染**

**状态变化 → 触发订阅 → 自动保存 → 后端 API → SQLite 持久化**

**页面加载 → 读取本地会话 ID → 后端获取会话 → 解析状态 → 重建 Store → 渲染 UI**
