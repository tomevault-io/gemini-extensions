## 02-memory

> Rules for reading and updating memory/ files across import, build, and QA flows.


# Memory rules

## Purpose

The `memory/` directory is shared state across all agent sessions. It reduces token cost (agents read small curated files instead of replaying history) and prevents re-learning the same things.

## When to read

- **Always**: read `memory/architecture.md`, `memory/patterns.md`, `memory/decisions.md`, and `memory/stack-guidance.md` at session start before writing any code.
- **Before building a UI task**: check `memory/mcp-cache/` for any cached MCP context for that feature.

## When to update

### `memory/architecture.md`

Update when the task introduced:
- A new module, directory, or major file
- A new data flow or service boundary
- A change to the auth flow or DB schema

Keep entries as short bullet points or a brief module map update. Do not write prose.
Updated by QA after approving a task.

### `memory/patterns.md`

Append when the task established:
- A new component or hook structure that will repeat across features
- A new API handler shape
- A new test helper or fixture pattern
- A new way of handling errors, loading states, or validation

**Do not duplicate** an existing pattern. If a pattern already exists, only update it if the new task changed it.
Updated by QA after approving a task.

### `memory/decisions.md`

Append when the task required a non-obvious choice that future tasks should not revisit:
- A library or approach chosen over alternatives
- A security or auth decision
- A naming or structural convention that differs from the default

Format: `## Short title → Context. Decision made. Reason.`
Updated by QA after approving a task.

### `memory/stack-guidance.md`

Update when:
- `import docs` confirmed a tech stack and wrote/updated `SCOPE.md`
- `SCOPE.md` architecture or stack changed enough that Builder guidance is now stale

Keep it short and operational:
- stack summary
- architecture defaults
- module/file conventions
- data/state/API guidance
- UI guidance
- testing guidance
- anti-patterns to avoid
Generated by `import docs` after `SCOPE.md` is confirmed, not by QA.

### `memory/mcp-cache/`

The `fetch-mcp` skill writes files here. Do not edit these manually.
File naming convention: `{feature-slug}-{mcp-type}.md` (e.g. `auth-figma.md`, `items-openapi.md`).

Freshness: each cache file carries `Fetched:`, `TTL hours:`, and `Source hash:`
headers. Before reading a cache, Builders and QA must verify:
- `Fetched + TTL hours > now`
- `Source hash` matches the current task's MCP URL

Stale or URL-mismatched caches are refetched via `fetch-mcp` (or `figma ingest`
for Figma) before the task proceeds. Never implement against a stale cache.

Trigger `refresh mcp` (or set `Force refresh: true` on a task) to force a
refetch regardless of TTL.

## What NOT to put in memory/

- Full file contents — link to the file instead
- Long explanations — bullet points and short snippets only
- Temporary notes — if it's not reusable across tasks, it doesn't belong here
