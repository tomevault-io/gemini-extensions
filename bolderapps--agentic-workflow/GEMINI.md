## 07-concurrency

> Concurrency-safe writes for shared workflow files during parallel builds.


# Concurrency rules

Parallel Builders + QA + Orchestrator share a small set of files. Without
discipline, two agents editing the same file simultaneously will corrupt
state. These rules define who owns which writes.

## Single-writer principle

For each shared file, exactly one agent type is the "writer" for a given
section. Other agents read only.

| File | Writer | Notes |
|------|--------|-------|
| `TASKS.md` — task block body (fields, AC, files) | Builder owning that task | Only during `in-progress` or `needs-fix` |
| `TASKS.md` — task `Status:` field | Builder (to `in-progress`, `in-review`, `blocked`) and QA (to `done`, `needs-fix`) | No other agent flips status |
| `TASKS.md` — `QA notes:` | Builder (handoff note) and QA (rejection note) | Append-only within a task |
| `TASKS.md` — Build progress table | Orchestrator (or the `update-task-counts.py` script invoked by the last status-flipper) | Recalculated from status fields, never hand-edited |
| `memory/architecture.md` | QA (after `done`) | Builders never write here |
| `memory/patterns.md` | QA (after `done`) | Builders never write here |
| `memory/decisions.md` | QA (after `done`) | Builders never write here |
| `memory/stack-guidance.md` | `import-docs` skill only | Not written during build |
| `memory/mcp-cache/*` | `fetch-mcp` and `figma-plugin-ingest` only | Builders/QA read only |

## Task-block locking protocol

Before a Builder edits any field inside a task block, it must:

1. Read the current `Status:` field.
2. Confirm the task is assigned to its own builder slot (`builder-N` matches
   the Builder's ID) or is unassigned and ready to be claimed.
3. If `Status:` is already `in-progress` with a different builder ID, stop
   and surface to Orchestrator.

QA must:

1. Never edit a task block whose `Status:` is `in-progress`.
2. Only edit blocks whose `Status:` is `in-review`.

## Read-modify-write protocol for `TASKS.md`

Because `TASKS.md` is a single file touched by many agents:

1. Every edit re-reads the whole file immediately before writing.
2. If between the read and the write another agent has modified the same
   task block, abort the write, re-read, and retry. Do not overwrite another
   agent's edit.
3. The progress table is always regenerated by `scripts/update-task-counts.py`
   — never hand-incremented during a write.

## Memory file append protocol

For `memory/patterns.md`, `memory/architecture.md`, `memory/decisions.md`:

1. QA reads the full file before appending.
2. QA scans existing entries for intent-duplication before writing a new one
   (see patterns dedup protocol below). If a near-duplicate exists, QA
   updates the existing entry rather than appending.
3. QA appends under an explicit section heading so simultaneous appends from
   two QA invocations do not conflict on the same line.

## Patterns dedup protocol

Before appending to `memory/patterns.md`, QA must:

1. Build a short "intent fingerprint" for the candidate pattern (one line
   describing the problem it solves, not the code).
2. Grep existing headings and intent lines for close matches.
3. If a match exists:
   - update the existing entry if the new task refines it,
   - or skip the append if the existing entry already covers the case.
4. Only append a new section when no intent match is found.

This stops `patterns.md` from accumulating near-duplicates like `API handler`
and `API route handler` and `REST endpoint shape`.

## File ownership across Builders

Already covered in `03-subagents.mdc`. Recap:

- Files listed under a task's `Files to create/modify:` are owned by the
  assigned Builder while the task is `in-progress` or `needs-fix`.
- No other Builder may edit those files during that window.
- Shared scaffolding files (`types/`, `config/`, `memory/`) are created by
  the first Builder that needs them and then read-only for others until the
  originating task is `done`.

## Orchestrator safety net

On every scheduler loop tick, Orchestrator runs these integrity checks and
surfaces violations before assigning new work:

- Every `in-progress` task has exactly one assigned Builder ID.
- No two `in-progress` tasks share a file in `Files to create/modify:`.
- No task shows `done` without a `QA notes:` body.
- The progress table sum matches the per-status count in task blocks.

If any check fails, Orchestrator halts new assignments and reports the
inconsistency to the user before continuing.
