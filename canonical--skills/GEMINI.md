# Agent Instructions — canonical/skills

## 1. Project Overview

`canonical/skills` is the public, agent-agnostic Agent Skills repository for
Canonical. Each skill is a folder containing a `SKILL.md` file that any
skills-compatible agent (GitHub Copilot CLI, Claude Code, Cursor, Codex,
Gemini CLI, Windsurf, and others) can discover and install via
`npx skills add canonical/skills`.

Skills are organized by category under `skills/`, validated by CI, and
published as a GitHub Pages site at `canonical.github.io/skills`.

---

## 2. Tech Stack

- **Language:** Python 3.10+ (validation and page-generation scripts only)
- **Package manager:** `uv` — use `uv run python` for all script invocations
- **Skill format:** YAML frontmatter + Markdown body (`SKILL.md`)
- **Markdown linter:** `pymarkdownlnt` (run via `python -m pymarkdown`)
- **CI:** GitHub Actions (`.github/workflows/`)
- **Pages:** Static HTML generated by `scripts/generate_skills_page.py`

## 3. Repository Structure

```text
skills/
├── meta/           # Skills about authoring or managing skills
├── products/       # Product-specific skills (sub-folder per product)
│   └── <product>/
│       └── <skill-name>/SKILL.md
├── engineering/    # Cross-product engineering practices
├── documentation/  # Cross-product documentation standards
├── practices/      # Cross-functional ways of working (planning, retros, delivery)
├── operations/     # Infrastructure, deployment, operational runbooks
└── security/       # Security practices and compliance workflows

scripts/
├── validate_skills.py      # Frontmatter validator (run via make validate)
├── generate_skills_page.py # GitHub Pages generator (run via make pages)
└── template.html           # HTML template for the Pages site

.github/
├── workflows/
│   ├── validate.yml        # PR CI — runs validate, lint, pages
│   └── pages.yml           # Deploys docs/index.html to GitHub Pages
└── copilot-instructions.md # Copilot-specific alias (points here)

docs/
└── index.html  # Generated — never edit directly; always gitignored
```

## 4. Constitutional Rules

### Skill placement

- Place a skill under `products/<product>/` only if it applies to **one specific
  Canonical product** without modification.
- Place it under an expertise category (`engineering/`, `practices/`, etc.) if
  it applies across products or is cross-functional.
- Use `practices/` for ways-of-working skills: retrospectives, planning rituals,
  epic breakdowns, delivery reviews.

### SKILL.md frontmatter — all fields are required

```yaml
---
name: <kebab-case>        # lowercase; unique across repo; matches folder name
description: >
  <What the skill does — at least 20 words before the WHEN clause.>
  WHEN: <8+ comma-separated trigger phrases covering beginner and expert vocab>.
license: Apache-2.0
metadata:                 # spec-standard location for author, version, tags
  author: Canonical             # or Canonical/<team>, e.g. Canonical/platform-engineering
  version: "1.0.0"       # SemVer, quoted string
  summary: One or two sentence human-readable blurb shown on skill cards (≤ 160 chars).
  tags:
    - <domain-or-product>
---
```

### Naming

- Folder name must match the `name` field exactly (kebab-case).
- Skill names are globally unique across all categories.

### Content rules

- Skill body must be agent-agnostic — no references to specific agents or
  tools (Copilot, Claude, Cursor, etc.).
- Description must include a `WHEN:` clause with at least 8 trigger phrases.
- Large skills (>200 lines or >3 sub-topics) use a `references/` sub-folder;
  `SKILL.md` acts as a router to those files.

### Code and scripts

- All Python scripts run via `uv run python <script>` — never call `python`
  or `python3` directly.
- Scripts live in `scripts/`; no logic belongs in CI workflow files.
- `docs/index.html` is generated — edit `scripts/template.html` instead.

### Versioning

- Bump `version` on every skill change: patch for corrections, minor for
  additions, major for breaking interface changes.

### Security

- Store secrets in environment variables or GitHub Actions secrets.
- Skill content must not include credentials, tokens, or internal hostnames.

## 5. Development Workflow

```bash
make install    # Install Python deps with uv (first-time setup)
make validate   # Validate all SKILL.md frontmatter and structure
make lint       # Lint Markdown files (README, CONTRIBUTING, skills/)
make pages      # Generate docs/index.html (smoke test the page generator)
make check      # Run all three above in sequence (use before every commit)
```

`make validate` exits non-zero on any **error** (missing fields, wrong name,
bad SemVer). **Warnings** (short description, few trigger phrases) are printed
but do not block CI.

## 6. Adding or Modifying a Skill

### Before you start — install `generate-agent-skills`

The `generate-agent-skills` skill from this repository is the recommended
authoring tool for any new or updated skill. Install it once and your agent
will guide you through every step below automatically:

```bash
# local (scoped to current project)
npx skills add canonical/skills --skill generate-agent-skills

# global (available everywhere)
npx skills add canonical/skills --skill generate-agent-skills --global
```

Skipping this step frequently leads to frontmatter and description quality
issues that block PR review. Use it before writing a single line of `SKILL.md`.

### Authoring steps

1. **Draft** — invoke `generate-agent-skills` with your agent, or follow
   CONTRIBUTING.md Step 1 for the full prompt.
2. **Create folder** — `skills/<category>/<skill-name>/SKILL.md`
   (or `skills/products/<product>/<skill-name>/SKILL.md`).
3. **Write frontmatter** — all six required fields; description ≥ 20 words
   + `WHEN:` clause with 8+ phrases.
4. **Validate** — `make check` must pass with zero errors before opening a PR.
5. **CODEOWNERS** — add an entry if a team other than Platform Engineering
   owns the new skill or product folder.

## 7. Category Decision Reference

| Skill applies to… | Folder |
|---|---|
| One specific Canonical product | `products/<product>/` |
| Multiple products / general engineering | `engineering/` |

| Infrastructure, deployment, ops runbooks | `operations/` |
| Security practices or compliance workflows | `security/` |
| Authoring or reviewing other skills | `meta/` |
