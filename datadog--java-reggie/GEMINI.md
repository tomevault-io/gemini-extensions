## Project Overview

Reggie is a high-performance Java regex library with dual compilation modes (compile-time and runtime) that generates specialized bytecode for each pattern, achieving 7-389x speedup over JDK Pattern.

**Key Facts**:
- Language: Java 21+
- Build: Gradle 8.11+
- Architecture: Thompson NFA → DFA → Specialized Bytecode
- Performance: 7-389x faster than JDK Pattern
- PCRE Conformance: 91.3% (303/332 tests)
- Modules: 6 (annotations, codegen, processor, runtime, benchmark, integration-tests)

## Architecture

### Core Pipeline
```
Pattern String → RegexParser → AST → ThompsonBuilder → NFA → SubsetConstructor → DFA
                                                                    ↓
                                              PatternAnalyzer (strategy selection)
                                                                    ↓
                                              BytecodeGenerator (28+ strategies)
                                                                    ↓
                                              Specialized Java bytecode
```

### Module Structure
- **reggie-annotations**: Single `@RegexPattern` annotation
- **reggie-codegen**: Core compilation engine (shared by both modes)
  - `ast/`: 12 AST node types
  - `parsing/`: RegexParser (pattern → AST)
  - `automaton/`: ThompsonBuilder (NFA), SubsetConstructor (DFA)
  - `analysis/`: PatternAnalyzer (strategy selection)
  - `codegen/`: 20+ bytecode generators
- **reggie-processor**: Annotation processor (compile-time path)
- **reggie-runtime**: Public API + RuntimeCompiler (runtime path)
- **reggie-benchmark**: 322 JMH benchmarks
- **reggie-integration-tests**: PCRE/RE2 test suites

### Key Design Patterns
- Strategy Pattern: 28+ bytecode generation strategies
- Visitor Pattern: AST traversal
- Builder Pattern: NFA/DFA construction
- Factory Pattern: PatternAnalyzer strategy selection
- Service Provider Interface: Compile-time pattern discovery

### Strategy Selection & Pattern Analysis

The `PatternAnalyzer` in reggie-codegen selects the optimal bytecode generation strategy:

```
Pattern Analysis Decision Tree:
├─ Has backreferences? → NFA_WITH_BACKREFS (ThompsonBuilder)
├─ Has lookahead/lookbehind? → HYBRID_DFA_NFA
├─ Pure regular expression?
│  ├─ <50 DFA states? → DFA_UNROLLED (fastest, straight-line code)
│  ├─ 50-300 states? → DFA_SWITCH (switch-based state machine)
│  └─ >300 states? → OPTIMIZED_NFA (graceful fallback)
└─ State explosion risk? → OPTIMIZED_NFA
```

**Bytecode generators** (all in `reggie-codegen/src/main/java/com/datadoghq/java-reggie/codegen/codegen/`):
1. `DFAUnrolledBytecodeGenerator`: <50 states - maximum performance, straight-line code
2. `DFASwitchBytecodeGenerator`: 50-300 states - switch-based state machine
3. `NFABytecodeGenerator`: Complex patterns - uses BitSet for state tracking
4. `OnePassBytecodeGenerator`: Single-pass for non-ambiguous patterns
5. `RecursiveDescentBytecodeGenerator`: Patterns requiring backtracking (backrefs, subroutines)
   - Supports nested backtracking for multiple greedy quantifiers
   - Dynamic slot allocation (`slotBase` parameter) for arbitrary nesting depth
   - Cascading backtrack via `outerTryMatchCountSlot` parameter
6. Plus 20+ other specialized generators for specific pattern types

### Important Context

**Why Two Compilation Modes?**
- **Compile-time**: Zero overhead for known patterns (95% of use cases). Errors caught at build time.
- **Runtime**: Flexibility for user-provided patterns, config-driven patterns, dynamic search.

**Why Linear Time Matters**
Traditional backtracking engines (like JDK Pattern) have exponential worst-case time complexity, leading to ReDoS (Regular Expression Denial of Service) vulnerabilities. Reggie guarantees O(n) time by using DFA/NFA without backtracking.

**ASM Library Usage**
All bytecode generation uses ASM 9.7 (org.ow2.asm). Generated matchers are standard Java classes that extend ReggieMatcher. No reflection or dynamic proxies in hot path.

**Generating Specialized Forms**
Never generate generic bytecode. Use the exact knowledge of the pattern you are compiling to generate highly specialized bytecode. With that you will be able to eliminate branches, extra allocations and unroll loops.

```java
public class MyBytecodeGenerator implements BytecodeGenerator {
    @Override
    public void generateMatchesMethod(ClassWriter cw, String className) {

            ACC_PUBLIC, "matches", "(Ljava/lang/String;)Z", null, null);
```

### Bytecode Generation Dual-Path Rule
**CRITICAL**: Changes to bytecode generation require updates to BOTH:
1. `RuntimeCompiler.java` (runtime path: `Reggie.compile()`)
2. `ReggieMatcherBytecodeGenerator.java` (compile-time path: `@RegexPattern`)

Verify both:
```bash

./gradlew :reggie-benchmark:build       # Test compile-time
```

### Structural Hash Rule
When creating `PatternInfo` subclasses, `structuralHashCode()` MUST include ALL fields affecting bytecode:
```java
public int structuralHashCode() {
    int hash = getClass().getName().hashCode();
    hash = 31 * hash + minCount;
    hash = 31 * hash + (charset != null ? charset.hashCode() : 0);  // Include all fields!
    hash = 31 * hash + groupNumber;
    return hash;
}
```
Missing fields → cache collisions → wrong results.

### Bytecode Generation Best Practices
1. **Never hardcode local variable slots**:
   ```java
   // WRONG: int var = 50;
   // RIGHT: int var = highestSlot + 1;
   ```

2. **Never use visitLdcInsn with primitive int**:
   ```java
   // WRONG: mv.visitLdcInsn(value);  // autoboxes!
   // RIGHT: BytecodeUtil.pushInt(mv, value);
   ```

3. **Document slot allocation** in method comments

## Automatic Fallback to java.util.regex

Certain pattern structures trigger known correctness bugs in the reggie engine. For these patterns,
`Reggie.compile()` automatically falls back to `java.util.regex` and emits a `WARNING` log:

```
Falling back to java.util.regex for pattern '<pattern>': <reason>
```

**Patterns that fall back** (handled by `FallbackPatternDetector`):

| Bug | Example | Reason |
|-----|---------|--------|
| Multiple `\1` to same group in NFA/variable-capture path | `(\w+)\s+\1\s+\1` | `multiple backreferences to group 1 in NFA mode` |
| Lookahead inside a quantified group | `(?:(?=\d)\d)+` | `lookahead inside quantified group` |
| Lookbehind followed by unbounded quantifier | `(?<=\d)[a-z]+` | `lookbehind followed by unbounded quantifier` |
| Alternation inside lookbehind | `(?<=a\|b)c` | `alternation inside lookbehind` |
| Lookbehind and lookahead used together | `(?<=\[)[^\]]+(?=\])` | `lookbehind and lookahead combined` |

> **Note:** Bug 1 (multiple backreferences to same group) only applies when the analyzer selects
> `OPTIMIZED_NFA_WITH_BACKREFS` or `VARIABLE_CAPTURE_BACKREF` strategy. Patterns routed through
> other backref strategies (`SPECIALIZED_BACKREFERENCE`, `FIXED_REPETITION_BACKREF`, etc.) are
> not subject to this fallback.

The fallback applies only to `Reggie.compile()` (runtime path). Patterns compiled via the
`@RegexPattern` annotation processor that trigger a known bug will fail at build time with an
`UnsupportedOperationException` — use `Reggie.compile()` instead for those patterns.

The fallback is transparent to callers of `Reggie.compile()` — correctness is guaranteed at the
cost of reggie's allocation-free performance. All other patterns continue to use the fast reggie
engine.

## Known Limitations

- **Capturing groups**: Work in progress (Phase 5 hybrid approach)
- **Backreferences**: Partially supported with significant limitations:
  - Patterns with fixed quantifiers work: `(a{2})\1` matches "aaaa"
  - Patterns with variable quantifiers (*, +, ?, {n,m}) in capturing groups require backtracking and only work for minimal matches
  - Example: `(a+)\1` matches "aa" but NOT "aaaa" (would need backtracking to try group="aa", backref="aa")
  - Root cause: Thompson NFA doesn't support backtracking, which is required for greedy/non-greedy quantifiers with backreferences
  - Specialized patterns like `<(\w+)>.*</\1>` (HTML tags) use optimized non-NFA implementations and work correctly
  - **Self-referencing backreferences**: Patterns where a group references itself (e.g., `(a\1?){4}`, `(a\1?)(a\2?)`) don't work correctly
    - Root cause: Quantifiers don't implement "last iteration semantics" - they don't update the group's captured value on each iteration
    - Example: `(a\1?){4}` matching "aaaa" should work by having `\1` reference the previous iteration's capture, but currently fails
    - Fix requires implementing per-iteration group capture updates in RecursiveDescentBytecodeGenerator.visitQuantifier()
    - Tests for this are skipped by default; run with `-Dreggie.test.knownFailures=true` to enable
- **Recursive patterns**: Limited support via RECURSIVE_DESCENT strategy:
  - Subroutines (`(?R)`, `(?1)`) and conditionals (`(?(1)yes|no)`) work for most cases
  - **Recursive palindromes**: Patterns like `^(\w)(?:(?1)|\w?)\1$` (palindrome checker) don't work
    - Root cause: Subroutines in RecursiveDescentBytecodeGenerator save/restore groups but don't support backtracking
    - Proper support requires backtrackable subroutines (estimated 8-12 hours implementation)
    - Tests for this are skipped by default; run with `-Dreggie.test.knownFailures=true` to enable
- **Unicode properties**: Not yet supported (`\p{L}`, `\p{N}`, etc.)
- **Atomic groups**: Not supported (`(?>...)`)
- **Possessive quantifiers**: Not supported (`*+`, `++`, `?+`)
- **Inline flags**: Partially supported - `(?i)`, `(?m)`, `(?s)`, `(?x)` work globally; scoped flags like `(?i:...)` not yet supported
