## Team-Lead Control Plane

- team-lead = the main conversation, not a spawned agent
- team-lead owns user alignment, scope control, task decomposition, and phase transitions
- team-lead maintains project-global truth: main `task_plan.md`, `decisions.md`, and this `CLAUDE.md`
- team-lead decides whether a workflow improvement is project-local or should be written back into `CCteam-creator`
- **No standalone subagents**: Once the team exists, ALL work goes through teammates. Do NOT spawn standalone Agent/subagent — they bypass the team's planning files and coordination.

## Task Dispatch Protocol

### Message Delivery Timing (Critical)
`SendMessage` lands only when the recipient is idle — you cannot interrupt a running task. Front-load context in the initial dispatch, accept no preemption for broadcasts, and read `progress.md` / `findings.md` directly for live status.

### TaskCreate Description Format

TaskCreate description: one-line scope + acceptance criteria + `.plans/` path.
Example: `"深度调研代码库和竞品。Output: findings.md with conclusions. See .plans/excalidraw-showcase/researcher/task_plan.md"`

### Large Tasks -- STOP AND CHECK BEFORE SENDING

**Before sending a large task, verify your message includes ALL 4 items:**
1. **Scope & goals**: What to build, acceptance criteria
2. **Document reminder**: "Create a task folder with task_plan.md + findings.md + progress.md"
3. **Dependencies**: Key file paths and line numbers
4. **Review expectation**: Whether code review is needed after completion

Task folder prefixes by role:
- backend-dev / frontend-dev: `task-<name>/`
- researcher: `research-<topic>/`
- e2e-tester: `test-<scope>/`
- reviewer: `review-<target>/`

## Review Dimensions

| # | Dimension | Weight | STRONG looks like | WEAK looks like |
|---|-----------|--------|-------------------|-----------------|
| RD-1 | Product depth | high | Handles edge cases real users hit — empty states, error recovery, formula rendering failures | Happy path only; errors show raw exceptions |
| RD-2 | Test coverage | high | Key behaviors covered by tests; easy to add new cases | No tests, or tests coupled to implementation |
| RD-3 | API design | medium | Clean, consistent interfaces with proper field naming | Ad-hoc APIs with inconsistent naming |
| RD-4 | Visual polish | medium | Showcases look professional; responsive; accessible | Rough, unstyled, broken on mobile |
