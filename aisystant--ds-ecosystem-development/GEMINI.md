## 1. Тип репозитория

**DS/governance** — управленческая сборка экосистемы + операционная рабочая площадка, оформленная по S2R.

**НЕ является source-of-truth** — знания берутся из Pack'ов. Операционный контент (семинары, предложения) — промежуточная стадия; формализованное знание мигрирует в Pack через KE on Close.

### 1.1. Что содержит этот репозиторий

| Категория | Примеры | Жизненный цикл |
|-----------|---------|----------------|
| **Справочный контент** | Что есть, где искать, ссылки на Pack'и | Постоянный |
| **Реестры** | Объекты внимания, создаваемые системы, агенты, партнёры | Постоянный |
| **Планы и стратегии** | Q1 2026 roadmap, приоритеты, milestone'ы | Постоянный |
| **Процессы взаимодействия** | Кто с кем, как, SLA, ритуалы | Постоянный |
| **Организационная структура** | Роли команды, ответственности | Постоянный |
| **Управленческие решения** | ADR, трассировка решений | Постоянный |
| **Операционная работа** | Семинары, предложения, черновики архитектуры | Временный → KE → Pack |

### 4.5. Governance + Staging for Pack

> Этот репозиторий содержит governance-контент И операционную работу (семинары, предложения, черновики), которая на Close мигрирует в Pack через KE.

**Три вида контента:**

| Вид | Примеры | Жизненный цикл |
|-----|---------|----------------|
| **Governance** (постоянный) | Реестры, планы, ADR, оргструктура | Живёт здесь |
| **Операционный** (временный) | Семинары, предложения, inbox | Создаётся → работа → KE on Close → архив |
| **Pack-контент** (транзитный) | Знания домена, методы, архитектуры | Формализуется → мигрирует в Pack |

**При создании/изменении документа спроси:**
1. Governance (привязано к команде/срокам)? → Создай здесь
2. Операционная работа (семинар, предложение)? → Создай в inbox/OPS → на Close извлеки знание в Pack
3. Уже формализованное знание домена? → Сразу в Pack, здесь только ссылка

**Самоочистка:** KE on Close = механизм самоочистки. Операционный контент не накапливается — знание уходит в Pack, артефакт в архив.

## 5. Создание документа

### 5.1. Алгоритм

1. Определи ЯДРО (A/B/C)
2. Определи СЕМЕЙСТВО (F1-F9)
3. Прочитай `0.OPS/0.6.Repository-Processes/02-standards.md`
4. Создай документ с frontmatter

### 5.2. Frontmatter

**Канонический источник:** [`0.OPS/0.6.Repository-Processes/05-frontmatter-spec.md`](0.OPS/0.6.Repository-Processes/05-frontmatter-spec.md) — единственное место, где определяется словарь `status`, поля `blockers` / `adr_ref` / `superseded_by`, переходы между статусами, правило «размещение vs статус».

**Минимальный пример:**

```yaml
---
family: F5
kernel: A
system: A2
role: Architecture
status: draft
target_audience:
  - "Наставники"
depends_on: []
---
```

**Статусы (кратко):** `stub → draft → review → active → archived | superseded`. Подробности и семантика — в каноне.

**Правило размещения:** документ кладётся в правильное место по матрице 3×3 **сразу** при создании (даже если `stub`), смена статуса не двигает файл. См. [АП-23](0.OPS/0.1.Knowledge-Logic/08-anti-patterns.md#ап-23-папки-по-статусу-зрелости).

### 5.4. Выбор семейства — тест целевого читателя

> Канон: [`0.OPS/0.1.Knowledge-Logic/02-document-families.md`](0.OPS/0.1.Knowledge-Logic/02-document-families.md). Ниже — рабочая выжимка для быстрого ответа на вопрос «куда класть документ».

**Перед размещением любого документа прогнать три вопроса:**

1. **Какое ядро (A/B/C)?** — Какая SoI у вопроса?
   - A = Созидатель (человек, компетенции, развитие)
   - B = Экосистема (целиком: сообщество + ИТ-платформа + контент, как единое целое)
   - C = ИТ-платформа (техническая подсистема экосистемы)

2. **Какая система (1/2/3)?** — Где находится то, о чём пишем?
   - 1 = Надсистема (то, во что SoI встроена: общество, рынок, регуляторы)
   - 2 = SoI (сама целевая система)
   - 3 = Система создания (команда + процессы + инфраструктура создания SoI)

3. **Какая роль (.1/.2/.3)?** — С какой точки зрения пишем?
   - .1.Meaning = зачем, ценности, стратегические решения (разработчик: Визионер/Бизнесмен/Маркетолог)
   - .2.Architecture = как устроено, из чего состоит (разработчик: Инженер/Архитектор)
   - .3.Operations = как работает, процессы, ритуалы (разработчик: Менеджер/Оператор)

**Главный тест — «для кого написан документ?»**:

| Целевой читатель | Правильное семейство |
|---|---|
| Потенциальные участники, широкая аудитория | F1 (B1.1) |
| Пользователи продукта, UX-дизайнеры | F2 (B2.1) |
| **Партнёры, регуляторы, медиа, PR** | **F3 (B3.1)** |
| Ученики (ценность), потенциальные созидатели | F4 (A1.2 / B1.2) |
| Методологи, наставники | F5 (A2.2 / B2.2) |
| Ученики (путь), сопровождение | F6 (A3.2 / B3.2) |
| **Инвесторы, стратеги, члены правления** | **F7 (B1.3)** |
| Разработчики, архитекторы, DevOps | F8 (C2.3) |
| Операторы, HR, финансисты | F9 (B3.3 / C3.3) |

**Типичные ошибки классификации:**

| Документ содержит | Ошибочное семейство | Правильное |
|---|---|---|
| Compliance, регуляторика, юрисдикции | F1, F4, F7 | **F3** |
| Привлечение инвестиций, бизнес-модель | F1, F2 | **F3 или F7 — см. ниже** |
| Архитектура платформы, API | F4, F6 | F8 |
| Путь ученика, онбординг | F1, F7 | F6 |
| ROI, unit-экономика, оценка компании | F1, F4, F3 | F7 |

**Пограничный случай F3 vs F7** (частая путаница):
- **F3 (Meaning × Система-создания)** = **стратегические** решения про экономику, compliance, инвестиции; разработчик — Бизнесмен/PR; читатели — **партнёры, регуляторы, инвесторы** (как стейкхолдеры с которыми строим отношения).
- **F7 (Operations × Надсистема)** = **операционные** процессы работы с надсистемой; разработчик — GR/PR/бизнес-операции; читатели — **инвесторы, фаундеры, стратеги** (как потребители финансовой/операционной информации).
- **Правило:** если документ отвечает на «какое у нас решение про X» — F3; если на «как мы работаем с X на ежедневной основе / какие метрики / какой SLA» — F7. Документы часто связаны крестссылкой `related:`. См. [диагональные связи](0.OPS/0.1.Knowledge-Logic/02-document-families.md#диагональные-связи-критические).

## 8. Миграция контента в Pack-репозитории

### 8.1. Принцип работы с Pack-контентом

**Правило:** Pack-контент (знания, методы, характеристики) хранится в Pack-репозиториях (source-of-truth).
DS-ecosystem-development содержит governance + операционную работу. Операционный контент (семинары, предложения) — транзитный: на Close знание мигрирует в Pack через KE, артефакт → архив (см. §4.5).

| Ядро | Source-of-truth | Что в Pack |
|------|-----------------|------------|
| A (Созидатель) | PACK-personal | Знания, характеристики, состояния |
| B (Экосистема) | PACK-ecosystem | Знания об экосистеме, компоненты |
| C (ИТ-платформа) | PACK-digital-platform | Архитектура платформы, агенты |

**Структура сохраняется по s2r.** Каждое ядро содержит полную иерархию X1/X2/X3 × Meaning/Architecture/Operations.

### 8.3. Роль _index.md файлов

Файлы `_index.md` в подпапках — это **навигационные документы**, которые:
1. Содержат ссылки на source-of-truth в Pack
2. Описывают назначение раздела
3. Дают реестр объектов внимания
4. Указывают на подразделы и governance-контент

**Важно:** _index.md НЕ заменяет весь контент папки. В подпапках находится:
- Governance-контент (реестры, описания, связи)
- Контекстные документы, связывающие области знаний
- Операционные материалы (семинары, предложения, черновики)

### 8.4. Правило для контента

| Тип контента | Где находится | Жизненный цикл |
|--------------|---------------|----------------|
| Формализованное знание | Pack (source-of-truth) | Постоянный в Pack |
| Ссылки и навигация | `_index.md` | Постоянный здесь |
| Governance (реестры, планы) | Файлы в подпапках | Постоянный здесь |
| Операционная работа | inbox / OPS / подпапки | Временный → KE → Pack + архив |
