## zzz-simple-ai

> - **项目名**：ZZZ白话讲AI（ZZZ-Simple-AI）

# AGENTS.md

## 项目概况

- **项目名**：ZZZ白话讲AI（ZZZ-Simple-AI）
- **目标**：用白话、极简、可复述的方式，帮零基础读者建立 AI 的基本认知
- **写给谁**：零基础职场人士、大学生，以及想先建立整体框架的读者

**核心原则**：一切内容都服务于零基础读者。他们不需要知道所有细节，只需要能听懂、能复述、能用上。

---

## 核心理念：三个学习方法

| 方法 | 核心 | 写作检验 |
|:---|:---|:---|
| **费曼学习法** | 用更简单的话重讲 | 一个零基础的朋友坐在我面前，我能让他听懂吗？ |
| **苏格拉底式提问** | 用问题引导思考 | 不用术语开头，先回到原始问题 |
| **第一性原理** | 回到最原始的问题和约束 | 这个概念最初是为了解决什么问题？ |

**五个锚点**（所有内容都要能回到这里）：

| 锚点 | 一句话理解 |
|:---|:---|
| 🎯 **本质一**：概率模型 | 大模型最底层做的事，是根据前文预测下一个更可能出现的词 |
| 📋 **本质二**：上下文（Context） | 后面很多技术，本质上都在补充、整理和管理模型当前能看到的信息 |
| **主线一**：大模型派 | 更关心模型本身能不能继续变强 |
| **主线二**：驾驭工程派 | 更关心怎样用好模型的能力，让它在真实场景里靠谱地跑起来 |
| 🧩 **框架**：八部件 | 智能体的八个核心部件：大脑、目标、上下文、记忆、工具、边界、观测、循环 |

## 写作红线（从错误中学习）

### ❌ 不要啰嗦

**错误示范**：第五章初稿8个小节，逐部件对比，十几张表格，4500字。

**问题**：初学者被信息淹没，记不住核心观点。

**正确做法**：
- 一章最多3-4个小节
- 每个小节只说一个核心观点
- 表格最多1-2张，能用文字说清就不用表格

---

### ❌ 不要自嗨

**错误示范**：详细解释"第一性原理"、"核心矛盾"、"驾驭工程"等理论框架。

**问题**：读者要的是"这是什么"，不是"我是怎么分析的"。

**正确做法**：
- 理论留给自己，白话留给读者
- 不要用"主要矛盾""次要矛盾"等分析术语
- 让读者通过故事和例子自己领悟

---

### ❌ 不要炫技

**错误示范**：复杂的流程图、架构图、对照表。

**问题**：形式大于内容，核心观点被淹没。

**正确做法**：
- 一个简单比喻胜过一张复杂图表
- 户型图、瑞士军刀、机器人——用生活化比喻
- 让读者能复述，而不是能看图

---

### ❌ 不要忘记受众

**核心提醒**：读者是零基础，不是同行。

| 不要这样写 | 要这样写 |
|:---|:---|
| "八部件框架的透视法应用" | "学会这套方法，看透任何AI产品" |
| "逐部件对比分析" | "它们都有这八个东西，只是用法不同" |
| "场景决定产品形态" | "同样的工具，不同的人用，效果不同" |

### ❌ 不要用学术概念代替生活比喻

**错误示范**：用"有限（7±2个组块）"解释人脑记忆容量。

**问题**：读者是零基础，不需要知道心理学论文里的概念。学术术语虽然精确，但制造了理解门槛。

**正确做法**：
- 用读者熟悉的场景替代学术概念："脑子里同时想几件事，比如'先切菜，再开火'"
- 用烹饪、开会、搬家等日常经验做类比
- 术语只在必要时出现，且首次出现必须解释

### ❌ 不要用抽象代号举例

**错误示范**："A告诉B要做什么，B告诉C...第10步J还记得吗？"

**问题**：ABCD代号让读者无法代入，抽象的场景难以产生共鸣。

**正确做法**：
- 举例要具体、有画面感："开会时，开头说要优化登录流程，结尾却忘了最初要解决什么问题"
- 用职场/校园场景：写代码、做PPT、开会、考试复习
- 让读者能立刻想到"哦，我也遇到过这种情况"

### ❌ 中英混杂

**错误示范**："一个全栈工程师 **sequential** 做"

**问题**：普通词汇用英文显得不伦不类，打断阅读流畅度。

**正确做法**：
- 普通词汇一律用中文："按顺序做"而非"sequential做"
- 英文只在必要时出现：术语首次出现（如"子代理（Sub-Agent）"）、产品名称（如"ChatGPT"）、代码片段
- 检查清单中增加"中英文混杂"检查项

## 章节结构标准

### 章节开头（2块）

**1. 读前先想**

用1-2个口语化问题拉回原始问题：
- ❌ "为什么大模型第一次走进大众视野..."
- ✅ "为什么AI的第一副面孔是聊天框？"

**2. 章节定位**

```markdown
| 维度 | 内容 |
|:---|:---|
| 横向定位 | 这一章在整个项目中的位置 |
| 本章回扣 | 1. 如何回到本质一（概率模型）<br>2. 如何回到本质二（上下文） |
```

---

### 小节结构（6块模板）

每个小节默认包含：

1. **提问**：1-2个精炼问题，聚焦核心疑惑
2. **术语**：中文、英文（缩写）
3. **一句话解释**：最简版本，方便复述
4. **你可以先这样理解**：白话展开，带场景感
5. **示例**：具体场景，帮助代入
6. **你只需要记住**：2-3条核心要点

**灵活使用**：
- 核心概念首次出现：完整使用6块
- 对比/归类/总结性小节：保留"一句话+表格+总结"
- 承上启下的过渡小节：只用"过渡句+核心内容"

---

### 叙事连贯

**小节之间**：用 💭 过渡句承接
```markdown
💭 **上一节我们知道了**：...（回顾）...（抛出本小节问题）
```

**章末结构（本章要义）**：
1. 一句话核心结论（金句）
2. 核心脉络回顾（表格）
3. 回到两个本质（表格）
4. 为下一章铺垫（抛出疑问）

## 三方法检查规范

基于**费曼学习法、苏格拉底式提问、第一性原理**三种学习方法，建立以下检查标准：

### 1. 费曼检验（Feynman Check）

**核心**：用更简单的话重讲，让读者能复述

| 检查项 | 要求 | 问题示例 |
|:---|:---|:---|
| **一句话解释** | 每小节必须有，用大白话概括 | ❌ 第5章仅有1个；❌ 第7-8章缺少 `> 💡 **` 格式 |
| **你只需要记住** | 2-3条要点，方便复述 | ❌ 第5章仅有1处；❌ 格式不统一（有的用表格、有的用列表） |
| **生活化比喻** | 用读者熟悉的场景类比，避免学术概念 | ❌ "有限（7±2个组块）" → ✅ "脑子里同时想几件事" |
| **术语密度** | 每章术语首次出现后，后续用白话替代 | ❌ 一段话出现3个以上未解释的新术语 |

### 2. 苏格拉底检验（Socratic Check）

**核心**：用问题引导思考，不用术语开头

| 检查项 | 要求 | 问题示例 |
|:---|:---|:---|
| **读前先想** | 必须存在，1-2个口语化问题 | ❌ 格式不统一（有的用 `> 💭`，有的直接列表） |
| **问题先行** | 先问"为什么会出现"，再讲"是什么" | ❌ "RAG是一种检索增强生成技术" → ✅ "大模型学了很多，为什么答不上公司新制度？" |
| **小节提问** | ### ❓ 提问 必须用问题开头 | ❌ 用陈述句代替问题 |
| **避免术语开头** | 任何解释不能以术语定义开始 | ❌ "Agent是能够自主决策的AI系统" |

### 3. 第一性原理检验（First Principles Check）

**核心**：回到最原始的问题和约束，回扣两个本质

| 检查项 | 要求 | 问题示例 |
|:---|:---|:---|
| **章节定位** | 必须包含横向定位 + 本章回扣 | ❌ 第5-8章格式不统一（有的缺emoji、有的字段名不一致） |
| **回到本质** | 每章必须明确回扣本质一和本质二 | ❌ 只说"回到本质"但不具体说明如何回扣 |
| **一句话核心结论** | 章末必须有，金句形式 | ❌ 第5-8章缺失；❌ 格式不统一 |
| **章末结构** | 必须包含：金句 → 脉络回顾 → 回到本质 → 铺垫下文 | ❌ 第5-8章普遍缺少完整的"本章要义"结构 |
| **主线归属** | 明确属于主线一/二，或两者兼顾 | ❌ 第6-7章未明确主线归属 |

## 文档元数据规范（YAML Frontmatter）

所有 `docs/` 下的章节文件必须在开头包含统一的 YAML frontmatter，格式如下：

```yaml
---
title: 章节标题（与正文一级标题一致）
slug: 英文短链接（如 ai-programming-workflow）
category: 分类（如 AI基础认知 / AI使用方法）
tags: [标签1, 标签2, 标签3]
status: draft
version: 版本号（如 0.7 / "0.8"）
updated: YYYY-MM-DD
summary: 一句话概括本章核心内容
---
```

### 检查要点

| 字段 | 要求 |
|:---|:---|
| `title` | 必须与正文一级标题保持一致；含引号时用单引号包裹 |
| `slug` | 小写英文，单词间用连字符，简短易读 |
| `category` | 从固定分类中选择：`AI基础认知` / `AI使用方法` / `Agent理解` |
| `tags` | 3-6 个标签，中文或英文，统一格式 |
| `status` | 统一为 `draft`，正式发布后改为 `published` |
| `version` | 数字格式，如 `0.7`（不用引号），每次实质性修改后递增0.1 |
| `updated` | 格式 `YYYY-MM-DD`，最后更新日期 |
| `summary` | 50 字以内，概括本章核心内容，用于章节速览 |

### ❌ 禁止出现

- `description` / `date` / `author` —— 使用 `summary` 和 `updated` 替代
- `layout` / `template` —— 本项目不依赖这些字段
- 引号嵌套错误：`"AI的"手艺""` → 应改为 `'AI的"手艺"'`

## 长文视觉排版规范（2026-04-10新增）

适用于内容较深、篇幅较长的章节（如扩展章节），通过视觉分割提升阅读体验。

### 1. 引用块（>）的使用场景

用于突出核心观点，增强视觉强调：
- **一句话解释**——每小节的核心定义
- **金句/核心观点**——需要读者记住的关键洞察
- **他人引言**——行业专家或报告的重要论断
- **阶段性总结**——段落之间的小结

**示例：**
```markdown
> CLI开源不是技术更新，而是软件产业内部矛盾集中爆发的必然结果。
```

### 2. 分隔线（---）的使用场景

用于划分内容层次，提供视觉休息点：
- **小节内部大段落之间**——不同话题的切换
- **层级内容区隔**——从概念到示例的过渡
- **前后对比**——"过去vs未来"、"问题vs方案"的分割

**示例位置：**
- 在"一句话解释"之后
- 在长列表或表格之前
- 在核心段落与延伸说明之间

### 3. 缩进与列表的层级规范

**有序列表（1. 2. 3.）用于：**
- 步骤说明——有先后顺序的操作
- 阶段划分——演进过程的时间线
- 层级递进——从基础到深入的概念

**无序列表（- - -）用于：**
- 并列要点——同等重要的多个观点
- 特征列举——某一概念的多个维度

**缩进（Tab或4空格）用于：**
- 子层级说明——对上一级的细化
- 示例说明——对概念的实例化
- 补充信息——非核心但相关的延伸

### 4. 表格的替代原则

**优先使用表格的场景：**
- **对比关系**——A vs B 的对照
- **多维度数据**——3个及以上维度的信息
- **结构化概念**——术语定义、阶段划分、框架组件

**表格替代长文本对比的优势：**
- 一目了然，减少阅读负担
- 强制提炼核心差异点
- 便于后续查阅

### 5. 粗体与斜体的强调规则

**粗体（**）用于：**
- 术语首次出现
- 关键转折词（但是、然而、因此）
- 数字和比例的强调（**100倍**、**40%**）

**斜体（*）用于：**
- 语境说明（*注意*、*提示*）
- 轻微的语气强调

### 6. 段落长度控制

**理想段落长度：**
- 普通段落：2-4句话
- 核心观点段：1-2句话（配合引用块）
- 列表项：不超过2行

**段落分割信号：**
- 逻辑转折时——新起一段
- 举例说明时——新起一段
- 递进深化时——新起一段

### 7. 代码块与示意图的使用

**代码块用于：**
- CLI命令示例
- 架构层级展示
- 流程逻辑说明

**示意图（ASCII或Mermaid）用于：**
- 系统架构
- 流程流转
- 层次关系

### 应用示例

`docs/extras/05-当软件不再给人用.md` 作为首个完整应用该规范的章节，可作为后续长文章的参考模板。

## 从错误中学习的规范（2026-04-09更新）

基于全面修复工作总结的新规范：

### 🎯 核心概念一致性检查

**新增/修改/删除核心概念时必须**：
- [ ] 列出所有已有核心概念（两个本质、两条主线、八部件框架）
- [ ] 确认变更是新增、修改还是替换
- [ ] 使用全局搜索确认所有相关章节
- [ ] 同步修改所有受影响章节

**常见错误**：
- ❌ 第4章6部件，第5章变成8部件
- ✅ 建立`indexes/核心概念词典.md`，写作前查阅

### 📋 章节结构调整SOP

**增删改章节位置时必须**：
1. 修改目标章节的文件位置/名称
2. 更新本章的"章节定位"
3. 更新前一章的"为下一章铺垫"

**常见错误**：

- ✅ 结构调整后用此清单逐项检查

### 👥 受众校准强制要求

**每章写作前必须回答**：
1. 这一章的读者在此之前知道了什么？
2. 这个概念对他们有用吗？
3. 是否提供了非技术读者的入口？
4. 例子是否都是他们熟悉的场景？

**硬性要求**：
- 每章至少1个非技术示例（技术章节用📌框标注"方法通用"）
- 术语首次出现必须解释，后续用白话替代
- 禁止假设读者有专业背景

**常见错误**：
- ❌ 第8章全是编程示例，零基础读者看不懂
- ✅ 第8章增加"写项目方案""组织活动"等通用示例

### 🔄 概念生命周期管理

**首次出现（预告）**：
- 简单解释概念是什么
- 明确标注"后面会详细讲"
- 不展开技术细节

**正式讲解**：
- 开头标注"前面预告过X，这一章正式深入"
- 完整展开概念内涵
- 提供具体示例

**后续引用**：
- 不再重复解释，直接引用
- 如需复习，提供回链

**常见错误**：
- ❌ MCP/CLI/Skill多次出现，读者觉得重复
- ✅ 第2章预告→第7章正式讲→后续直接引用

### 🗣️ 术语简化指南

**原则**：
- 首次出现：中文（English，缩写）
- 后续使用：只用中文
- 禁止：中英混杂（如"sequential做"）

**学术术语替代表**：
| 学术表达 | 简化表达 |
|:---|:---|
| 规范驱动开发（SDD） | 先写需求再动手 |
| 规格即代码 | 需求文档优先 |
| 单一事实来源 | 以需求文档为准 |
| 约束驱动 | 按规矩来 |
| 可回溯性 | 有据可查 |

**检查方法**：
- 读给非技术朋友听，不懂就换说法
- 用"说白了"开头重写

---

### 📚 示例复杂度控制

**简单示例**：
- 1-2个要点
- 一句话能说清
- 用于快速理解概念

**完整示例**：
- 最多3步
- 每步一个核心动作
- 用于说明实际应用

**禁止使用**：
- Step 1-2-3-4-5 的冗长流程
- 大段代码/模板（除非必要）
- 多个示例连续堆砌

**替代方案**：
- 流程长→拆成多个小节
- 代码长→提供伪代码或核心逻辑
- 多个例子→只保留最典型的1个

**常见错误**：
- ❌ 第6章"周报生成器"Step 1-3占大量篇幅
- ✅ 简化为一个完整流程示例

|:---|:---|
| `README.md` | 项目简介、阅读入口 |
| `docs/` | 知识库正文，按顺序编号 |

## 协作约定

1. **先理解，再写作**：用户口述想法时，先整理成结构化要点
2. **敢纠正**：如果用户表述不准确，主动纠正，不要为了顺从而保留错误说法
3. **主动收窄**：发现内容偏离"简单、可复述、贴近工作场景"时，主动提出简化方案
4. **先问后写**：不确定时先问清楚，不要基于猜测写大段内容
