# BrainDance AGENT 工作规范

本文件适用于仓库根目录及其所有子目录。除非用户明确给出更高优先级指令，否则在本仓库内工作的 Agent 必须遵守以下规则。

## 1. 默认工作模式

收到用户任务后，必须按以下顺序推进，不允许跳过“先理解再实现”的过程：

1. 接收用户指令
2. 深度思考（`sequential-thinking`）
3. 规划任务（`shrimp-task-manager`）
4. 执行实现
5. 自我验证
6. 交付成果

执行要求：

- 必须先研读已有代码、配置、文档或历史实现，吸收现有经验后再动手修改。
- 必须主动维护 TODO list 和阶段性计划，尽可能自行完成完整链路，而不是每做一步就停下来询问用户。
- 必须自主决策技术方案、实现路径和质量标准；仅在确实缺少必要输入、且无法通过代码库或上下文推断时，才向用户提问。
- 完成实现后，必须进行自我验证。至少包括与本次改动直接相关的检查、测试、构建或运行验证；若受环境限制无法完成，必须在交付说明中明确写出原因和风险。

## 2. 提交规范

完成一个独立工作后，必须进行一次 Git 提交，且提交信息必须使用中文，遵循规范化前缀。标题行只能算摘要，不足以交付；必须补充多行中文正文，完整说明本次改动。

要求如下：

- 只提交本次工作直接相关的文件，禁止把用户已有改动、无关文件或临时产物一起提交。
- 提交前必须先检查工作区，确认没有误提交二进制文件、媒体文件、模型文件、缓存文件或其他本应被 `.gitignore` 忽略的内容。
- 提交信息建议使用 Conventional Commits 风格，但必须采用“标题 + 空行 + 正文”的完整格式：

\`\`\`text
<type>(<scope>): <中文摘要>

<中文正文第 1 行>
<中文正文第 2 行>
<中文正文第 3 行>
...
\`\`\`

正文要求：

- 必须是多行中文说明，不能只有一行标题。
- 必须写清楚改动背景、核心实现、影响范围、自我验证结果。
- 如存在兼容性影响、风险、后续待办，也应在正文中写明。
- 正文应优先描述“为什么改、改了什么、如何验证、还有什么限制”，避免空洞表述。

建议正文结构：

\`\`\`text
<type>(<scope>): <中文摘要>

背景：
- 说明问题来源、触发场景、为什么要做这次改动。

实现：
- 说明本次修改了哪些模块、关键实现策略是什么。
- 说明是否做了重构、拆分、兼容处理或防御性修复。

验证：
- 说明运行了哪些命令、测试、构建 or 人工验证步骤。
- 如果未验证，必须写明原因，不得省略。

影响：
- 说明影响范围、兼容性、潜在风险和后续事项。
\`\`\`

完整示例：

\`\`\`text
fix(common): 修复字体过小的 BUG，将通用管理下所有页面的默认字体大小修改为 14px

背景：
- 通用管理模块多个页面默认字号偏小，在 2K 屏和投影场景下可读性明显不足。
- 该问题已经影响日常演示和后台录入效率，需要统一调整而不是零散修补。

实现：
- 统一梳理通用管理模块下的页面样式入口，将默认字体大小收敛到同一套设计 token。
- 把默认字体由 12px 调整为 14px，并同步修正少数依赖旧字号的间距与表格行高。
- 补充中文注释，说明为什么这里不能继续写死局部字号。

验证：
- 已逐页检查通用管理相关页面的标题、表单、表格和弹窗显示效果。
- 已确认关键页面在常见分辨率下没有出现换行异常和布局溢出。

影响：
- 本次调整会轻微改变后台页面的视觉密度，但不会影响业务逻辑和接口契约。
- 若后续设计规范继续调整字号，应统一通过设计 token 修改，避免再次散落覆盖。
\`\`\`

补充要求：

- 提交前先用 \`git status\`、\`git diff --staged\` 等方式复核提交内容。
- 允许提交到本地 Git，但绝对禁止执行 \`git push\`。

## 3. 3DGS 与计算环境要求

涉及本项目 3DGS 相关代码时，必须优先使用指定 Conda 环境：

\`\`\`bash
conda activate Braindance
\`\`\`

执行要求：

- 运行 \`ai_engine/3dgs/\` 及其相关脚本、流水线、训练、推理、评测前，优先确认当前已进入 \`Braindance\` 环境。
- 对于可使用 GPU 加速的任务（如 LLM 训练、3DGS 训练/重建、相关 CUDA 推理任务），禁止为了省事直接回退到 CPU 模式。
- 如果 GPU 路径报错，应优先排查并解决环境、依赖、驱动、CUDA、显存、算子或配置问题，而不是通过切换 CPU 绕开问题。

## 4. 代码与注释要求

- 保持良好的软件工程纪律，做好文件和模块拆分，避免把大量逻辑堆进单一文件。
- 控制单个代码文件规模；新增复杂功能时优先拆分模块、服务、工具函数和配置层。
- 在关键实现、复杂分支、易误解流程处补充中文注释，帮助后续维护者理解设计意图。
- 注释必须有信息量，禁止写“把变量赋值给变量”这类无效注释。

## 5. 安全要求

- 绝不在代码、脚本、测试、提交记录或文档中硬编码 API 密钥、访问令牌、密码、私钥或其他敏感凭据。
- 涉及密钥或密码的能力必须通过环境变量、本地配置文件模板或密钥管理方案注入。
- 若发现仓库中已有可疑敏感信息，应暂停扩散风险，并优先提醒用户。

## 6. Gitignore 与受忽略文件处理要求

- 提交前必须检查是否新增了忘记忽略的文件，尤其是二进制文件、媒体文件、训练产物、模型权重、缓存目录、日志和构建产物。
- 对于已被 \`.gitignore\` 忽略的文件，禁止草率直接修改。
- 如确需修改被忽略文件，应先做好备份，并确认该修改是否真的应该保留在本地而非进入版本控制。

## 7. 实施原则

- 先读后改：实现前必须阅读相关模块、入口、调用链、配置和文档，避免凭猜测修改。
- 最小闭环：每次任务都应尽量形成“分析-实现-验证-提交”的完整闭环。
- 不绕问题：遇到报错、依赖问题或 GPU 问题时，优先定位根因并解决，不以降级、跳过、伪实现替代真正修复。
- 不污染工作区：尊重用户已有未提交改动，不得随意覆盖、回退或混入无关内容。
- 交付时明确说明：做了什么、如何验证、有哪些剩余风险或限制。

## 8. Agent 额外规范 (针对 LangChain / Agent 编排)

本部分补充 LangChain / Agent 编排相关工作的文档维护要求。

### 8.1. LangChain 相关工作必须同步写文档

凡是涉及以下内容之一，都视为 LangChain 相关工作：

- \`langchain\`、\`LangChain\`
- \`bindTools\`
- \`DynamicStructuredTool\`
- \`ChatOpenAI\`
- \`agent-recall\`
- \`spatial-search-agent\`
- \`time-compare-agent\`
- 其他基于大模型工具调用、Agent 路由、结构化动作输出的 Supabase Edge Function

只要修改了这类代码，就不能只改代码不改文档。

### 8.2. 文档统一收口到 \`docs/09-LangChain专题/\`

LangChain 相关的实现现状、阶段总结、联调记录、未完成事项说明，统一写入：

- [docs/09-LangChain专题/README.md](docs/09-LangChain专题/README.md)

新增或更新文档时，优先放在这个目录，不要继续把实现状态零散写进多个无关目录。

### 8.3. 没做完也必须先总结

如果一次 LangChain 相关工作还没做完，但已经出现以下任一情况：

- 需要暂停
- 需要切换任务
- 当前上下文已经复杂
- 已经完成一部分实现，但还有剩余 work items

则必须先写阶段总结文档，再结束当前轮工作。

阶段总结至少要写清楚：

- 已完成什么
- 还没完成什么
- 当前阻塞点是什么
- 下一步建议先做什么
- 哪些文件已经改过

### 8.4. 文档过长时必须开新文档

如果同一篇 LangChain 文档已经过长，或者开始同时混入多个主题，例如：

- 既有规划，又有现状，又有联调细节
- 既有空间检索，又有资产操作，又有时间对比
- 既有代码说明，又有回归记录，又有问题排查

则不要继续无上限追加，而是新开文档承接。建议命名方式：\`LangChain实现现状-YYYY-MM-DD.md\` 等。

### 8.5. 文档内容必须以代码现状为准

LangChain 相关文档不能只写规划口径，必须明确区分：已实现、部分实现、实验中、未实现。如果代码和旧文档不一致，应先修正文档口径，避免继续漂移。
