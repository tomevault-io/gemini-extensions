## Non-Negotiables (always true, no exceptions)

1. **Never raise bare `HTTPException`** ‚Äî use `APIError` / `not_found_error()` / `rate_limit_error()` etc. from `utils/error_responses.py`
2. **Never call `flag_modified()` zero times on a JSONB mutation** ‚Äî every write to a JSONB field needs it
3. **Background tasks use `get_session()`**, not `get_database()` (request-scoped)
4. **All `run_in_executor` LLM calls must be wrapped in `asyncio.wait_for()`**
5. **Never add inline `onclick=` / `onchange=` attributes** ‚Äî use event delegation + `data-action`
6. **Never use `random` for security values** ‚Äî always use `secrets` (tokens, codes, state parameters)
7. **Never put a JWT in an HTTP redirect URL** ‚Äî use the OAuth exchange-code pattern; WebSocket `?token=` is the only exception
8. **Never return user-existence hints from unauthenticated endpoints** ‚Äî forgot-password and resend-verification always return identical responses
9. **`ENCRYPTION_KEY` must be set before any `JWT_SECRET` rotation** ‚Äî missing it corrupts all stored BYOK keys
10. **`DEBUG` must be `false` in any shared environment** ‚Äî debug mode enables insecure code paths
11. **Never hardcode `/static/js/` or `/static/css/` in templates** ‚Äî always use `{{ asset_url('js/...') }}` / `{{ asset_url('css/...') }}` for cache busting
12. **Never write `except Exception: pass`** ‚Äî swallowed exceptions are invisible in production; always log at minimum `logger.debug(..., exc_info=True)`
13. **Every background task top-level `except` must call `await report_exception(exc, user_id=user_id)`** and log with `exc_info=True` ‚Äî background tasks bypass FastAPI's global exception handler
14. **Every `logger.error(...)` inside an `except` block that references an exception variable must include `exc_info=True`** ‚Äî without it the full traceback is lost in production logs
15. **After any bulk search-and-replace on Python files, validate syntax with `ast.parse()`** ‚Äî naive regex replacements can introduce double-quote artifacts (`f"...{e}"", exc_info`) or missing parentheses; grep for `}"",` after any bulk exc_info fix
16. **URL fields in dataclasses must validate the scheme** ‚Äî `job_url` and any other URL input must assert `startswith(("http://", "https://"))`; never accept arbitrary strings as URLs
17. **Gemini health checks must use `client.models.get(model=...)` or `client.models.list()`, never `generate_content()`** ‚Äî model metadata fetches are quota-free; `generate_content` burns a real token on every server restart and triggers 429s on quota-limited projects
18. **429 / RESOURCE_EXHAUSTED from Vertex AI or Google AI Studio means the service is reachable** ‚Äî treat it as healthy (`return True`) in health-check paths; do not surface it as "service unavailable"
19. **`requirements.txt` must be regenerated with `pip freeze` after any package change** ‚Äî never hand-edit pinned versions; always run `pip check` afterwards to confirm zero broken requirements
20. **macOS dev setup must go through `make setup`** ‚Äî it strips the `com.apple.quarantine` attribute from all venv `.so` files **and** `node_modules` binaries (esbuild); without it Gatekeeper blocks them with a security dialog. Use `make build-frontend` instead of `npm run build` directly on macOS
21. **`datetime.strptime()` results must have `.replace(tzinfo=timezone.utc)` before any comparison with a timezone-aware datetime** ‚Äî `strptime` always returns a naive datetime; comparing naive vs aware raises `TypeError` at runtime; this applies to every parsed date in Pydantic validators and anywhere else dates are compared
22. **In development, `npm run build` takes effect immediately without a server restart** ‚Äî `_load_asset_manifest()` re-reads `manifest.json` from disk on every request in non-production environments; a browser hard-refresh is all that's needed
23. **`check_account_lockout()` returns a tuple ‚Äî always unpack it, never use it as a bare bool** ‚Äî `if await check_account_lockout(email):` is ALWAYS truthy (any non-empty tuple is truthy), which permanently blocks every request; always write `is_locked, remaining = await check_account_lockout(email)` then `if is_locked: raise account_locked_error(retry_after=remaining)`
24. **`app.js` is only loaded on the landing page (`index.html`), not in `base.html`** ‚Äî `window.app` is `undefined` on all dashboard pages; every dashboard page JS file must register its own document-level click listener for `data-action="logout"` (and any other global navbar actions) rather than relying on `app.js`'s listener (see `frontend-js-strict.mdc`)
25. **Landing navbar must use `navbar-expand-xl`, not `navbar-expand-lg`** ‚Äî the landing nav has 6 links + 2 CTA buttons that wrap at tablet widths (992‚Äì1199px) with the lg breakpoint; xl collapses at 1200px and prevents the wrapping
26. **Never use `margin: 0 -Xrem` on `.navbar-collapse`** ‚Äî negative margin clips nav item text at the left edge on mobile; use `padding` on the child `.nav-cta` or `.navbar-nav` instead
27. **All horizontal tab/pill bars must be `overflow-x: auto; flex-wrap: nowrap` at `‚â§768px`** ‚Äî never let tab bars wrap to a second line or overflow the viewport; see `mobile-responsive.mdc` for the full scrollable pattern
28. **Playwright mock tokens must be valid 3-part JWTs** ‚Äî many client-side scripts check `token.split('.').length === 3` and redirect to `/auth/login` on failure; always use a real `header.payload.sig` string, never a bare string like `'mock-token'`
29. **Playwright `addInitScript` cookie consent must include `version: '1.0'`** ‚Äî `cookie-consent.js` re-shows the banner if the stored value lacks `version`; the banner overlays the page and intercepts all pointer events, breaking every subsequent test interaction
30. **Playwright API mock data for array fields must always be arrays** ‚Äî client render functions call `.map()` on fields like `what_to_bring`, `pre_interview_checklist`, `confidence_boosters`, `virtual_interview_tips`, and `three_key_selling_points`; providing a string causes `TypeError: s.map is not a function` inside a `try/catch`, which triggers `showError()` and keeps the main content hidden permanently
31. **`APIError` accepts an optional `headers` kwarg** ‚Äî `APIError(error_code, message, status_code=401, headers={"WWW-Authenticate": "Bearer"})` is valid; the headers are forwarded to `HTTPException.__init__`
32. **Integration tests live in `tests/test_api/`, NOT in `tests/`** ‚Äî the root-level `tests/test_auth.py`, `tests/test_tools.py` etc. are **live-server tests** that require `localhost:8000` running; never run them in CI without a live server; for CI use `tests/test_api/` (no server needed) or `tests/test_agents/` (unit tests)
33. **`asyncio_default_fixture_loop_scope = session` is required in `pytest.ini`** ‚Äî `function`-scoped loops cause asyncpg "Future attached to a different loop" errors when the SQLAlchemy NullPool engine is reused across test functions; session scope ensures all async tests share one event loop
34. **Never add `style="..."` HTML attributes to templates** ‚Äî the CSP `style-src` directive uses `'nonce-{value}'` with no `'unsafe-inline'`; inline `style=` attributes are blocked by the browser; use CSS classes instead (see `security-middleware.mdc` for the full class reference)
35. **Every `<style>` block in a template must include `nonce="{{ request.state.csp_nonce | default('') }}"`** ‚Äî both `script-src` and `style-src` are nonce-gated; a `<style>` block without a nonce is silently blocked; never write a bare `<style>` tag in any `.html` file
36. **`.is-hidden { display: none; }` has no `!important` ‚Äî never add it** ‚Äî JavaScript shows hidden elements via `element.style.display = 'block'` (inline style overrides non-`!important` class); adding `!important` would permanently lock elements hidden and break all JS-driven show/hide
36a. **When a page `<style>` block sets `display` on a class, add a companion `.that-class.is-hidden { display: none }` rule** ‚Äî a single-class rule like `.job-badge { display: inline-flex }` has equal specificity (0,1,0) to `.is-hidden { display: none }` in `app.css`, but the page style wins (loads last), silently making all instances of that class permanently visible; the fix is a two-class rule (0,2,0 specificity) in the same `<style>` block: `.job-badge.is-hidden { display: none }`
37. **Never use native `confirm()`, `alert()`, or `prompt()`** ‚Äî use `window.showConfirm()` from `confirm-modal.js` (loaded globally in `base.html`); native dialogs are unstyled, block e2e automation, and are inconsistent with the app design
38. **Never inject `<style>` elements from JavaScript** ‚Äî dynamically created `<style>` tags cannot carry a server-issued nonce and are silently blocked by the CSP `style-src` policy; all styles must be in a template `<style nonce="...">` block or in a CSS file loaded via `<link>`
39. **`make setup` and `make dev` both run `build-frontend` automatically** ‚Äî never skip the build step; stale dist files cause silent CSS/JS bugs (e.g. utility classes defined in source but missing from compiled output)
40. **`server_has_key` in `/api/v1/profile/api-key/status` is `True` when either `gemini_api_key` is set OR `use_vertex_ai=True`** ‚Äî both mean the user never needs to provide their own key; the frontend onboarding check must also include `use_vertex_ai` as a fallback: `has_user_key || server_has_key || use_vertex_ai`
41. **Never store the JWT token during registration ‚Äî only store it after email verification** ‚Äî `auth-register.js` must NOT call `storeAuthData()` on registration success; the `/auth/verify-code` endpoint issues its own fresh token on success which `auth-verify-email.js` stores; storing the pre-verification token lets users bypass email verification entirely by navigating away from the verify page
42. **`localStorage.clear()` must preserve `cookie_consent`** ‚Äî save/restore pattern required: `const cc = localStorage.getItem('cookie_consent'); localStorage.clear(); if (cc) localStorage.setItem('cookie_consent', cc);` ‚Äî only account deletion uses `clear()`; all other logouts use targeted `removeItem('access_token')` + `removeItem('authToken')`
43. **Auth page templates must use `<main class="auth-page">` inside `{% block content %}`, NOT `{% block body_attrs %} class="auth-page"`** ‚Äî `base.html` renders `.bg-glow` divs before the content block; if `<body>` is the flex container those divs become flex siblings of `<main>` and push the card off-center; `<main class="auth-page">` wrapping only the card is the correct pattern (matches `login.html`, `register.html`)
44. **Cookie consent CSS lives in `app.css` ‚Äî never in JS dynamic style injection** ‚Äî `cookie-consent.js`'s `_addStyles()` method is intentionally emptied; all cookie banner and modal CSS is in `app.css`; the old dynamic `document.createElement('style')` approach was blocked by CSP (same rule as onboarding.js)
45. **`btn-primary` and `btn-outline-primary` are globally styled in `app.css`** ‚Äî they use `var(--accent-gradient)` and `var(--accent-primary)` app-wide; never re-define `.btn-primary` per-page (e.g. in a `<style>` block) unless overriding for a very specific local context; the global rule ensures the cyan‚Üípurple gradient appears consistently on every page
46. **A 403 from `get_current_user_with_complete_profile` must show a friendly empty state, not a raw error** ‚Äî the `list_applications` endpoint (and any other endpoint guarded by this dependency) returns 403 when `profile_completed=false`; the frontend must handle it with a centered empty-state UI (icon + title + "Finish Setup" button linking to `/profile/setup`), never with `Failed to load: 403 Forbidden`
47. **URL-parameter messages on the login page must use `persist = true` in `showAlert()`** ‚Äî messages from `?verified`, `?registered`, `?password_reset`, `?account_deleted` params auto-clear in 5 s by default; users filling in the login form miss them; pass `persist = true` (third arg) to keep them visible with a dismiss button until the user explicitly closes them
48. **Auth tokens are stored in `localStorage`, not HTTP cookies** ‚Äî `access_token` and `authToken` live in localStorage; clearing browser cookies does NOT log the user out. Never describe auth as "cookie-based" in documentation, help text, or privacy policy. The `cookie-consent.js` consent preferences are also in localStorage, not cookies
49. **`UserProfile` has no `education` field, no `target_role` field, and no `location_preferences` field** ‚Äî the profile sections are: basic info (`professional_title`, `years_experience`, `summary`), location (`city`, `state`, `country`), work experience (JSONB), skills (JSONB), and career preferences (salary, job types, company sizes, work arrangement, relocation, visa, clearance, travel). Any content, UI label, or privacy text referencing the missing fields must be corrected
50. **Google OAuth stores only `email`, `full_name`, and `google_id` ‚Äî never `profile_picture`** ‚Äî the callback does not read or persist the `picture` field from Google's userinfo endpoint. Never reference `user.profile_picture` or `user.avatar`; the field does not exist on the `User` model
51. **Dashboard JS must hard-redirect to `/profile/setup` when `profile_completed` is false ‚Äî never show a banner** ‚Äî `loadUserData()` in every dashboard page JS file checks `data.completion_status?.profile_completed`; if false, immediately execute `window.location.href = '/profile/setup'; return;` ‚Äî a soft banner is wrong because the extension and all API endpoints will fail or return 403 for incomplete profiles
52. **All navbars must use the `.brand-icon` + `fa-rocket` + `.brand-text` + `.brand-accent` structure** ‚Äî the correct logo markup is `<div class="brand-icon"><i class="fas fa-rocket"></i></div><span class="brand-text">Apply<span class="brand-accent">Pilot</span></span>` (no space between Apply and Pilot); never use `fa-briefcase`, bare icon with `me-2`, `text-accent` instead of `brand-accent`, or a custom `.navbar-brand` CSS override in a page `<style>` block; dashboard navbars link to `/dashboard`, public/auth navbars link to `/`
53. **`base.html` already loads Bootstrap and Font Awesome ‚Äî never re-import them in `{% block extra_head %}`** ‚Äî duplicate CDN imports load after `app.css` in the cascade; since both Bootstrap and `app.css` use `!important` on utility classes like `.text-muted`, the last-loaded one wins and Bootstrap's colors override the dark-theme values, making text invisible
54. **Never mention specific third-party job site names in any user-facing text** ‚Äî do not write LinkedIn, Indeed, Glassdoor, ZipRecruiter, Monster, or any other job board name in UI copy, marketing text, landing page content, help docs, onboarding tours, or extension descriptions; use generic phrasing: "any job site", "any job board", "company careers pages"
55. **BYOK is for Google Gemini / AI Studio API keys ‚Äî Vertex AI is server-only** ‚Äî end users cannot bring their own Google Cloud project or Vertex AI credentials; `USE_VERTEX_AI` is a server-admin environment variable, not a user setting. Never document or suggest a "use your own Google Cloud project" path for users; key format validation is intentionally loose because Google rotates key shapes
56. **The workflow runs exactly 5 agents ‚Äî there is no "Strategy Advisor"** ‚Äî the agents in order are: Job Analyzer (step 1) ‚Üí Profile Matcher (step 2, gate) ‚Üí Company Research (step 3) ‚Üí Cover Letter Writer + Resume Advisor (step 4, parallel). Strategy tab content (`competitive_positioning`, `application_strategy`) is produced by the Profile Matcher agent. Never refer to six workflow agents or a separate Strategy Advisor in documentation, help text, or code comments
57. **The Chrome extension popup sends only `job_text` as FormData ‚Äî not title, company, or source URL** ‚Äî `extractAndSubmitJob()` in `popup.js` sends a single `job_text` field; the backend's Job Analyzer agent extracts the title and company from the text. `detected_title`/`detected_company` appear only in the background service worker flow, not the popup submit path
58. **`thinking_budget=0` must always be set in `GenerateContentConfig` for both Vertex AI and BYOK paths** ‚Äî Gemini 3 Flash (and 2.5 Flash) enables thinking mode by default; thinking tokens consume the output token budget, producing slow responses (40‚Äì75s) with silently truncated JSON output; `types.ThinkingConfig(thinking_budget=0)` disables it entirely; see `utils/llm_client.py` `_generate_with_vertex_ai` and `_generate_with_google_ai`
59. **`google-generativeai` is removed from this project ‚Äî never add it back** ‚Äî both Vertex AI and BYOK paths use `google-genai` (`Client(vertexai=True,...)` and `Client(api_key=...)` respectively); `google-generativeai` does not support `ThinkingConfig` and has been deleted from `requirements.txt`
60. **`sanitizeText()` and backend description/name patterns must strip only ASCII control characters ‚Äî never restrict to an ASCII allowlist** ‚Äî LLMs use Unicode bullet characters (`‚ñÝ`, `‚ñ™`, `‚ñ∏`, `‚Äì`, `‚Äî`) in parsed descriptions; the correct pattern is `/[\x00-\x08\x0B\x0C\x0E-\x1F\x7F]/g` (strip controls only); the old ASCII-only allowlist silently stripped all non-ASCII characters including bullets
61. **`completeProfile()` must validate all steps upfront before any API save calls** ‚Äî the correct order is: validate step 1 ‚Üí validate step 2 ‚Üí validate step 3 ‚Üí validate step 4 ‚Üí then save all; interleaving validate-then-save per step causes steps 1‚Äì3 to save before the user sees a step 4 validation error, making the form appear to "try to continue" for several seconds before failing
62. **`changeStep()` must NOT clear the error alert element** ‚Äî validation functions call `showErrorMessage()` then `changeStep()` is called to navigate to the failing step; if `changeStep()` hides the error, the user sees the navigation but no explanation; `goToNextStep()` already calls `hideAlerts()` before `changeStep()` for the forward-navigation case
63. **`escapeHtml()` must decode `&amp;` FIRST, then named entities ‚Äî the decode order is non-negotiable** ‚Äî `bleach.clean()` double-encodes `&` in HTML entities produced by Python's `html.escape` (e.g. `'` ‚Üí `&#x27;` ‚Üí `&amp;#x27;`); if `&#x27;` is decoded before `&amp;`, the compound entity is never resolved and `&#x27;` renders literally in the page; the canonical order is: `&amp;` ‚Üí `&`, then `&#x27;` ‚Üí `'`, then `&#039;` ‚Üí `'`, then `&quot;` ‚Üí `"`, then `&lt;` ‚Üí `<`, then `&gt;` ‚Üí `>`, then re-encode. This applies to EVERY `escapeHtml()` in every dashboard JS file.
64. **`.textContent` assignments of server-rendered strings must use `decodeEntities()`, not raw strings or `escapeHtml()`** ‚Äî `.textContent` does not interpret HTML entities so `&#039;` would display literally; use a decode-only helper (no re-encode step); `escapeHtml()` is for `.innerHTML` only. Example: `el.textContent = decodeEntities(job.job_title)` not `el.textContent = job.job_title`.
65. **`.filter-select-auto` must use `min-width: max-content; flex-shrink: 0` ‚Äî never `min-width: 0; flex-shrink: 1`** ‚Äî the flex-shrink pattern clips option text for longer labels like "Company A‚ÄìZ" and "Recently updated"; `max-content` ensures the element is always wide enough for its currently-selected option text.
66. **`analysis_complete` applications must show a "Generate Cover Letter & Resume Tips" button in the Cover Letter and Resume tabs** ‚Äî when `renderCoverLetter` or `renderResumeTips` finds no content and `currentSessionId` is set, render a `.btn.btn-primary` that calls `generateDocuments()`; the function POSTs to `/api/v1/workflow/generate-documents/{session_id}` and polls `/api/v1/workflow/status/{session_id}` every 3 s until the workflow reaches `completed` or `failed`, then calls `loadApplicationData()` to refresh the page.
67. **`years_experience` is total career years ‚Äî never use it as domain-specific tenure in agent prompts** ‚Äî all six content agents (`profile_matching`, `resume_advisor`, `cover_letter_writer`, `interview_prep`, `job_comparison`, `salary_coach`) carry a SYSTEM_CONTEXT rule: derive domain-specific year counts only from relevant work history entries; if uncertain, say "experience with [skill]" without a number. Copying this rule to any new agent that uses profile data is mandatory.
68. **Company `website` field must be validated as a real URL before rendering a link** ‚Äî check `/^https?:\/\//i.test(website)` before building any `<a href="...">` around the company website value; LLMs sometimes return `"Unknown"`, `"N/A"`, or other placeholder strings that would navigate the user to a relative path (e.g. `/Unknown`) and show a 404 error page.
69. **The `.percentile-marker` position must be set via JavaScript after `innerHTML` ‚Äî never with `style="left: X%"` inside an HTML string** ‚Äî set a `data-pct` attribute in the template string, then after assignment do `marker.style.left = marker.dataset['pct'] + '%'`; the inline style approach is unreliable in dynamically injected HTML.
70. **All subpage navbars (Help, Terms, Privacy, Settings, application detail) must use `navbar_subpage.html` or identical CSS** ‚Äî `.navbar` padding `1rem 0`, `.brand-icon` `36√ó36px / 1rem`, `.brand-text` `1.2rem`; this matches the dashboard exactly. The landing page logo also uses these same values (`landing.css`). Never let any page use a custom floating back-link element.
71. **The Help page link in the dashboard navbar must be `/help?from=dashboard`** ‚Äî the Help page reads this parameter and switches the back button to "‚Üê Back to Dashboard"; without it the button always returns to the landing page regardless of where the user came from.
72. **`POST /api/v1/workflow/start` returns `RES_3002` (409) when the user already has a non-deleted application for the same job** ‚Äî matched by canonical URL, normalized title+company, or content fingerprint (‚â•80 chars of normalized text); dashboard and extension must show a **warning/info** message, not a generic error; see `applypilot-core.mdc` (`RES_3002`)
73. **`profile-setup.js` must `PUT` empty `work_experience` when ‚Äúno relevant experience‚Äù is checked** ‚Äî skipping the API call leaves `user_profiles.work_experience` as SQL `NULL`, so `POST /profile/complete` fails with ‚ÄúWork Experience‚Äù incomplete; `[]` in JSONB means the user explicitly completed step 2 with no roles (`_check_work_experience_completion` treats `NULL` as incomplete and non-null `[]` as complete).
