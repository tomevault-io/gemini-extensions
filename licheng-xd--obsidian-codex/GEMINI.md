## 项目概述

`obsidian-codex` 的目标是实现一个桌面端 Obsidian 社区插件，将 Codex 工作流嵌入 Obsidian，参考 `YishenTu/claudian` 的产品方向，但不机械复刻其全部功能。

当前阶段以可落地的 MVP 为优先：先建立稳定的插件骨架、清晰的会话边界、可控的上下文注入方式，以及对本地代理能力的安全封装。

这里的 `Codex` 明确定义为 OpenAI Codex，而不是 Claude Code 的别名，也不是泛指“任意代码助手”。

## 核心原则

### 1. 语言与文档

- 与用户沟通默认使用中文。
- 项目文档默认使用中文，必要的 API 名称、类型名、命令名保留英文。
- 代码标识符、文件名、目录名、提交信息中的技术关键词使用英文。
- 注释优先解释“为什么”，避免解释显而易见的“做了什么”。

### 2. 产品定位

- 目标用户：在 Obsidian 中进行知识管理、写作或开发辅助的高级用户，尤其是已经在本机使用 Codex/CLI 工作流的用户。
- 核心价值：让用户在不离开 Obsidian 的前提下，获得面向当前 Vault 的 AI 协作能力。
- 范围边界：第一阶段优先支持桌面端，不以移动端兼容为目标。
- 参考策略：借鉴 Claudian 的交互与架构思路，但所有功能都必须重新评估是否适合 Codex。

### 3. 开发规范

- 默认技术栈：TypeScript + Obsidian Plugin API + esbuild。
- 开发流程：设计先行，小步迭代，先验证边界再扩功能。
- 非平凡功能改动需要先明确目标、约束、方案对比和验收标准。
- 优先做最小可交付版本，不做未经验证的“大而全”实现。
- MVP 阶段避免为“未来可能的第二实现”提前抽象接口；出现第二种真实实现需求后再提炼通用层。

### 4. 角色职责

- 用户：定义产品目标、确认范围、审批关键设计取舍。
- Codex 代理：负责调研、设计、实现、验证、更新文档。
- 插件代码：保持模块边界清晰，避免 UI、会话管理、命令执行、安全策略互相耦合。

### 5. 决策流程

- 涉及架构、权限模型、数据存储、CLI 集成方式的决策，必须先说明备选方案和取舍理由。
- 一旦做出跨模块决策，需要记录到 `docs/adr/`。
- 当前阶段的普通实现决策和范围调整，优先直接更新当前设计文档。
- 进度、阻塞项、验证结果优先记录在根目录代理工作文件中。

### 6. 技术理念

- 本地优先：优先集成 OpenAI 官方 `codex` 本地工作流或其官方推荐接入方式，而不是先做自定义协议层。
- 安全优先：任何涉及文件写入、命令执行、外部路径访问的能力，都必须有清晰的权限边界和默认安全策略。
- 可验证优先：每个阶段都必须能通过构建、类型检查或最小手工验证证明其状态。
- 演进优先：先做一个可工作的后端集成，再根据真实扩展需求重构。
- Obsidian 原生优先：遵循 Obsidian 插件生态的目录结构、命令注册、设置页、视图生命周期管理方式。

### 7. 代码与项目维护

- 单个模块只承担一个明确职责。
- 能写成纯函数的逻辑，不依赖 Obsidian 运行时。
- 所有外部副作用都要集中封装，便于测试和权限控制。
- 不提交密钥、令牌、本地路径等敏感信息。
- 新增配置项时，必须同步更新默认值、设置 UI、文档说明和兼容处理。

## MVP 范围基线

在没有额外确认前，默认按以下顺序推进：

1. 插件基础骨架：`manifest.json`、入口文件、构建链路、基础命令与设置页。
2. Codex 会话入口：侧边栏或面板中的基础聊天交互。
3. 上下文注入：当前笔记与选中文本。
4. 安全约束：Vault 边界、危险操作默认关闭或显式确认。
5. 扩展能力：内联编辑、命令模板、更多上下文源。

## 测试与验收要求

- 每次可交付改动至少通过一次 `build` 和 `typecheck`。
- MVP 不设覆盖率指标。
- 纯逻辑模块优先补单元测试。
- 与 Obsidian 生命周期强相关的功能，以手工验证为主。
- 未验证的能力不能在交付说明中表述为“已完成可用”。

## 文档约定

- `docs/adr/`：只记录真正影响实现方向的架构决策。
- `docs/plans/`：保留当前有效的设计稿和实现计划。
- 根目录下的 `task_plan.md`、`findings.md`、`progress.md` 是代理工作文件，不作为长期项目文档扩张。

## 当前已知假设

- 项目是个人或小团队驱动的插件开发。
- 首选语言为 TypeScript。
- 目标平台为桌面端 Obsidian。
- 参考项目是 Claudian，但本项目以 Codex 最佳实践为准。

如果这些假设后续被推翻，需要先更新本文件，再继续扩展实现。
