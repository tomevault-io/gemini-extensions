## Common commands

The project uses **mise** to drive **uv**. Tasks are defined in `mise.toml` — invoke them with `mise run <task>`.

```sh
mise run pytest             # full test suite (uv run pytest)
mise run pytest-cov         # tests + coverage XML/text reports
mise run ty                 # type-check via uvx ty (depends on uv-sync)
mise run ruff               # format (uv format)
mise run pymarkdown         # markdown lint/fix
mise run pyproject-fmt      # format pyproject.toml
mise run pre-commit         # ruff + ty + pymarkdown + pyproject-fmt
mise run ci                 # pre-commit + pytest-cov (matches CI)
mise run docs               # generate pdoc HTML under ./docs
mise run build              # uv build (sdist + wheel)
mise run update-shops       # regenerate saizeriya/shops.py via scripts/update_shops.py
```

Run a single test file or test:

```sh
uv run pytest tests/test_client.py
uv run pytest tests/test_page_parser.py::test_account_summary
uv run pytest -k "crawl"
```

Invoke the CLI during development:

```sh
uv run saizeriya help
uv run saizeriya start mysession "<qr_url>" --people 2
uv run saizeriya fetch-menu --max-code 200 --shop 0001
```

`uv sync --locked` (or `mise run uv-sync`) is required before `ty` to materialize the venv.

## Architecture

The package wraps Saizeriya's in-store mobile ordering pages (`ioes.saizeriya.co.jp`) — there is no public JSON API, so most "calls" are HTML/form round-trips. Layers separate concerns deliberately:

- **`saizeriya/process_qr.py`** — entry point for a real session. Follows the QR URL's redirect chain, parses the landing page, and returns a `ProcessedQR` carrying `base_url`, `shop_id`, `table_no`, the next form action `id`, and the detected `PageKind`.
- **`saizeriya/page_parser.py`** — `PageParser` wraps BeautifulSoup over a single page's HTML. It extracts page kind, form ids, hidden inputs (token, session), cart contents, account/receipt rows. Parsing is **page-shape-driven**: changes in Saizeriya's HTML structure are absorbed here.
- **`saizeriya/forms.py`** — builds the URL-encoded POST bodies. `create_base_fields`, `create_order_submit_body`, `now_order_time`, `encode_form` produce the exact field set the server expects (proc, token, time strings, repeated cart fields).
- **`saizeriya/client.py`** — `SaizeriyaClient` is the orchestrator. It holds an `httpx.Client` plus a single `ClientState` dataclass and threads them through every action (`lookup_item`, `add_item`, `go_to_cart`, `submit_order`, `get_account`, `get_receipt`, `call_staff`, `reorder`, `confirm_alcohol`, `check_*`). Each action: build form (forms) → POST → parse response (page_parser) → mutate state. Sessions can be reconstructed from a saved `ClientState` instead of a QR URL.
- **`saizeriya/types.py`** — single source of truth for the shared dataclasses (`ClientState`, `CartItem`, `AccountSummary`, `AccountLine`, `ReceiptSummary`) and the `PageKind` literal.
- **`saizeriya/shops.py`** — auto-generated `{store_name: shop_code}` dict consumed by `fetch_menu`. Regenerated by `scripts/update_shops.py`; do not edit by hand.
- **`saizeriya/fetch_menu.py`** — independent crawler. Iterates `shops × item_codes` against the `get_item.php` endpoint, persists incrementally to `data/menu-by-shop.json`, and is **resumable**: it reads the existing file and skips known item ids. It does *not* touch the ordering flow.
- **`saizeriya/cli.py`** — thin REPL on top of `SaizeriyaClient`. `start`/`use` open a session (saving cookies + `ClientState` to `~/.saizeriya-cli/sessions.json` after every command), `list`/`rm` manage saved sessions, `fetch-menu` proxies to the crawler. The REPL uses a custom `_ReplArgumentParser` that raises instead of calling `sys.exit`.

### Logging

All runtime output is routed through `logging.getLogger(__name__)`. `cli.py` calls `logging.basicConfig(level=INFO, format="%(message)s", stream=sys.stdout)` so user-facing REPL output stays clean; `fetch_menu.py` and `scripts/update_shops.py` use the same pattern. Avoid reintroducing `print()` in library or CLI code — write through the module logger instead.

### Tests

Tests under `tests/` are pure unit tests against parsed HTML fixtures and `httpx.MockTransport` — they do not hit the network. New tests should follow the same pattern (build a fixture HTML / mock transport, call the parser/client/crawler, assert on the returned dataclass or persisted JSON).

`tests/test_update_shops.py` puts the repo root on `sys.path` to import `scripts.update_shops`. When asserting on log output, use the `caplog` fixture scoped to the relevant logger name (e.g. `caplog.at_level(logging.WARNING, logger="scripts.update_shops")`).

## Repo conventions

- Lint config in `pyproject.toml` selects **all** Ruff rules; only `COM812`, `D203`, `D213` are globally ignored. `tests/test_*.py` ignores `D` and `S101`. Prefer fixing rather than `# noqa`.
- Docstring convention: **Google** (`lint.pydocstyle.convention = "google"`).
- Line length: 120.
- `mise run ty` is the type-checker (Astral's `ty`, not mypy/pyright) — keep code typed.
- Python ≥ 3.10 (`from __future__ import annotations` is used throughout).
- `data/menu-by-shop.json` is the canonical crawl output but is **not** committed; treat it as a local artifact.
