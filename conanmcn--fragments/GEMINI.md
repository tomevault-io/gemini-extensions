# Decisions

- Base UI headless primitives for component foundation (over: Radix UI, Headless UI, Ariakit) ŌĆö Unstyled primitives with full ARIA, custom styling via SCSS modules + tokens
- SCSS Modules + CSS custom properties (over: styled-components, Emotion, Tailwind, vanilla-extract) ŌĆö Dual fallback `var(--fui-token, $fui-token)` for compile-time and runtime theming. No JS runtime cost. SSR-safe
- 5-seed theming system deriving 120+ tokens (over: Per-token configuration, theme objects) ŌĆö 5 values (`$fui-brand`, `$fui-neutral`, `$fui-density`, `$fui-radius-style`, `$fui-danger`), system derives everything
- Fragment files as machine-readable component metadata (over: Storybook MDX, JSDoc) ŌĆö Colocated with code, JSON-serializable for AI, runtime-validated, compiled to fragments.json
- Monorepo with Turborepo + pnpm workspaces (over: Separate repos, Nx, Lerna) ŌĆö Shared types, atomic commits, single CI with caching
- Hybrid search with Reciprocal Rank Fusion (k=60) (over: Pure keyword, pure vector) ŌĆö RRF merges keyword + vector ranked lists robustly across different score ranges
- Two-tier CLI exports (cli vs cli/core) (over: Single package, separate packages) ŌĆö cli/core is browser-safe. Full cli needs Node.js. Docs app imports core without Node deps
- Object.assign compound component pattern (over: React Context per sub-component, render props) ŌĆö Zero-overhead composition. Card.Header discoverable via dot notation. TypeScript infers sub-component props
- MCP tool factory pattern (buildMcpTools) (over: Duplicate tool definitions per server) ŌĆö Single source of truth for 12 MCP tools. Both CLI and standalone MCP use same factory
- Optional peer deps for heavy libraries (over: Bundle everything, separate packages) ŌĆö DatePicker, Chart, CodeBlock etc use lazy require(). Core bundle stays small
- Dual license: UI is MIT, tooling is FSL-1.1-MIT (over: All MIT, all proprietary) ŌĆö UI MIT for adoption. Tooling FSL ŌĆö free for non-competing use, converts to MIT after 2 years

# Conventions

## Patterns
- **Component structure**: Each component is a folder with `index.tsx` + `.module.scss` + `.test.tsx` + `.fragment.tsx`
- **Compound components**: `Object.assign(Root, { Sub: ... })` pattern for `Card.Header`, `Dialog.Body`, etc. Used by 63/67 components
- **Barrel exports**: `libs/ui/src/index.ts` re-exports main + Root + all sub-components + all prop types per component
- **SCSS dual fallback**: `var(--fui-token, $fui-token)` mandatory in all component styles ŌĆö CSS vars for runtime theming, SCSS vars for build-time
- **Token prefix**: All tokens use `fui-` prefix (e.g., `--fui-color-accent`, `$fui-space-2`)
- **Seed-based theming**: 5 seeds derive 120+ CSS custom properties
- **Fragment files**: `defineFragment({ meta, usage, props, relations, contract, variants })` from `@fragments-sdk/cli/core`
- **Block files**: `defineBlock({ name, description, category, components, tags, code })` from `@fragments-sdk/cli/core`
- **Test pattern**: `vitest` + custom `render`/`screen`/`userEvent`/`expectNoA11yViolations` from `../../test/utils`
- **Role-based selectors**: Tests use `screen.getByRole()` and a11y assertions (`toHaveAccessibleName`, `toHaveAccessibleDescription`)
- **Optional peer deps**: Heavy libs (recharts, shiki, react-day-picker, @tanstack/react-table) are optional peer deps with lazy `require()`
- **Build strategy**: UI lib ŌåÆ Vite (preserveModules, CJS+ESM); tooling packages ŌåÆ tsup
- **CSS Modules convention**: `localsConvention: 'camelCase'` ŌĆö SCSS class `.my-class` ŌåÆ JS `styles.myClass`
- **MCP tool factory**: `buildMcpTools(brand)` creates all 12 tools; shared between CLI server and standalone MCP

## Anti-patterns
- Don't use hardcoded CSS values ŌĆö always use `var(--fui-token, $fui-token)` dual fallback
- Don't import from `@fragments-sdk/cli` in browser contexts ŌĆö use `@fragments-sdk/cli/core` (browser-safe)
- Don't create default exports for components ŌĆö always use named exports via barrel files
- Don't import individual sub-components directly ŌĆö import compound parent (e.g., `import { Card } from '@fragments-sdk/ui'`, then use `Card.Header`)
- Don't inline constant values ŌĆö legacy alias constants live in `cli/src/core/constants.ts` and `mcp/src/constants.ts` (intentionally duplicated)
- Don't skip `meta.status` in fragment files ŌĆö needed for deprecation warnings and search filtering

## AI failure modes
- **Compound component breakage**: AI refactors often break `Object.assign` compound pattern by splitting Root and sub-components into separate exports
- **CSS Module imports**: Models suggest `className="my-class"` strings instead of `className={styles.myClass}` module imports
- **Token guessing**: Models invent token names like `--fui-primary` instead of using actual tokens (`--fui-color-accent`)
- **SCSS fallback omission**: Models write `var(--fui-space-2)` without the SCSS fallback `var(--fui-space-2, $fui-space-2)`
- **Wrong defineFragment import**: Models import from `@fragments-sdk/cli` instead of `@fragments-sdk/cli/core`
- **React Fragment confusion**: "Fragment" in this codebase means component definition file, not React.Fragment

# Vocabulary
- **Fragment**: A component definition file (.fragment.tsx) with metadata, props schema, usage guidelines, variants, and AI contract ŌĆö NOT a React Fragment
- **Block**: A composition pattern (.block.ts) showing how multiple components wire together for a use case. Defined via `defineBlock()`
- **Seed**: One of 5 root theming values (brand, neutral, density, radius-style, danger) that derive 120+ CSS custom properties
- **Token**: A CSS custom property (`--fui-*`) or SCSS variable (`$fui-*`) representing a design decision. Always used with dual fallback pattern
- **Compound component**: A component with sub-components via `Object.assign` (`Card.Header`, `Dialog.Body`). Import parent, access children via dot notation
- **Capsule**: A compiled, portable snapshot of a design system in fragments.json format
