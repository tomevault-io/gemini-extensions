# Mercury — Agent Instructions

Personal AI assistant for chat platforms. Runs agents in Docker containers using [pi](https://github.com/badlogic/pi) as the runtime.

## Commands

```bash
bun run check        # Typecheck + lint + test (run before PR)
bun run check:fix    # Same but auto-fix lint issues
bun test             # Tests only
bun run typecheck    # TypeScript only
bun run lint         # Biome only
```

## ⚠️ Safety Rules

- **Never kill processes by port** (e.g. `lsof -ti:8787 | xargs kill`). This can kill the agent process itself if it has an open connection to that port. Use `mercury service uninstall` to stop Mercury cleanly.
- **Never run `mercury run` directly** — always use `mercury service install`. Direct runs block the terminal and don't auto-restart.

## Running a Mercury Project

To start a mercury project (e.g. after changing extensions or `.env`):

```bash
cd /path/to/mercury-project
mercury service install   # Installs and starts as a system service
```

That's it. The derived Docker image (with extension CLIs) is built automatically on startup if needed, and cached by content hash. You do **not** need to run `mercury build` — that command is only for developing the base mercury-agent image from source.

## Running in Background

The preferred way to run Mercury in the background is via system service (not tmux):

```bash
mercury service install   # Install as launchd (macOS) or systemd (Linux)
mercury service status    # Check if running
mercury service logs -f   # Tail logs
mercury service uninstall # Remove service
```

This provides auto-restart on crash and proper system integration. See [deployment.md](docs/deployment.md) for details.

## Releasing

Releases are tag-triggered via CI (`.github/workflows/release.yml`). The pipeline runs tests, publishes to npm, creates a GitHub release, and builds+pushes Docker images (amd64+arm64).

### Stable Release

```bash
npm version 0.5.0 --no-git-tag-version
git add package.json
git commit -m "chore: bump version to 0.5.0"
git push
git tag v0.5.0
git push origin v0.5.0
```

CI publishes `mercury-ai@latest` to npm.

### Experimental Release

For testing unreleased changes without affecting stable users:

```bash
npm version 0.5.1-exp.0 --no-git-tag-version
git add package.json
git commit -m "chore: bump version to 0.5.1-exp.0"
git push
git tag v0.5.1-exp.0
git push origin v0.5.1-exp.0
```

CI publishes `mercury-ai@exp` to npm. The npm tag is derived from the prerelease identifier (the part before the first `.` after `-`). Examples:
- `0.5.1-exp.0` → `@exp`
- `0.5.1-next.3` → `@next`
- `0.5.1-canary.1` → `@canary`

To use in a project: `mercury dockerfile --version exp`

## Project Structure

A Mercury project is a standalone directory with everything needed to run a bot:

```
my-bot/
├── .env                    # Bot config (identity, model, adapters) — NOT committed
├── .gitignore
├── Dockerfile              # Generated by `mercury dockerfile` — committed
├── state.db                # SQLite database — NOT committed
├── extensions/             # Mercury extensions (installed via `mercury ext add`)
├── workspaces/             # Per-workspace directories
│   ├── <name>/
│   │   ├── .env            # Workspace-specific env overrides — NOT committed
│   │   ├── AGENTS.md       # Per-workspace system prompt
│   │   ├── .pi/skills/     # Auto-populated from extensions
│   │   ├── inbox/
│   │   ├── outbox/
│   │   ├── knowledge/      # Knowledge vault (if knowledge extension installed)
│   │   ├── .messages/      # Daily message logs
│   │   └── sessions/       # Per-conversation session files
│   └── ...
├── pi-agent/               # Pi auth tokens — NOT committed
└── whatsapp-auth/          # WhatsApp credentials — NOT committed
```

### Dockerfile Workflow

The Dockerfile is generated, committed to the project repo, and built by CI:

```bash
mercury dockerfile                # Generate from current mercury version
mercury dockerfile --version exp  # Generate using experimental mercury
mercury build                     # Generate + build locally (dev convenience)
```

The generated Dockerfile installs mercury from npm (`bun add -g mercury-ai@<version>`) and includes all extension CLI installs. It is fully self-contained — no dependency on the mercury source tree.

CI publishes:
- **npm**: `mercury-ai` package (stable or tagged)
- **Docker**: `ghcr.io/<owner>/mercury-agent:latest` and `ghcr.io/<owner>/mercury-agent:<version>`

## Database Schema

Tables in `state.db` (all scoped tables include `workspace_id`):

- `workspaces` — Named workspaces (`id`, `name`, `created_at`, `updated_at`)
- `conversations` — Platform conversations with `workspace_id` assignment
- `messages` — Message history scoped by `workspace_id` + `conversation_id`
- `tasks` — Scheduled tasks (cron + one-shot) scoped by `workspace_id`
- `roles` — User role assignments, PK `(workspace_id, user_id)`
- `config` — Key-value config, PK `(workspace_id, key)`
- `mutes` — User mutes, PK `(workspace_id, user_id)`
- `session_boundaries` — Session compaction markers, PK `(workspace_id, conversation_id)`
- `extension_state` — Extension key-value store, PK `(workspace_id, extension, key)`
- `users` — User tracking (global, not workspace-scoped)
- `api_keys` — API key management (global)

## API

Internal API used by `mrctl` from inside containers. All workspace-scoped endpoints read the `X-Mercury-Workspace` header.

| Endpoint | Method | Description |
|----------|--------|-------------|
| `/health` | GET | Health check |
| `/api/whoami` | GET | Caller + role + workspace info |
| `/api/workspaces` | GET | List all workspaces |
| `/api/workspaces` | POST | Create a workspace |
| `/api/workspaces/:name` | DELETE | Delete a workspace |
| `/api/workspaces/:name/conversations` | GET | List conversations in workspace |
| `/api/workspaces/:name/pairing-code` | GET | Get workspace pairing code |
| `/api/tasks` | GET/POST | List/create tasks (workspace-scoped) |
| `/api/tasks/:id` | DELETE | Delete task |
| `/api/tasks/:id/pause` | POST | Pause task |
| `/api/tasks/:id/resume` | POST | Resume task |
| `/api/tasks/:id/run` | POST | Trigger task immediately |
| `/api/roles` | GET/POST/DELETE | Role management (workspace-scoped) |
| `/api/permissions` | GET/PUT | Permission management |
| `/api/config` | GET/PUT | Config (workspace-scoped) |
| `/api/conversations` | GET | List all conversations |
| `/api/conversations/pairing-code` | GET | Get pairing codes for all workspaces |
| `/api/conversations/:id/unpair` | POST | Unassign conversation from workspace |
| `/api/conversations/:id/workspace` | PUT | Assign conversation to workspace |
| `/api/conversations/:id/workspace` | DELETE | Unassign conversation from workspace |
| `/api/mutes` | GET/POST | List/create mutes (workspace-scoped) |
| `/api/mutes/:userId` | DELETE | Unmute a user |
| `/api/stop` | POST | Abort current run |

| `/api/ext` | GET | List installed extensions |
| `/api/ext/:name/auth` | POST | Permission check for extension CLI |

Auth: `X-Mercury-Caller` + `X-Mercury-Workspace` headers. API key via `Authorization: Bearer <key>`.

### Chat API (direct bridge)

| Endpoint | Method | Description |
|----------|--------|-------------|
| `/chat` | POST | Send a message and get a synchronous reply |

No auth required. Request body: `{ text, callerId?, workspace?, authorName?, files?: [{ name, data(base64) }] }`. Returns `{ reply, files: [{ filename, mimeType, sizeBytes, data(base64) }] }`.

The `workspace` field is required when sending files. Input files are saved to the workspace's `inbox/`. Output files are read from `outbox/` and returned as base64.

CLI wrapper:
```bash
mercury chat "hello"
mercury chat --file photo.jpg "what's in this?"
echo "msg" | mercury chat
```

## Extension System

Mercury has a TypeScript extension system. Extensions live in `extensions/*/` and export a setup function:

```typescript
import type { MercuryExtensionAPI } from "../extensions/types.js";

export default function(mercury: MercuryExtensionAPI) {
  mercury.cli({ name: "napkin", install: "bun add -g napkin-ai" });
  mercury.permission({ defaultRoles: ["admin", "member"] });
  mercury.env({ from: "MERCURY_NAPKIN_API_KEY" });
  mercury.skill("./skill");
  mercury.on("workspace_init", async ({ workspace }, ctx) => { ... });
  mercury.job("distill", { interval: 3600_000, run: async (ctx) => { ... } });
  mercury.config("enabled", { description: "...", default: "true" });
  mercury.widget({ label: "Status", render: (ctx) => "<p>OK</p>" });
  mercury.store.get("key");
}
```

Key types are in `src/extensions/types.ts`. See [docs/extensions.md](docs/extensions.md) for the full design.

### Built-in vs extension commands

`mrctl` has two types of commands:
- **Built-in**: `tasks`, `roles`, `permissions`, `config`, `conversations`, `mutes`, `stop`, `compact` — HTTP calls to host API
- **Extension**: Called directly in bash (e.g., `napkin search "query"`) — RBAC enforced by pi extension at bash level

Built-in names are reserved — extensions cannot collide with them.

### Permissions

Permissions are dynamic. Built-in permissions are static; extensions register new ones at runtime via `registerPermission()`. Admin always gets all permissions. See `src/services/roles/service.ts`.
