# Swarf Development

Go CLI tool. Build with `just build`, test with `just check`.

## Committing

1. **Always commit your work before finishing.** Do not leave uncommitted changes.
2. If you find uncommitted changes from a previous session, commit them first to keep a clean working directory.
3. Commit after every logical change — don't batch unrelated work into one commit.
4. Use conventional commits: `feat:`, `fix:`, `docs:`, `chore:`, `refactor:`, `test:`
5. Commit straight to main.
6. Write clear messages: lead with *what changed and why*, not a list of files.
7. Keep the first line under 72 characters; add a body for non-trivial changes.
8. If the commit is rejected by a pre-commit hook, fix the issue and retry — do not bypass the hook.
9. Do not add AI attribution trailers (Co-Authored-By, Generated by, etc.)

## Quality Gate

- `just check` — runs `go vet` and `go test ./...`
- All tests must pass before committing.

## Notes

Private notes, research, and design docs go in `swarf/`, not this repo.
`swarf/` is gitignored — do not `git add` anything from it.
