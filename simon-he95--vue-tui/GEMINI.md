# TUI Package Guidelines

This file supplements the repository-level guide in `../../AGENTS.md` and applies to work inside `packages/tui/`.

## Package Overview

- Package name: `@simon_he/vue-tui`
- Purpose: Vue 3 terminal UI component library and renderer/runtime toolkit
- Build tool: `tsdown`
- Test stack: Vitest for unit tests, Playwright for e2e
- Docs stack: VitePress with generated component API docs

## Directory Layout

- `src/core/` - terminal primitives, buffer, ANSI/theme support, terminal creation
- `src/renderer/` - DOM and CLI/stdout renderers
- `src/events/` - event manager, coordinate mapping, recording helpers
- `src/runtime/` - runtime wiring
- `src/cli/` - input parsing, mouse/kitty parsing, headless and terminal helpers
- `src/vue/` - Vue-facing components, composables, render manager, router helpers
- `test/` - Vitest coverage
- `docs/` - VitePress docs and generated API pages
- `examples/` - runnable sample apps for browser and terminal builds
- `scripts/` - docs/build/benchmark helpers

## Common Commands

```bash
# Build and dev
bun run build
bun run dev
bun run build:examples
bun run build:examples:terminal

# Docs
bun run docs:gen
bun run docs:dev
bun run docs:build
bun run docs:preview

# Quality
bun run typecheck
bun run lint
bun run lint:fix
bun run format
bun run format:check

# Tests
bun run test
bun run e2e

# Demo runners
bun run run:basic:terminal
bun run run:basic:multi-select:terminal
```

From the repository root you can also use:

```bash
bun run build:tui
bun run test:colors
```

## Editing Guidance

- Prefer editing source under `src/`; treat `dist/`, `dist-terminal/`, `docs/generated/`, and `test-results/` as generated outputs unless the task explicitly targets them.
- Keep terminal semantics, renderer behavior, and Vue abstractions separated; changes should usually stay within one layer unless the feature truly crosses boundaries.
- Reuse existing components and composables before creating new primitives.
- For documentation-sensitive component changes, update docs or generated API content expectations as needed.

## Code Style

- Follow the repository-level TypeScript and import rules.
- Keep APIs composable and renderer-agnostic where possible.
- Match existing naming and file layout patterns such as `stdout-renderer.ts`, `TerminalProvider.ts`, and `use-layout.ts`.
- Preserve Vue 3-style composable/component ergonomics; most package code is TypeScript modules rather than `.vue` SFCs.

## Testing Expectations

- Unit tests run with `vitest`.
- E2E tests run with `playwright`.
- For renderer or terminal capability changes, run the narrowest relevant test first, then broaden if needed.
- For example or app-level behavior changes, consider one of the terminal demo runners when it helps verify UX behavior.
- Keep tests deterministic; avoid timing-sensitive assertions unless the package already uses that pattern.

## Docs and Examples

- Component API docs are generated by `scripts/generate-component-api-docs.ts`.
- If you change public component props/events or documented behavior, update the docs flow with `bun run docs:gen`.
- Examples under `examples/basic` are useful for validating interaction and renderer behavior; prefer updating them when the public API changes.

## Validation

- Narrow validation first: `bun run test`, a targeted Vitest file, or a specific demo command.
- For public API or docs changes, run `bun run docs:gen` and, if needed, `bun run docs:build`.
- For larger package changes, combine `bun run typecheck`, `bun run lint`, and the most relevant test command before finishing.
