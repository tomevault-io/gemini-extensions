## Overview

This repository manages a homelab Kubernetes cluster running on Talos Linux, with GitOps-based deployment using Flux CD. The cluster configuration is split into two main directories:

- `talos-config/`: Talos Linux OS configuration and cluster bootstrap
- `flux-config/`: Kubernetes resources managed by Flux CD

## Environment Variables

The `.envrc` file defines critical environment variables (loaded via direnv):

- `CLUSTER_NAME`: homelab-cluster
- `CONTROL_PLANE_IP`: 10.0.1.6 (current control plane node)
- `KUBERNETES_ENDPOINT`: kube.internal (cluster API endpoint)
- `KUBECONFIG`: Path to kubeconfig file
- `TALOSCONFIG`: Path to talosconfig file

Always source `.envrc` or ensure these variables are set before running Talos/kubectl commands.

## Secrets Management

Secrets are encrypted using SOPS with age encryption:

- Configuration: `.sops.yaml` defines encryption rules for `*.enc.(yml|yaml)` files
- Age key: Referenced as `&bitwarden` in SOPS config
- Encrypted secrets: `talos-config/secrets.enc.yaml`

**Decrypt secrets:**
```sh
sops decrypt secrets.enc.yaml --output .decrypted.secrets.yaml
```

**Encrypt secrets:**
```sh
sops encrypt -i secrets.enc.yaml
```

**Important:** Never commit decrypted files (`.decrypted*` pattern is gitignored).

## Talos Linux Configuration

### Architecture

Talos configuration is patch-based and reproducible:

- `patches/common.yaml`: Shared configuration applied to all nodes
  - Disk encryption (LUKS2 for ephemeral and state partitions)
  - Longhorn volume mounts with `rshared` propagation
  - Kernel modules (nvme_tcp, vfio_pci)
  - Hugepages configuration
  - API server certificate SANs
- `patches/controlplane-1.yaml`: Node-specific overrides
- `schematics/bare-metal.yaml`: Talos system extensions manifest
  - iscsi-tools (required for Longhorn)
  - util-linux-tools

### Common Commands

**Generate Talos secrets (first time setup):**
```sh
talosctl gen secrets -o secrets.enc.yaml && sops encrypt -i secrets.enc.yaml
```

**Generate control plane configuration:**
```sh
sops decrypt secrets.enc.yaml --output .decrypted.secrets.yaml
talosctl gen config \
  --with-secrets .decrypted.secrets.yaml \
  --config-patch @patches/common.yaml \
  --config-patch @patches/controlplane-1.yaml \
  --output-types controlplane \
  --output generated-controlplane-1.yaml \
  $CLUSTER_NAME https://$KUBERNETES_ENDPOINT:6443
```

**Apply configuration to node:**
```sh
talosctl --nodes $CONTROL_PLANE_IP apply-config --file generated-controlplane-1.yaml --insecure
```

**Bootstrap cluster (first time only):**
```sh
talosctl --nodes $CONTROL_PLANE_IP bootstrap
```

**Generate talosconfig:**
```sh
talosctl gen config \
  --with-secrets .decrypted.secrets.yaml \
  --output-types talosconfig $CLUSTER_NAME https://$KUBERNETES_ENDPOINT:6443
talosctl config endpoint $CONTROL_PLANE_IP
```

**Generate kubeconfig:**
```sh
talosctl --nodes $CONTROL_PLANE_IP kubeconfig kubeconfig
```

**Upgrade Talos OS:**
```sh
talosctl upgrade --nodes $CONTROL_PLANE_IP \
  --image factory.talos.dev/installer/613e1592b2da41ae5e265e8789429f22e121aab91cb4deb6bc3c0b6262961245:v1.11.3
```

**Upgrade Kubernetes:**
```sh
talosctl --nodes $CONTROL_PLANE_IP upgrade-k8s --to 1.34.0 --dry-run
talosctl --nodes $CONTROL_PLANE_IP upgrade-k8s --to 1.34.0
```

**Query node status:**
```sh
talosctl --nodes $CONTROL_PLANE_IP get <resource>
talosctl --nodes $CONTROL_PLANE_IP version
```

### Talos Factory Schematics

To regenerate Talos installation image with system extensions:

```sh
curl -X POST --data-binary @schematics/bare-metal.yaml https://factory.talos.dev/schematics
```

## Flux CD Configuration

### Architecture

Flux follows a hierarchical structure in `flux-config/clusters/homelab/`:

```
flux-config/clusters/homelab/
├── flux-system/           # Flux bootstrap (DO NOT EDIT - generated by flux)
│   ├── gotk-sync.yaml     # GitRepository and root Kustomization
│   └── gotk-components.yaml
├── infrastructure/        # Core infrastructure components
│   ├── longhorn/          # Persistent storage (Longhorn)
│   ├── metallb-helm/      # Load balancer (MetalLB controller)
│   └── metallb-crds/      # MetalLB CRDs (IPAddressPool, L2Advertisement)
└── apps/                  # Application workloads (currently empty)
```

**Key principles:**

- Flux syncs from `main` branch at `flux-config/clusters/homelab/` path
- Infrastructure resources are deployed before apps
- MetalLB is split into two Kustomizations (CRDs first, then Helm chart) to handle dependency ordering
- HelmReleases use `crds: CreateReplace` to ensure CRDs are managed by Helm

### GitOps Workflow

Changes to this repository automatically reconcile to the cluster:

1. Commit changes to `flux-config/` directory
2. Push to `main` branch
3. Flux pulls changes within 1 minute (GitRepository interval)
4. Kustomizations reconcile within 10 minutes (or immediately if triggered)

**Force reconciliation:**
```sh
flux reconcile source git flux-system
flux reconcile kustomization flux-system
```

### Infrastructure Components

**Longhorn (v1.10.1):**
- Distributed block storage for persistent volumes
- Namespace: `longhorn-system`
- Requires: iscsi-tools system extension, Longhorn mount in Talos config

**MetalLB (v0.15.2):**
- Bare metal load balancer
- Namespace: `metallb-system`
- IP Pool: 10.0.1.11-10.0.1.15 (L2 mode)
- Configured via IPAddressPool and L2Advertisement CRDs

### Common Flux Commands

**Check Flux status:**
```sh
flux check
flux get all
```

**View specific resources:**
```sh
flux get sources git
flux get helmreleases -A
flux get kustomizations
```

**Debug reconciliation issues:**
```sh
flux logs
flux get helmrelease <name> -n <namespace>
kubectl describe helmrelease <name> -n <namespace>
```

**Suspend/resume reconciliation:**
```sh
flux suspend kustomization <name>
flux resume kustomization <name>
```

## Cluster Configuration Notes

- **Control plane scheduling:** Enabled via `allowSchedulingOnControlPlanes: true` (single-node cluster)
- **Virtual IP:** 10.0.1.10 (API server high availability endpoint)
- **Disk encryption:** Full system disk encryption using LUKS2 with node-specific keys
- **Storage:** Longhorn requires `rshared` mount propagation for `/var/lib/longhorn`
- **Networking:** MetalLB provides LoadBalancer services on local network (10.0.1.11-15)

## Adding New Applications

To deploy a new application via Flux:

1. Create a subdirectory in `flux-config/clusters/homelab/apps/<app-name>/`
2. Add Kubernetes manifests, HelmRelease, or Kustomization
3. Update `flux-config/clusters/homelab/apps/kustomization.yaml` to include the new directory
4. Commit and push to main branch

For Helm-based apps, follow the pattern in `infrastructure/longhorn/` or `infrastructure/metallb-helm/`.

## Working with Generated Files

Generated files are gitignored to prevent accidental commits:

- `talos-config/generated-*.yaml`: Talos machine configs
- `.decrypted*`: Decrypted secrets
- `talosconfig`: Talos client config
- `kubeconfig`: Kubernetes client config

Always regenerate these files when updating Talos configuration or secrets.
