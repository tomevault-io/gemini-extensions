# AGENTS.md

## 0. 目的与适用范围
本文件约束 Agent 在本仓库的执行方式，目标是：架构合理、实现简单、结构清晰、交付可验证。

指令优先级：
1. 更深层目录的 `AGENTS.md`
2. 本文件
3. 项目其他文档与默认规范

## 1. 核心工程原则
1. 先架构后编码：先明确模块边界、职责划分、依赖方向，再开始写代码。
2. 简单优先：能用简单方案解决的问题，不引入复杂实现。
3. 清晰一致：避免重复实现、冲突逻辑和混乱结构。
4. 生态复用优先：优先使用官方 SDK、主流库、项目现有能力，减少新增维护面。

## 2. 角色与决策边界
1. Agent 负责完整交付闭环：需求理解、方案设计、实现、测试、审查、文档更新。
2. 默认自主决策并直接执行，仅在以下情况需要用户确认：
   - 破坏性数据变更（如 DROP/不可逆 schema 变更）
   - 删除核心配置文件或关键基础设施文件
   - `git push` 到远程分支
   - 连续 3 次同类失败后需要切换策略
3. 不在上述清单中的事项，默认自动执行。

## 3. 执行流程（核心版）
### 阶段A：上下文与架构
1. 快速扫描相关代码、配置、测试与相似实现。
2. 识别核心约束：接口契约、数据流、边界条件、风险点。
3. 形成最小可行架构方案：先保证正确性与一致性，再考虑扩展。

### 阶段B：实现
1. 小步修改，保持每次变更可运行、可验证。
2. 优先复用现有模块与模式，新增抽象必须有明确收益。
3. 发现缺陷优先修复，再扩展功能。

### 阶段C：验证
1. 执行与改动匹配的本地验证（单测、集成、冒烟中最相关的子集）。
2. 无法执行的验证必须记录原因、影响范围与风险。
3. 连续 3 次失败必须暂停，回到架构与假设重新评估。

### 阶段D：交付
1. 给出方案说明、改动摘要、验证结果、剩余风险。
2. 明确哪些外围问题被有意忽略及原因。

## 4. 自动评审（核心优先，外围降噪）
1. 每次改动默认执行自动评审。
2. 评审先做影响分层：
   - 核心层：主流程正确性、数据一致性、接口契约、安全权限、并发与事务、关键性能路径、异常恢复。
   - 外围层：文案、样式微调、非关键日志、注释措辞、演示代码。
3. 仅核心问题阻断交付：
   - P0：错误结果、数据损坏、安全漏洞、系统不可用。
   - P1：架构冲突、重复实现导致行为分叉、关键边界条件错误、显著性能退化。
4. 外围问题默认不阻断，除非已影响核心正确性、安全性或可维护性。
5. 评审输出固定为：
   - Core Findings（仅 P0/P1）
   - Ignored Peripheral（已忽略外围项与理由）
   - Risks and Missing Tests（残余风险与测试缺口）

## 5. Tools / Skills / MCP 使用策略
1. 本地证据优先：先看代码与测试，再做结论。
2. 内部检索优先级：`code-index`（可用时） > `rg`/`grep` > 其他方式。
3. 深度思考策略：复杂或高风险任务优先使用 `sequential-thinking`；简单任务可直接执行。
4. 外部文档策略：涉及第三方库 API、版本差异、最佳实践时，优先使用 `context7` 获取最新文档。
5. 网络检索策略：`exa` 可用时优先；不可用则降级到其他可用搜索工具，并说明降级原因。
6. 写入策略：优先使用可审计的补丁式改动（如 `apply_patch`），保持 diff 清晰。

## 6. 编码与重构约束
1. 单一职责：函数或类应聚焦单一目的。
2. 三次重复再抽象：避免过早抽象，也避免复制粘贴扩散。
3. 可读性优先于技巧性：实现应直观、可维护。
4. 遵循项目既有风格、格式化和测试组织方式。
5. 及时清理失效代码与冗余分支，避免新旧逻辑并存冲突。

## 7. 测试与记录
1. 验证以本地自动执行为主，不依赖人工外包验证。
2. 记录最小必要证据：执行命令、结果摘要、失败原因、风险判断。
3. 对关键改动，至少覆盖：正常路径、关键边界、主要错误恢复路径。

## 8. 输出规范
1. 先说明架构与实现策略，再给具体改动。
2. 说明为何该方案比替代方案更简单、更一致。
3. 明确列出核心问题结论与剩余风险。
4. 回答应简洁、可执行、可追溯，避免空泛描述。

## 9. Strict 已启用条目（当前项目）
1. `S03` 新增抽象前必须给出收益说明，至少满足以下之一：减少重复、降低耦合、提升可测性。
2. `S07` 重复逻辑出现第 3 次必须抽象，不得继续复制粘贴扩散。
3. `S08` 默认禁止聪明写法（过深链式、隐式副作用、难以直读的技巧性实现）。
4. `S09` 禁止同义逻辑新旧并存；若新实现已生效，旧路径应清理或下线。
5. `S10` 复杂或高风险任务必须先做 `sequential-thinking`，再进入实现。
6. `S11` 涉及第三方库/框架 API、版本差异、最佳实践时，必须先查 `context7`。
7. `S16` 核心路径改动必须覆盖三类验证：正常路径、关键边界、主要错误恢复。
8. `S17` 连续 3 次同类失败必须暂停，回到方案层重评，不得盲目重试。
9. `S19` 交付前必须输出 `Core Findings`（仅 P0/P1）。
10. `S20` 一旦发现 P0/P1，必须先修复再继续功能扩展。
11. `S23` 破坏性变更（不可逆 schema 变更、删除核心配置、远程 push）必须用户确认。
12. `S24` 未经确认禁止执行高风险不可逆操作（如大范围删除、重写历史）。

## 10. Skill 优先策略（新增）
1. 默认优先使用 Skill 进行任务执行与协作编排。
2. 当任务可映射到已定义 Skill 时，优先按映射选择，不自行发明新流程。
3. 若映射 Skill 不可用或失败，允许降级执行，但需在交付中说明降级原因。

### 10.1 任务到 Skill 的优先映射（强约束）
1. 产品评审、规划、路演、运营：`gstack`
2. 工程实施：`superpowers`
3. 前端开发：`frontend-dev`
4. 后端开发集成：`fullstack-dev`
5. Android 开发：`android-native-dev`
6. iOS 开发：`ios-application-dev`
7. PDF 编辑：`minimax-pdf`
8. 生成 PPT：`pptx-generator`
9. Excel 表格分析：`minimax-xlsx`

### 10.2 冲突处理
1. 若同一任务同时命中多个 Skill：先按产品/工程主线分类，再选对应主 Skill。
2. 产品类任务优先 `gstack`；工程类任务优先 `superpowers`。
3. 需要跨域协同时，可组合使用，但必须保持主 Skill 唯一且明确。
