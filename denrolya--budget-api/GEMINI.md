# CLAUDE.md — Codebase Standards & Rules

This document defines the **mandatory** coding standards for this Symfony 6.4 / API Platform 3 financial backend. Every rule here must be followed in all new and modified code. No exceptions without explicit justification in a comment.

---

## Stack

- PHP 8.2+, Symfony 6.4 LTS, API Platform 3, Doctrine ORM 2.x
- PHPStan level **max** (`phpstan.dist.neon`)
- PHPUnit 9.6 with DAMA doctrine-test-bundle transaction isolation

---

## Naming

### Variables and Parameters

- **No abbreviations, ever.** Use full English words.
  - `$transaction` not `$tx`, `$txn`
  - `$account` not `$acc`
  - `$exchangeRate` not `$rate` (when context is ambiguous)
  - `$repository` not `$repo`
  - `$entityManager` not `$em`
  - `$queryBuilder` not `$qb`
  - `$identifier` not `$id` (except Doctrine `$id` field on entities)
  - `$result` not `$res`
  - `$index` not `$i` (except tightly scoped `for` loops of ≤5 lines)
  - `$temporary` not `$tmp`
  - `$configuration` not `$config`
  - `$parameter` not `$param`
  - `$request` not `$req`, `$response` not `$res`
- Boolean variables and properties must read as a predicate: `$isActive`, `$hasCategory`, `$wasProcessed`.
- Collection variables must be plural: `$transactions`, `$accounts`.
- Avoid generic names like `$data`, `$info`, `$item`, `$object`, `$value` unless the type itself is genuinely generic. Prefer `$transactionData`, `$accountInfo`, etc.

### Methods and Functions

- Methods must be named as **verb + noun**: `calculateTotalAmount()`, `resolveExchangeRate()`, `buildCategoryTree()`.
- No abbreviations in method names: `getTransactions()` not `getTx()`.
- Boolean-returning methods must start with `is`, `has`, `can`, `was`, `should`, or `needs`.
- Private helpers must clearly describe what they do, not how: `groupTransactionsByPeriod()` not `doGrouping()`.

### Classes and Interfaces

- Classes are nouns or noun phrases: `TransactionRepository`, `ExchangeRateSnapshotResolver`.
- Interfaces are adjective or capability nouns ending with `Interface`: `OwnableInterface`, `ValuableInterface`.
- Abstract classes may use `Abstract` prefix only when the abstraction is meaningful: `AbstractBankProvider`.
- Traits describe the behavior they add: `TimestampableTrait`, `SoftDeletableTrait`.
- DTOs end with `Dto` (not `DTO`): `BankTransactionDto`.
- No `Manager` suffix unless the class truly orchestrates multiple services. Prefer `Service`, `Calculator`, `Builder`, `Resolver`, `Exporter`.

## Complexity

### Cyclomatic and Cognitive Complexity

- **Maximum cyclomatic complexity per method: 10.**
- **Maximum cognitive complexity per method: 15.**
- These are hard ceilings, not targets. Aim lower. Refactor when a method starts accumulating branches.
- Prefer early returns (guard clauses) over nested `if/else` blocks.
- Extract any block that requires a comment to explain it into a named private method.

```php
// WRONG — nested complexity
public function process(Transaction $transaction): void
{
    if ($transaction->hasCategory()) {
        if ($transaction->getCategory()->isExpense()) {
            if ($transaction->getAmount() > 0) {
                // do something
            }
        }
    }
}

// CORRECT — guard clauses
public function process(Transaction $transaction): void
{
    if (!$transaction->hasCategory()) {
        return;
    }
    if (!$transaction->getCategory()->isExpense()) {
        return;
    }
    if ($transaction->getAmount() <= 0) {
        return;
    }

    $this->applyExpenseCategoryLogic($transaction);
}
```

### Method Length

- Methods must not exceed **50 lines** of executable code (excluding blank lines, braces, comments).
- Constructor injection counts as 0 toward the method limit.
- If a method approaches the limit, extract cohesive chunks into private methods.

### Class Length

- Entity classes: ≤ 250 lines (excluding getters/setters generated by IDE).
- Service/Repository classes: ≤ 300 lines.
- If a class exceeds these limits, consider splitting responsibilities.

## SOLID Principles

### Single Responsibility

- Each class has exactly one reason to change.
- Services must not contain query-building logic — delegate to repositories.
- Repositories must not contain business logic — delegate to services.
- Controllers and API Platform actions must not contain business logic — delegate to services.
- Event listeners handle exactly one cross-cutting concern.

### Open/Closed

- Extend behavior via new classes (strategy pattern, decorators) rather than modifying existing ones.
- Use interfaces for dependency injection so implementations can be swapped.

### Liskov Substitution

- Subtypes must be usable wherever their parent type is expected.
- Do not override a parent method to throw an exception or return a different type.
- Entity inheritance (e.g., `Expense extends Transaction`) must honor every contract defined on the parent.

### Interface Segregation

- Keep interfaces focused. A class should not be forced to implement methods it does not use.
- Prefer multiple small interfaces (`OwnableInterface`, `ValuableInterface`) over one `EntityInterface` with everything.

### Dependency Inversion

- Depend on interfaces, not concrete implementations — especially for bank providers, exchange rate providers, and external services.
- Use Symfony's DI container; never instantiate services with `new` inside other services.

---

## DRY — Don't Repeat Yourself

- Any logic duplicated in two places must be extracted. Three occurrences is unacceptable.
- Common entity behaviors go in traits (`TimestampableTrait`, `OwnableEntityTrait`, etc.).
- Common query filters go in repository base methods or a query builder helper.
- Common validation constraints go in reusable constraint classes.
- Test setup shared across multiple test classes goes in a base class or trait.

---

## KISS — Keep It Simple

- Always implement the simplest solution that correctly solves the problem.
- Do not add configuration options, feature flags, or generic abstractions for hypothetical future needs.
- Do not use design patterns for their own sake. Introduce a pattern only when a concrete second use case exists.
- Prefer a straightforward `if` over a strategy pattern until you have at least 3 strategies.

---

## PHP Type System

- All method signatures must have complete type declarations: parameter types, return types, property types.
- Use `?Type` only when `null` is a meaningful, intentional value — not as a shortcut for "not yet set."
- Prefer union types (`int|string`) over `mixed`.
- Never use `mixed` unless forced by a third-party interface.
- PHPStan level max must pass with zero errors. `@phpstan-ignore` is forbidden unless the false positive is filed as a known PHPStan bug, with a comment linking the issue.
- Avoid `/** @var Type */` inline casts — use proper assertions (`assert()`, `instanceof` guard) or refactor to eliminate the uncertainty.

## Doctrine ORM

### Entity Design

- Use PHP 8 attributes for all Doctrine mappings. No YAML, no XML, no annotations.
- All entity relationships must have explicit `cascade`, `fetch`, and `orphanRemoval` configured deliberately — not left at defaults.
- **No lazy-loading proxies in loops.** If you iterate over a collection and access a relation, eagerly load it in the query.
- Collections on entities are `ArrayCollection` and are initialized in the constructor.
- Entity constructors may contain business rules. Avoid anemic entities — put simple domain logic (invariant guards, computed values) on the entity itself.
- Bidirectional relationships must document which side is the owning side.

### Repositories

- Every repository method that returns a list must have typed array returns documented via `@return Type[]` and verified by PHPStan.
- Use `QueryBuilder` for dynamic queries. Use DQL for static complex queries. Raw SQL is a last resort (see below).
- Compose query builders: define a base builder (e.g., `getBaseQueryBuilder()`) and add criteria on top.
- Never call `findAll()` on an entity that can have more than a few hundred rows — always add criteria or a limit.
- Repositories must not call `EntityManager::flush()`. Flushing is the responsibility of the calling service or controller.
- Avoid `EntityManager::getRepository()` inside services — inject the specific repository.

### Batch Operations

- When persisting large batches (e.g., bulk transaction creation, bank sync):
  1. Call `EntityManager::persist()` inside the loop.
  2. Call `EntityManager::flush()` **once** after the loop, never inside the loop.
  3. If the batch size exceeds ~500 items, flush and call `EntityManager::clear()` in chunks (configurable batch size constant, not a magic number).
- Disable Doctrine SQL logging in batch command contexts via `$connection->getConfiguration()->setSQLLogger(null)`.
- Use `INSERT ... ON DUPLICATE KEY UPDATE` (raw SQL or DBAL) only when upsert semantics are genuinely needed and cannot be expressed via ORM without multiple round-trips.
- For read-heavy aggregations over large transaction sets, use a single `QueryBuilder` with `SUM`, `COUNT`, `GROUP BY` rather than loading entities into PHP and summing in a loop.

### Raw SQL Policy

- Raw SQL (via `DBAL->executeQuery()` or native queries) is **only permitted** when:
  1. A DQL/ORM equivalent would require multiple queries or hydration of thousands of entities that are immediately discarded.
  2. A database-specific feature (e.g., `ON DUPLICATE KEY UPDATE`, `JSON_ARRAYAGG`, window functions) is required.
- Raw SQL must be accompanied by a comment explaining **why** DQL/ORM is insufficient.
- Raw SQL must use bound parameters — never string interpolation or concatenation.
- PHPStan `literalString: true` is enforced; all query strings must be literal strings or concatenations of literal strings — never from user input.

### Avoiding N+1 Queries

- Any collection fetch that will be iterated must use `JOIN FETCH` or a second query with `IN` (batch fetch).
- Use Doctrine's second-level cache only if profiling confirms a hot spot — do not add it speculatively.
- After adding a new endpoint or query, verify the query count with the Symfony profiler or tests.

## Symfony Best Practices

- Use constructor injection for all dependencies. No setter injection, no property injection (`#[Inject]`).
- Service IDs are class FQCNs. No string aliases unless required by a bundle.
- `services.yaml` uses `autowire: true` and `autoconfigure: true` for the default namespace. Explicit service definitions are for special cases only.
- Configuration values injected via `#[Autowire(param: '...')]` or `%parameter%` must have a corresponding `config/packages/*.yaml` definition.
- Use Symfony Messenger for async operations if a job does not need to block the HTTP response.
- Events and subscribers: use `EventSubscriberInterface` with `getSubscribedEvents()`. No `#[AsEventListener]` tag on the subscriber itself when it already implements the interface.
- Validation constraints are defined on DTOs and Entities via PHP attributes. No manual validation in services.
- Use `#[MapRequestPayload]` or custom `ValueResolver` for controller input instead of reading from `Request` directly inside the controller body.

## API Platform

- Define API resources via PHP attributes on entity classes or dedicated API resource classes.
- Use `#[ApiResource]` with explicit `operations` array — do not rely on default operation inference.
- Custom actions use `#[ApiResource]` `uriTemplate` + a dedicated `Provider`/`Processor` class or a single-purpose action class in `src/ApiPlatform/Action/`.
- DTOs for request bodies are separate classes in `src/DTO/` with validation constraints.
- State processors (`DataPersister` / `StateProcessor`) handle persistence. They must not contain query logic.
- State providers handle reads. They must not persist anything.
- Filters are defined via attributes on the resource class. Custom filters live in `src/ApiPlatform/` and implement `FilterInterface`.
- Security: `security` expressions on operations use `is_granted()`. Do not perform authorization checks inside services — keep them at the API layer.
- Serialization groups follow the convention: `{resource}:read` for collection, `{resource}:item:read` for single item, `{resource}:write` for input.

## Testing

- Every new service method must have a corresponding unit or integration test.
- Every new API endpoint must have a corresponding API test in `tests/ApiPlatform/` or `tests/Feature/`.
- Use `BaseApiTestCase` for all API-level tests.
- Do not mock the `EntityManager` or repositories in integration tests — use the real database with DAMA isolation.
- Mock only external HTTP clients (bank APIs, exchange rate APIs) and time-dependent services.
- Test method names follow the pattern: `test{MethodOrAction}_{scenario}_{expectedOutcome}`.
  - Example: `testCreateTransaction_withInvalidAmount_returns422`
- Test fixtures must use `FakerPHP` for realistic data. No magic numbers like `12345.67` unless testing a specific boundary.
- Assert on behavior, not on implementation: assert response codes, response bodies, database state — not on which private methods were called.
- Resource usage listener (`ResourceUsageListener`) is registered; tests that allocate unexpected memory should be investigated.

## Code Style

- PSR-12 code style is required. Run `php-cs-fixer fix` before committing.
- Strict types declaration at the top of every PHP file: `declare(strict_types=1);`
- No `else` after a `return`, `throw`, or `continue`.
- No empty `catch` blocks. At minimum, log the exception.
- Constants in `SCREAMING_SNAKE_CASE`. Class constants preferred over global constants.
- Magic numbers must be named constants: `private const BATCH_FLUSH_SIZE = 500;` not `if ($index % 500 === 0)`.
- String keys used in more than one place must be constants.
- `use` statements are grouped: PHP built-ins, then external vendor, then internal `App\`, alphabetically within each group.

## Comments and Documentation

- Do not comment on *what* the code does — it must be readable enough to be self-explanatory.
- Comment only *why* a non-obvious decision was made.
- PHPDoc blocks are only required on public methods of interfaces and when the return type or parameter type cannot be fully expressed in PHP type syntax (e.g., `@return Transaction[]`).
- Do not leave `TODO` comments in committed code. Open a GitHub issue and reference it: `// TODO: GH-123 — replace with DQL after Doctrine 3 upgrade`.

---

## Git and Commits

- Each commit addresses exactly one concern (feature, fix, refactor).
- Commit messages follow Conventional Commits: `feat(transaction): add bulk import endpoint`.
- Do not commit commented-out code.
- Do not commit `var_dump`, `dump`, `dd`, or `print_r` calls.
- PHPStan and tests must pass before merging to `master`.

## Prohibited Patterns

| Pattern | Reason |
|---|---|
| `$em->flush()` inside a loop | Causes N round-trips to the database; flush once after the loop |
| `findAll()` without criteria on large tables | Full table scan; always constrain |
| Nested `if` deeper than 2 levels | High cognitive complexity; use guard clauses |
| `array_map` / `array_filter` chained 3+ times | Unreadable; extract into a named method |
| `new SomeService(...)` inside a service | Breaks DI and testability |
| Raw SQL without a comment justifying it | SQL must always explain why ORM was insufficient |
| Abbreviations in any identifier | Harms readability; use full words |
| `mixed` return or parameter types | Defeats static analysis; use proper types |
| `@phpstan-ignore` without a linked issue | Masks real bugs; fix the code instead |
| Business logic in a Controller or Action class | Controllers only route; delegate to services |
| Query building in a Service class | Services call repositories; repositories build queries |
