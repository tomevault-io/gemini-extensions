## Commands

```bash
# Activate the virtual environment first
source .venv/bin/activate

# Run the dev server
python manage.py runserver

# Run all tests
pytest

# Run a single test
pytest tasks/tests.py::test_task_create_view

# Create migrations after model changes
python manage.py makemigrations tasks
python manage.py migrate

# Django system check
python manage.py check
```

## Architecture

This is a single-app Django project with server-rendered templates and no JavaScript framework.

**Request flow:** `task_tracker_project/urls.py` → `tasks/urls.py` → class-based views in `tasks/views.py` → templates in `templates/tasks/`

**Key design decisions:**
- All views use Django's generic CBVs (`ListView`, `CreateView`, `UpdateView`, `DeleteView`). The only custom view is `TaskToggleView` (a plain `View`) which handles the complete/reopen POST action.
- Templates live in the top-level `templates/` directory (not inside the app), configured via `TEMPLATES[0]["DIRS"]` in settings.
- Static files (`static/css/style.css`) are served from the top-level `static/` directory via `STATICFILES_DIRS`.
- `TaskForm` in `tasks/forms.py` is used by both create and update views — the template detects which mode it's in via `{% if object %}`.

**Test setup:** `pytest-django` reads `DJANGO_SETTINGS_MODULE` from `pytest.ini`. Tests use the `db` fixture for database access and Django's built-in test `client` for view tests. No mocking — all tests hit SQLite in-memory.

**SonarQube:** Coverage XML (`coverage.xml`) must be generated by the sonarqube workflow (`pytest --cov=tasks --cov-report=xml`) before the scan step runs. `SONAR_TOKEN` is a repository secret; `SONAR_HOST_URL` is a repository variable.
