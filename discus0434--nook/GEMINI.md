あなたはこのリポジトリ内で "AGENT" と呼称される。AGENT はこのファイル内のルールとワークフローを遵守し、ユーザー要望に応じて適切に行動する。

## ルール

AGENT は以下の基本ルール、Python / CDK ルール、TDD / テスト実行フロー、docs / specs の運用ルールに従う。

### 基本ルール

- 思考や内部作業は英語で行い、ユーザーとの会話、Issue / PR コメント、レビュー指摘は日本語で行う。
- PR の作成、コメント確認、返信などの GitHub 操作には `gh` コマンドを使う。
- 後方互換は不要なので実装しない。
- 無駄なフォールバックを記述しない。潔くエラーをraiseする。
- エラーをraiseするときはtracebackを必ずログに出力する。
- アプリケーションの状態の監視・分析を容易にするため、詳細な構造化ログを出力する。
- 瑣末なものはロギングしない。重要なことだけログに出力する。
- 「その場限りの修正/更新」は絶対に行わず、コードベースを俯瞰した上で根本的にreasonableな変更を加える。
- 複雑なロジックは分解する。関数は小さく、目的が明確な、名前の良いものを書く。
- タイムアウトによる無駄な処理中断とリトライを防ぐため、shell ツールの `block_until_ms` は 18000000 程度のかなり長時間のタイムアウトを設定して実行するべきである。
- 100 行を超える変更、複数モジュール横断の変更、挙動変更を伴う変更では、実装前に `specs/` 配下へ技術仕様書を作る。spec は作業中の一時ファイルであり、ライフサイクル完了後に削除する。

### Python / CDK ルール

- ルートの依存管理と実行は `uv` に統一する。`uv sync --dev` で環境を構築し、コマンド実行は `uv run ...` を使う。
- ルートでは `requirements.txt` / `requirements-dev.txt` を使わない。依存定義は `pyproject.toml` と `uv.lock` を正とする。
- Lambda ランタイム依存は引き続き各関数ディレクトリの `requirements.txt` を使う。これはデプロイ用であり、ルート開発環境とは役割が異なる。
- Pythonの依存関係管理と実行に uv コマンドを利用する。`uv sync` で環境を構築し、スクリプト実行は `uv run xxx.py ...` のように行う。
- コードの可読性維持のため、Python の import 文は原則ファイル冒頭にまとめ、関数内での遅延 import は行わない。
- Pythonのバージョンは3.32を想定し、型アノテーションをそれに合わせる。
  - 例えば、`typing.Dict[Key, Value]` ではなく `dict[Key, Value]` を使用する。
  - `from __future__ import annotations` は不要なので記述しない。
- データモデルの定義に Pydantic を利用する。
- `cdk.json` は `uv run python app.py` で CDK アプリを起動する前提とする。
- CDK で必要な一時ファイルや build asset は `make cdk-*` で作成 / 削除し、手動で編集しない。

### TDD / テストのルール

- 非テストコードの変更では、先に失敗する自動テストを書くことを基本とする。
- 自動テストは `tests/unit/` に置き、`pytest` で常時実行可能なものだけを含める。
- ネットワーク、外部 API、認証情報、手動確認を要するチェックは自動テストに混ぜない。必要なら手動スクリプトや `make manual-*` を使う。
- 条件分岐が増える場合は、正常系と異常系 / 境界系の両面をカバーする。
- 「落ちないこと」ではなく、戻り値、状態変化、副作用、例外メッセージを具体的に検証する。

### テスト実行フロー

- Python / HTML テンプレート / 設定ファイルを編集したら、原則 `make post-change` を実行する。
- 対象を絞る場合は `make test ARGS='-k <pattern>'` を使う。
- 外部 API の動作確認が必要な場合は `make manual-feed`, `make manual-rss`, `make manual-tech-news-feeds`, `make manual-product-hunt` を使う。
- `cdk` の差分確認や synth が必要な場合は `make cdk-diff`, `make cdk-synth` を使う。

## ドキュメント

- `docs/` に現況を反映したドキュメントを記述する。
- `docs/` 下のドキュメントは確定した仕様のみを記述し、これから実装予定の内容は `specs/` に技術仕様書として記述する。
- 仕様変更・追加・修正があったら必ず更新し、常に最新状態を保つ。
- ドキュメントは追記で履歴を溜めず、既存の内容を上書きして現況に揃える。
- 変更履歴は Git history / PR / Issue で追跡し、履歴専用ドキュメントは作成・更新しない。
- `docs/DESIGN.md` に現行のアーキテクチャ / 設計を記述する。
- その他、必要な現況ドキュメントがあれば `docs/` 下に追加する。

## 中規模以上の仕様追加・変更のライフサイクル

1. 要件が曖昧ならユーザー確認を行い、実装に入る前に合意を取る。
2. `specs/TEMPLATE.md` を雛形として技術仕様書を作成する。
3. 必要なら active spec を `specs/` に 1 件だけ置いたうえで `make spec-review` を実行し、曖昧さや欠落を潰す。
4. TDD でテストを先に書き、最小実装を行う。
5. `make post-change` で format / lint / pytest を通す。
6. active spec を `specs/` に 1 件だけ置いたうえで `make spec-implementation-review` を実行し、技術仕様書との差分を確認する。
7. 4. で作成されたテストをリファクタリングする。実装の詳細に密結合している脆いテストを削除し、振る舞いに焦点を当てた高品質なものへと改良する。
7. ユーザー承認後、`docs/` を現況へ更新する。
8. 今回のライフサイクルで作成した spec を `specs/` から削除する。spec は保存物ではなく、実装完了までの作業ファイルとして扱う。
