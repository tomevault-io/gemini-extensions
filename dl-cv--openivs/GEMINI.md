# OpenIVS 项目规则

## 项目基础

- 项目使用 `.NET Framework 4.7.2`
- WinForms 工程：界面定义在 `Form1.Designer.cs`，代码逻辑在 `Form1.cs`
- WPF 工程优先使用 WPF 风格（绑定显示等）
- 代码保持规范、清晰、便于开源维护

## 文档规范

- 开始任务前先读取 `开发文档.md`
- 更新文档时，以 `开发文档.md` 的规则为准，不在子文档重复

## 构建规则

- **唯一入口**：必须使用 `.cursor/skills/vs-build/scripts/build.py` 执行编译/构建/重建
- **默认配置**：`OpenIVS.sln`、`Debug`、`x64`、`Build`、`minimal`
- 禁止直接使用 `msbuild`、`dotnet`、`powershell` 等命令编译
- 构建脚本不可用或报错时，立即停止并报告

## 核心执行原则

- **不包火**：遇到报错时，必须先按用户指令定位根因，禁止用 fallback、绕过、静默忽略等方式掩盖真正的问题。
- **按指令执行**：用户明确要求检查某处代码（如 `dlcv_infer`）时，必须优先检查指定代码，确认其正确定义/导出/实现后，再排查调用侧或上下游问题；禁止跳过指定检查直接在外层做兼容性兜底。

## Git 工作流

- **PR → 合并 → 拉取 → 检查 → 删分支**：
  1. 推送功能分支到远程；
  2. 创建 PR；
  3. 合并 PR（远程分支由平台自动删除）；
  4. **必须先拉取最新代码到本地**；
  5. **通过 `git log` 检查本地确实已包含合并提交**；
  6. **确认后才删除本地分支**。
- 禁止在尚未拉取并验证本地已合并前，提前删除本地分支。

## 工程一致性原则

- **核心缺口必须在核心层补齐**：当核心逻辑（如加载器、解析器、资源绑定器）与输入数据格式不匹配时，必须在核心逻辑内部完成适配或提供默认行为，严禁在外围通过修改输入数据、写临时补丁文件、运行时打二进制补丁等方式绕过。外围修补会破坏数据完整性、引入隐藏依赖，且无法在日志和异常中体现真实问题。
- **修改核心逻辑必须同步所有入口**：调整模型加载、推理、资源绑定等核心行为时，必须同步检查并更新所有相关入口（如不同文件格式、不同协议、不同调用路径），确保行为一致。禁止仅在单一入口修复而在其他入口通过外围补丁兜底。
