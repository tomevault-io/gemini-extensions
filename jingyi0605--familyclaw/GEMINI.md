# 项目全局规则

## 沟通规则

1. 始终使用中文和用户沟通。
2. 表达直接、清楚、少废话，不要故作高深。
3. 如果一句话能说清，就不要堆三句术语。

## 文档更新规则

1. `docs/Documentation/` 下的 Markdown、`docs/Documentation/.vitepress/` 下的配置与主题文件都属于正式文档源码，必须纳入 Git；只有构建产物和缓存不追踪。
2. 只要代码改动影响安装启动、配置、用户可见行为、开发流程、接口语义或运维排障，对应正式文档必须在同一变更里同步更新。
3. 如果实现已经变化而文档没改完，这个任务不算完成；不要把文档更新留到“后面再补”。
4. 只要新增、删除、重命名官方文档文章，必须同步检查 `docs/Documentation/.vitepress/theme/i18n.ts` 的中英文文章映射。
5. 如果该文章有对应翻译，必须在 `TRANSLATION_PAIRS` 中补齐映射；如果暂时没有对应翻译，也必须确认分区兜底跳转仍然正确，不能让语言切换落到 404。

## Spec 文档规则

后续只要在 `specs/` 目录里新建或修改 Spec，必须遵守下面这些规则。

### 1. 文档必须写成人话

- 文档首先是给人看的，其次才是给工具追踪的。
- 不要写“能力路由抽象层闭环”“服务治理底座”“多维能力编排”这种空话，除非后面立刻用一句人话解释它到底是什么。
- 每一段都应该让一个刚接手项目的人看懂“这是要干什么”。

### 2. 任务文档必须优先回答这几个问题

每个任务至少要让人一眼看明白：

1. 这一步到底做什么
2. 做完以后能看到什么结果
3. 这一步依赖什么
4. 主要改哪些文件
5. 这一步明确不做什么
6. 怎么验证是不是真的做完了

如果这些问题答不清，这个任务就不合格。

### 3. 允许保留追踪信息，但不能拿追踪信息当正文

- 可以写“对应需求”“对应设计”
- 但不能整段都在堆 `requirements.md`、`design.md`、章节号、术语名
- 追踪信息是辅助，不是主体

### 4. 避免黑话和假大空表述

禁止优先使用这类写法：

- 闭环
- 抽象层
- 赋能
- 治理
- 编排
- 底座
- 能力沉淀

如果确实要用，必须马上补一句人话解释，不然就换成普通表达。

### 5. 推荐写法

优先使用下面这种表达：

- “这一步把什么东西建出来”
- “做完以后页面/接口/数据会发生什么变化”
- “先不做什么，避免范围失控”
- “别人接手时第一眼该看什么文件”

### 6. 模板也必须说人话

- `specs/000-Spec规范/Spec模板/` 下的模板必须保持人话风格
- 后续创建新 Spec 时，默认沿用人话模板
- 如果模板变回黑话，优先修模板，不要把垃圾继续复制到新 Spec

## 数据库迁移规则

1. 只有 Alembic 可以修改数据库表结构。
2. 严禁在应用启动、业务代码、脚本里使用 `Base.metadata.create_all()`、`drop_all()`、自动补表、自动补列。
3. 新增表、字段、索引、约束时，必须编写 Alembic migration，不允许只改 model。
4. 如果数据库表已经存在但版本号没跟上，先核对结构，再用 `alembic stamp` 校准；不要靠重复建表碰运气。
5. AI IDE 修改后端数据库相关代码时，必须优先检查 `migrations/` 和 `alembic_version` 的一致性，不能自行绕过迁移系统建表。
6. 进行任何数据库字段操作前，必须先阅读：apps\api-server\migrations\20260311-数据库迁移规范.md。

## 后端开发规范

1. 只要涉及后端开发，尤其是新增或修改 `async` 入口、周期任务、常驻 worker、后台轮询、插件执行链路，必须先阅读：
   - `docs/开发者文档/后端/20260316-后端事件循环与周期任务开发规范.md`
2. 不要在 `AGENTS.md` 里复制后端规范原文。后端规则统一维护在项目文档里，避免多处漂移。
3. 如果后端实现和该文档冲突，以文档里的执行边界、Session 线程边界、worker 运行规范和回归要求为准。

## 插件启停规则

1. 只要涉及插件启用/禁用状态、插件新增入口、插件执行入口、插件自动触发链路，必须先阅读：
   - `docs/开发设计规范/20260317-插件启用禁用统一规则.md`
2. 不要在 `AGENTS.md` 里复制插件启停规则原文。插件状态规则统一维护在项目文档里，避免不同模块各写各的。
3. 如果具体实现和该文档冲突，以文档里的统一状态源、执行边界、错误语义和 UI 反馈规则为准。

## 前端代码编写原则
1. 不要编辑编译产物，js文件不要添加到git
2. 新建前端页面的时候，页面的文案不要采用工程师视角的语言，按照最终交付给普通用户的标准编写文案
3. 必须使用统一的i18n字典，不允许使用硬编码的显示文字
4. 新增和修改前端代码，必须修改user-app，不允许再改已经下线的历史前端目录
添加任何新页面、标签页组件之前，先阅读：docs\开发设计规范\前端页面设计语言规范.md

## 仓库代码提交规则
1. 如果你认为当前修改的代码已经满足git提交的最小模块，应该主动提出提交代码
2. 提交代码的时候根据功能、模块、文档分配提交，不允许混合提交
3. 提交信息的格式要求：（类型：spec编号-内容；），修复类或者临时修改，没有spec编号的，不体现编号，不要写“无编号”
