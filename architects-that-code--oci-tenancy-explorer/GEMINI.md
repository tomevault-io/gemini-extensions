# AGENTS.md

## Purpose

This file defines how agents must operate in this repository.

Primary goals:

- keep `README.md` safe, accurate, and structurally aligned with `main`
- avoid accidental content loss in docs
- communicate in a direct, technical style

## Non-Negotiable README Rules

1. `README.md` section parity with `main` is mandatory.
- The working branch README must keep the same top-level `##` section set and order as `origin/main:README.md`.
- Do not invent, remove, or reorder top-level sections unless the user explicitly asks to change README structure.

2. Never silently lose README content.
- If edits would remove meaningful existing content, stop and ask permission first.
- If the user has edited README text, preserve that text whenever possible.
- If simplification is requested, summarize exactly what will be removed and get explicit approval before deleting non-trivial content.

3. Preserve disclaimer text.
- Keep the `## Disclaimer (Sample Code)` section present.
- Do not weaken or paraphrase legal disclaimer text unless user explicitly requests it.

## Required README Validation Before Finalizing

For any README change, run and verify:

1. Section parity check against `origin/main`:
- Compare top-level `##` headings between working tree and `origin/main:README.md`.

2. Loss check:
- Review README diff and explicitly confirm whether any content was deleted.
- If deletion occurred, confirm user permission is recorded in thread.

3. Report verification commands used.

Suggested commands:

```bash
git show origin/main:README.md | rg '^## '
rg '^## ' README.md
git diff -- README.md
```

## Communication Style

Use direct, technical communication.

- No fluff, no hype, no vague language.
- State assumptions and constraints explicitly.
- Prefer concrete file/command references.

## Response Format Requirements

When reporting work, use bullet points and include:

- what changed (files and intent)
- what was verified (tests/checks/commands)
- any remaining risks or open items

If nothing was changed, state that clearly with a short reason.

## Change Safety Rules

- Do not revert user changes unless explicitly requested.
- If unexpected local changes appear, stop and ask how to proceed.
- Keep commits focused and atomic.
- Prefer small, reviewable documentation diffs.

## Repo Notes

- The app is served by `portal_server.py`.
- Snapshot data is generated by collector scripts and should be treated as sensitive tenancy metadata.
- Use `oci_config.example` as the safe reference for local OCI config structure.
