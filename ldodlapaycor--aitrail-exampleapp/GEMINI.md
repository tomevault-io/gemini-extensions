## Project Organization

- **Feature-based (vertical slice) folder structure** — group controller, service, models, and tests for a feature together. Never use a flat `/Controllers` / `/Services` / `/Models` layout.
- Place new features under `src/<FeatureName>/` alongside their own unit tests.
- Separate business logic from infrastructure (DB, file system, HTTP clients) using interfaces.

---

## Clean Code Principles

- Each class has **one reason to change** (SRP). Each method does **one thing** (≤ 20 lines).
- No duplicate logic — extract to a shared method or class when code appears more than twice.
- Max **3–4 levels of nesting**; use guard clauses / early returns to stay flat.
- Depend on **abstractions**, not concretions (DIP). Use dependency injection throughout.
- Avoid god classes, long parameter lists (>3–4 params), and primitive obsession.
- Names must reveal intent; comments explain *why*, not *what*.

---

## Coding Style

- Follow existing patterns and formatting in the file being edited.
- Use C# record types for immutable DTOs/request/response models.
- Prefer expression-bodied members for single-line implementations.
- Use `var` when the type is obvious from the right-hand side.
- One class per file; file name matches class name.
- Use string interpolation for simple concatenation; `StringBuilder` for loop concatenation.

---

## Security Rules

- Every controller and action must carry `[Authorize]`. Anonymous access requires explicit `[AllowAnonymous]` with a documented justification.
- Add specific privilege checks (e.g. `[RequirePrivilege("Resource.Action")]`) beyond bare authentication.
- **Never return PII in list/collection endpoints.** Single-resource endpoints may return PII only when `?includeSensitiveData=true` is passed, and the access must be audit-logged.
- Never expose stack traces, exception messages, or internal details in 5xx responses.
- Validate and sanitize all inputs at controller boundaries using Data Annotations (`[Required]`, `[StringLength]`, `[RegularExpression]`, etc.).
- Never accept sensitive filter data (e.g. SSN) as query string parameters — use a POST body instead.

---

## Data Access Layer Rules

- Return **business objects**, never raw database entities, from API responses.
- Use async repository methods exclusively (`await _repo.GetAsync(...)`) — no blocking calls.
- Always paginate collection queries; enforce `pageSize` ≤ 100.
- Avoid N+1 queries — use `.Include()` / projection queries or batched calls.
- Use `HttpClientFactory` for all outbound HTTP calls (connection pooling).
- Implement the circuit-breaker pattern for calls to external services.

---

## Language Features

- **Async/await everywhere** — never call `.Result` or `.Wait()` on a `Task`. Propagate `async` up the full call stack.
- **`DateTimeOffset` not `DateTime`** — store all timestamps in UTC; convert to local time only at the presentation layer.
- Use `StringBuilder` for any string concatenation inside a loop or involving more than 3–4 operations.
- Verify framework/NuGet versions in the `.csproj` before using APIs — do not assume the latest version is installed.
- Use `using` declarations (C# 8+) for `IDisposable` resources to guarantee cleanup.
- Prefer `IEnumerable<T>` / `IReadOnlyList<T>` in method signatures over concrete collection types.

---

## Exception Handling Rules

- Use exceptions for **exceptional** conditions only — not for control flow.
- Catch specific exception types; never swallow exceptions with an empty `catch` block.
- Always log exceptions with a **correlation ID** before handling or rethrowing.
- Return the standard error response shape for all 4xx responses:
  ```json
  { "correlationId": "<GUID>", "status": "<HTTP status>", "title": "...", "detail": "...", "source": { "<field>": "<message>" } }
  ```
- Let unhandled 5xx exceptions bubble to a global exception-handling middleware — do not catch-all in individual controllers.
- Clean up resources in `finally` blocks or with `using` statements.

---

## Performance Best Practices

- All I/O (DB, HTTP, file) must be async; target < 200 ms for simple queries, < 2 s for complex operations.
- Use `StringBuilder` in loops (see Language Features).
- Batch bulk operations in chunks of ≤ 100 items; cap bulk endpoints at 1 000 items per request.
- Call `await Task.Yield()` inside long-running loops to yield the thread scheduler.
- Set `[RequestSizeLimit(52428800)]` (50 MB) on endpoints that accept large payloads.
- Use `ETag` / `If-None-Match` for conditional GET responses where appropriate; never cache API results at intermediate devices.
