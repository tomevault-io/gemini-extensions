# nutpieR - R Bindings for the nutpie NUTS Sampler

## What This Is

An R package providing native bindings to the [nuts-rs](https://github.com/pymc-devs/nuts-rs) NUTS sampler (written in Rust), using [BridgeStan](https://github.com/roualdes/bridgestan) as the Stan model backend. The R equivalent of [nutpie](https://github.com/pymc-devs/nutpie) (Python).

**R users can sample Stan models with nuts-rs without touching Python or reticulate. R is not in the hot loop — parallel chains run entirely in Rust.** Benchmarked at ~4x faster than CmdStanR on a 1040-parameter hierarchical model.

## Current State

The package is **functional** — compile Stan models, sample with parallel chains, get `posterior::draws_array` output with diagnostics. All core MVP features are implemented.

### What works
- `nutpie_compile_model(stan_file)` or `nutpie_compile_model(code = "...")` — compiles Stan models via BridgeStan
- `nutpie_sample(model, data, ...)` — parallel NUTS sampling with progress bars
- Full sampling parameters: `num_draws`, `num_warmup`, `num_chains`, `cores`, `seed`, `max_treedepth`, `target_accept`, `init_mean`, `refresh`, `verbose`
- Returns `posterior::draws_array` — works with all posterior/bayesplot tooling
- Diagnostics: `nutpie_diagnostics()` returns divergences, treedepth, energy, step size, etc.
- Warmup access: `nutpie_warmup_draws()` and `nutpie_warmup_diagnostics()` when `save_warmup = TRUE`
- Optional `store_divergences` and `store_mass_matrix` for detailed debugging
- Low-rank modified mass matrix adaptation (`low_rank_modified_mass_matrix = TRUE`)
- Cross-platform: builds on Windows, macOS, and Linux without LLVM/libclang

## Architecture

```
R user code
    │
    ▼
R layer (thin functional API)
    │  - nutpie_compile_model() → calls Rust → bridgestan::compile_model
    │  - nutpie_sample() → calls Rust via extendr
    │  - returns posterior::draws_array (via Arrow transfer)
    │
    ▼
Rust layer (extendr crate)
    │  - compiles Stan models via bridgestan Rust crate
    │  - implements nuts-rs Model trait + CpuLogpFunc trait
    │  - CpuLogpFunc::logp calls bridgestan::Model::log_density_gradient
    │  - Sampler::new() runs parallel chains via rayon
    │  - returns draws via Arrow (zero-copy to R via nanoarrow)
    │
    ├── nuts-rs 0.17.x (NUTS sampler, "arrow" feature)
    ├── bridgestan 2.7.x (patched — pre-generated bindings, no bindgen)
    └── arrow-extendr (Arrow C Data Interface for R ↔ Rust)
```

**R is NOT in the hot loop.** Each chain gets its own `bridgestan::Model` instance (Send+Sync via `Arc<StanLibrary>`). Parallel chains run entirely in Rust via rayon.

## System Requirements

- **Rust toolchain**: `rustc >= 1.85`, `cargo` (edition 2021)
- **C++ toolchain**: needed by `bridgestan::compile_model()` to compile Stan models
  - Windows: rtools45
  - macOS: Xcode Command Line Tools
  - Linux: build-essential (gcc, make)
- **No LLVM/libclang needed** — bridgestan bindings are pre-generated

## BridgeStan Patch

We carry a patched bridgestan 2.7.0 in `src/rust/patches/bridgestan-2.7.0/`, referenced via `[patch.crates-io]` in Cargo.toml. Two changes from upstream:

1. **Pre-generated bindings** (`src/pregenerated_bindings.rs`): bindgen output for 22 `bs_*` FFI symbols, generated once. The `build.rs` copies this file instead of running bindgen. Eliminates LLVM/libclang requirement on all platforms.

2. **Root cause**: upstream bridgestan runs bindgen with `dynamic_link_require_all(true)` and no allowlist. On Windows, system headers inject CRT symbols that `from_library()` then tries to resolve from the Stan model DLL — causing `GetProcAddress` error 127. The allowlist filters (`bs_.*`) fix this. Worth upstreaming.

**Regenerating bindings**: If bridgestan updates its C API, temporarily restore `bindgen` in the patch's `Cargo.toml`, build, copy `$OUT_DIR/bindings.rs` → `src/pregenerated_bindings.rs`.

## Windows Environment Notes

### Rscript newline handling
Multi-line `Rscript -e` commands cause segfaults. Use semicolons or multiple `-e` flags.

### Spaces in paths
The `Makevars.win.in` redirects `TARGET_DIR` and `CARGO_HOME` to `$(TEMP)/nutpieR-*` (no spaces) to avoid dlltool/assembler issues. `nutpie_compile_model()` copies stan files to temp dirs if paths contain spaces (make can't handle them).

### extendr `Result` type conflict
`extendr_api::prelude::*` imports `Result<T>` (1 generic param), shadowing `std::result::Result<T, E>`. Use fully qualified `std::result::Result<T, E>` in nuts-rs trait impls.

### TBB DLL
Stan models with `STAN_THREADS=true` need `tbb.dll` on PATH. `add_tbb_to_path()` in `model.rs` handles this by scanning `~/.bridgestan/` for TBB directories.

## Key APIs

### nuts-rs `CpuLogpFunc` trait
```rust
pub trait CpuLogpFunc: HasDims {
    type LogpError: Debug + Send + Sync + Error + LogpError + 'static;
    type ExpandedVector: Storable<Self>;
    fn dim(&self) -> usize;
    fn logp(&mut self, position: &[f64], gradient: &mut [f64]) -> Result<f64, Self::LogpError>;
    fn expand_vector<R: Rng>(&mut self, rng: &mut R, array: &[f64]) -> Result<Self::ExpandedVector, CpuMathError>;
}
```

### nuts-rs `Model` trait
```rust
pub trait Model: Send + Sync + 'static {
    type Math<'model>: Math where Self: 'model;
    fn math<R: Rng>(&self, rng: &mut R) -> Result<Self::Math<'_>>;
    fn init_position<R: Rng>(&self, rng: &mut R, position: &mut [f64]) -> Result<()>;
}
```

### nuts-rs `Sampler`
```rust
match sampler.wait_timeout(duration) {
    SamplerWaitResult::Trace(traces) => { /* Vec<ArrowTrace> */ },
    SamplerWaitResult::Timeout(sampler) => { /* get sampler back */ },
    SamplerWaitResult::Err(e, partial) => { /* anyhow::Error + Option<traces> */ },
}
```

## Design Decisions

- **extendr** for R-Rust bridge (not raw `.Call`)
- **No BridgeStan R package** — Rust handles compilation and loading
- **Arrow** for data transfer (nuts-rs `arrow` feature + `arrow-extendr` + `nanoarrow`)
- **DiagGradNutsSettings** (diagonal mass matrix NUTS) + low-rank modified mass matrix option
- **`posterior::draws_array`** as output format
- **Functional R API** (not R6 — matches Stan ecosystem conventions)
- **Not targeting CRAN** — no vendoring or MSRV constraints. R-universe for binary distribution.
