# AI/代码代理在该仓库的工作指导

**目标**：打造让绝对新手也能轻松完成服务器管理的面板，同时在默认路径下坚持不妥协的高安全标准。

## 1) 核心执行规则

- 优先进行小而完整的修改，而不是宽泛的推测性重构。
- 将所有编辑严格限定在用户请求范围内。
- 不要重写无关文件。
- 除非用户明确要求，否则绝不运行破坏性命令（`git reset --hard`、批量删除、强制重写等）。
- 将现有的本地修改视为用户所有，除非有明确证据否则。
- 任何输入控件都**不要**使用 `form-control` 作为 class。

## 2) 高效提示处理

当收到提示时：

1. 用一句话重述任务。
2. 快速识别受影响的文件（使用 `rg`、`rg --files`）。
3. 制定最小化计划（1-3 步）并立即执行。
4. 直接实现；除非用户明确要求，否则避免只做长篇设计响应。
5. 使用最小有意义命令进行验证（先运行当前 package 的 `go test`，必要时再扩大范围）。

操作默认设置：
- 使用 `rg` 进行搜索。
- 使用聚焦读取（`sed -n`、指定文件），不要全仓库 dump。
- 优先使用补丁风格（patch-style）编辑，保持 diff 最小。

## 3) 安全优先的代码生成

假设所有输入均为不可信，除非经过验证。

### 认证与授权 (AuthN/AuthZ)

- 在服务器端处理程序中强制执行权限检查。
- 绝不依赖 UI 可见性来控制访问。
- 在生产模式下，确保特权/开发专用控制不仅在前端隐藏，还必须在服务器端完全禁用。

### 输入验证

- 对请求参数进行验证和规范化。
- 尽早拒绝格式错误或空的标识符。
- 对枚举类值（角色、操作、状态）使用白名单。

### 数据与密钥

- 绝不硬编码密钥、令牌或凭证。
- 使用环境变量或配置加载器管理密钥。
- 不要记录密钥值、会话令牌、加密密钥或 PII。

### 数据库安全

- 仅使用参数化查询。
- 禁止通过请求值拼接 SQL 字符串。
- 应用最小权限数据访问模式。

### Web 安全

- 保留安全的 Cookie 设置（`HttpOnly`、`Secure`、`SameSite`）。
- 保持 CSRF 和会话保护完整。
- 在模板中对不可信输出进行转义/编码。
- 避免通过内联 JS/HTML 插值引入 XSS 向量。

### 文件/外部 I/O

- 验证文件类型、大小和路径使用。
- 通过规范化并限制文件路径防止路径遍历。
- 将外部 Webhook/API 负载视为不可信。

## 4) 特定质量标准

- 保持函数内聚且可预测，避免隐藏的副作用。
- 返回带有可操作上下文的明确错误。
- 在可行时，优先使用类型化的常量/枚举而不是魔术字符串。
- 在正常请求路径中避免 panic。

完成前必须检查：
- `make build` 必须成功并输出 `bin/server`。
- 对修改区域运行相关测试。
- 如果完整测试套件失败，报告具体失败的测试，并说明失败是否在修改前已存在。

## 5) 变更审查清单

在最终响应前确认：
- 请求已完全实现。
- 安全态势未退化。
- 没有仅通过前端隐藏暴露的特权路径。
- Diff 最小且易于理解。
- 清晰报告验证步骤和结果。

## 6) 调试要求 / Debugging

处理 bug、异常行为、线上报错、页面 `not found`、权限异常、路由异常、登录异常或“为什么不工作”类请求时，必须遵守以下顺序：

1. **先复现，再修改**
   - 必须先复现用户描述的“精确问题”。
   - 如果问题与 URL、路由、请求方法、权限、页面跳转或参数有关，优先检查原始请求路径、请求方法、解析结果、路由匹配顺序和服务器端处理链路。
   - 不要因为“看起来像前端问题”或“看起来像后端问题”就直接下结论。

2. **先定位故障层，再决定修复点**
   - 在动手改代码前，必须明确说明问题出在哪一层：
     - 浏览器/UI 层
     - 前端路由/状态层
     - AJAX / 请求构造层
     - 服务器路由匹配层
     - 处理程序 / 业务逻辑层
     - 数据/配置/外部依赖层
   - 如果没有定位到具体失败层，禁止做猜测性修复。

3. **禁止投机式修复**
   - 不要因为某个解释“听起来合理”就先补丁。
   - 不要只修表象，必须修根因。
   - 如果同时发现次要问题，先单独标明“次要加固”或“附带改进”，不得把它伪装成主修复。

4. **必须添加回归测试**
   - 每次修复 bug，都应尽量新增或更新一个能覆盖“本次精确故障”的测试。
   - 测试要证明：
     - 之前为什么会失败
     - 现在为什么不会再失败
   - 如果确实无法自动化测试，必须在最终响应中明确说明原因以及手工验证步骤。

5. **必须明确验证**
   - 最终响应必须说明：
     - 如何复现
     - 根因是什么
     - 修复了哪一层
     - 跑了哪些验证或测试
   - 不允许只说“应该已修复”或“现在应该可以了”。

**推荐调试提示词（中文）**

- `先复现这个精确问题，再修复。`
- `先定位具体失败层，不要猜。`
- `不要做投机式修复，只修根因。`
- `给这个问题补一个回归测试。`
- `修完后告诉我你验证了什么。`

**推荐用户提示模板（中文）**

```text
先复现这个精确问题。
识别具体失败层。
不要做猜测性修复。
修复根因。
为这个精确问题添加回归测试。
最后告诉我你验证了什么。
```

## 7) 对用户的响应风格

- 简洁、直接、以实现为中心。
- 说明改变了什么、在哪里、以及为什么。
- 如果被阻塞，明确说明阻塞点和下一步最佳行动。
- 不要让用户做代理自己可以直接完成的工作。

## 8) 更新 BUILD.md

- 每次实现或变更后，追加内容到 BUILD.md。如果文件为空，则表示这是新的构建，仍需总结变更。**不要**在 BUILD.md 中包含验证信息。

## 9) 文档要求

- 文档是必要上下文，而非可选润色。在改变行为前，必须先阅读 `docs/` 中最相关的文档，并将其视为规范的一部分。
- `README.md` 也属于必要上下文。开始实现前必须检查其是否与当前能力和运行方式一致；如果本次变更会影响项目概览、安装/运行方式、配置、目录结构或主要特性，必须同步更新 `README.md`。
- 从最窄的相关文档开始，而不是整个文件夹。使用 `rg --files docs` 和聚焦读取找到与任务精确匹配的工作流/管理员文档。
- 如果文档与代码不一致，不要默默选择一方。需在代码和/或文档中解决差异，并在响应中说明所做的假设。
- 对触碰的每一个功能、工作流、权限规则或面向运维的行为都必须维护文档。如果没有合适的文档，在最接近的相关章节中创建。
- 推荐使用两层文档结构（当有帮助时）。

**为未来代理快速理解而编写的文档要求**：
- 开头说明目的、路由/访问范围，以及特性存在的原因。
- 包含真实的 workflow 顺序，而非仅字段描述。
- 明确指出隐藏依赖：权限、后台效果、外部服务、必需配置等。
- 在合适时注明文件路径、表名、路由名称或处理程序名称。
- 优先使用紧凑的章节和平行项目符号，而不是大段叙述。
- 每次改变行为时，同步更新文档。不要把文档留作后续任务。
- 内部文档必须明确记录“gotchas”和调试线索，帮助未来代理避免重复发现相同问题。

## 10) 项目规格 / Project Specs

`cuddlePanel` 是一个内部服务器管理系统，目标是提供企业级、移动端友好的管理员控制台。

技术与交互约束：
- 后端使用 C++。
- 前端使用 Bootstrap CSS。
- 前端与后端通过 AJAX 通信；除页面导航外，不应依赖整页刷新或重载。
- 使用模板与 partials 复用侧边栏、顶部栏和页脚。

必须覆盖的核心能力：
- 极其安全的认证系统，密码哈希使用 Argon2id。
- 首次运行引导，用于创建超级管理员账户。
- 用户管理：添加/删除用户，并对每个页面分配权限。
- 权限模型：每个页面至少区分 `view` 与 `manage` 权限；仪表盘也适用该模型。
- 服务管理：创建新服务、编辑现有服务，以及启动、停止、重启。
- 系统账户管理：管理系统用户、sudoers、系统账户，以及 `chown`、`chmod` 相关操作。
- Nginx 管理：管理 `sites-available` 与 `sites-enabled`，并支持在面板中进行富文本配置编辑。
- 终端页面：提供 JavaScript 终端，返回流式命令输出，支持交互式终端程序、`screen` 等，会话行为应尽量贴近真实终端。
- Codex 页面：发送提示到 Codex，并查看输出和产生的变更。
- 部署页面：以表单方式镜像 `/sbin/deploy-site` 的工作流。

实现这些能力时必须遵守的额外要求：
- 所有高风险系统操作必须先完成服务器端权限校验、输入白名单校验和路径/命令约束，再暴露可执行入口。
- 交互式终端必须基于 PTY/流式会话实现，不允许简单地把任意命令透传给 shell。
- Nginx、系统用户、权限位与服务控制等能力必须有清晰的 allowlist、审计思路和失败反馈。
- 移动端可用性不是附加项；新页面默认需要兼顾桌面和移动端布局。
