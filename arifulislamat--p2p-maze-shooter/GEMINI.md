# AGENTS Instruction

## Big picture

- This is a single-page, vanilla JS P2P shooter (no build step). All source files (`.js`, `.css`, `.html`) are now located in the `src/` folder. The entry point is `src/index.html`, which loads scripts in order: `themes/retro-neon.js` -> `themes/midnight-void.js` -> `themes/sandstorm.js` -> `themes/cyber-sakura.js` -> `themes/index.js` -> `core/ThemeManager.js` -> `constants.js` -> `sound.js` -> `physics.js` -> `renderer.js` -> `network.js` -> `game.js`. Theme files must load before `constants.js` so theme data is available when the registry is queried.
- Core loop and state live in the `Game` module (IIFE). `Game` owns the tick, input handling, mode switching (local/online host/online guest), and state transitions.
- Networking is host-authoritative. The host runs the canonical physics simulation. **P1 (host's player)**: the host sends its input keys **and authoritative P1 position** to the guest at **60 Hz** (`host_input` message). The guest runs `processRemoteInput(p1)` for responsive bullet creation only; the rendered P1 position is **directly applied** from `hostP1State` every frame (not gated by physics tick). **P2 (guest's player)**: the guest sends its input to the host at **60 Hz** (`input` message); the host processes P2's movement authoritatively and sends back P2's position/direction at **60 Hz** inside the `host_input` message. The guest applies P2 positions directly from `hostP2State` every frame — a "dumb terminal" rendering model that eliminates jitter by simply displaying whatever the host last sent. The guest still handles P2 shooting locally for responsive bullet creation. The host also sends lightweight **authority corrections** at **10 Hz** (`correction` message) containing player health/scores, both player positions, bombs, zombies, health packs, explosions, and game state; correction positions also update `hostP1State`/`hostP2State`. Authority state (health, score, alive, game state) is snapped immediately. Bomb/zombie/health-pack **spawning** only happens on the host; the guest ticks existing entities but receives new ones via corrections. Full state (`broadcastState`) is only sent once after reconnection and also seeds `hostP1State`/`hostP2State`. See `sendHostInput()`/`sendCorrections()`/`applyCorrections()` in `game.js`.
- **Render interpolation** — the game loop uses a fixed 60 Hz physics timestep with `requestAnimationFrame` for rendering. On the **host/local** side, player and bullet positions are interpolated between the previous and current physics state using `alpha = physicsAccumulator / PHYSICS_STEP_MS` to avoid micro-stutter on high-refresh displays. `savePhysicsPositions()` snapshots `prevX/prevY` before each physics step; `interpolateForRender(alpha)` temporarily overwrites entity positions for drawing; `restorePhysicsPositions()` puts them back after rendering. On the **guest** side, player positions are **not** render-interpolated — they are applied directly from host state every frame (the "dumb terminal" model), so save/interpolate/restore only applies to **bullets**. Bombs, explosions, and zombies move slowly enough that jitter is imperceptible without interpolation, so they are not interpolated.

## Architecture and data flow

- All source files are now in the `src/` folder:
  - `constants.js` defines gameplay constants, maze data, and `parseMaze()`. It also owns the global `activeMaze` used by `Physics` and `Renderer`.
  - `physics.js` uses `activeMaze.walls` for collision checks; keep wall rectangles aligned to the maze grid.
  - `renderer.js` renders the maze grid (`activeMaze.grid`) and HUD every frame. It assumes the retro color palette in `COLORS` from `constants.js`.
  - `network.js` wraps PeerJS (via CDN) and manages room codes, connection lifecycle, and the host/guest roles. The DataChannel uses **`reliable: false`** (unordered, unreliable SCTP — true UDP semantics) to eliminate head-of-line blocking on the 60 Hz position stream. Critical one-time messages (`config`, `resync`, `ready`, `restart_request`) are sent via `Network.sendReliable()`, which stamps a unique `_rid` and transmits the message 3× (at t+0, t+40ms, t+95ms); the receiver deduplicates by `_rid` using a 128-entry ring buffer in `setupConnection`'s `data` handler. High-frequency messages (`host_input`, `input`, `correction`) use plain `Network.send()`. `host_input` carries an incrementing `seq` counter; the guest drops any packet whose `seq <= lastHostInputSeq` to discard out-of-order deliveries. Both counters reset to 0 in `startOnlineGame()` so a new match never inherits stale state from the previous session. It attaches a `peer.on('disconnected')` handler on every peer that calls `peer.reconnect()` automatically when the signaling WebSocket drops (e.g. mobile backgrounding). `restoreHost(roomCode, cbs)` and `restoreGuest(roomCode, cbs)` **reuse the existing Peer when possible** — `restoreHost` checks `peer.id === expectedId && !peer.destroyed` and, if true, just calls `peer.reconnect()` (for a dropped signaling WS) or fires `onReady` immediately (if the peer is already open) instead of destroying and creating a new Peer; `restoreGuest` similarly reuses the existing Peer and calls `peer.connect()` directly when `peer.open`, or `peer.reconnect()` when the signaling WS dropped. A new Peer is only created when the old one is destroyed or has the wrong ID. This avoids `unavailable-id` errors from the PeerJS server’s ~60 s peer ID TTL and eliminates redundant signaling WebSocket connections on every retry. All event handlers attached during original peer creation (open, connection, error) reference the module-level `callbacks` object so they stay correct after `callbacks = cbs` is updated in each restore call. `getLastRoomCode()` / `setLastRoomCode(code)` expose/set the saved code. `generateRoomCode()` is public so `game.js` can pre-generate a code before registering the peer. `createHostWithCode(roomCode, cbs)` creates a host peer with an explicit code (used in Phase 2 of deferred room creation). Key internal invariants: (1) `setupConnection(connRef)` and `startOpenPoll(connRef)` take an explicit `DataConnection` reference rather than reading the module-level `conn`, preventing stale references when `conn` is replaced during reconnects; (2) all paths that call `markConnected()` guard with `connRef === conn` before firing; `startOpenPoll` also cancels itself immediately when `connRef !== conn`; (3) `close`/`error` handlers guard with `connRef === conn` before firing callbacks, so events from replaced connections are silently dropped; (4) the `intentionallyDestroyedPeers` **Set** (replacing the old boolean `intentionalDestroy` flag) suppresses `onPeerClosed` when `peer.destroy()` is called deliberately; the peer reference is added to the Set before calling `destroy()` and removed when its `close` event fires, avoiding the async timing race of a reset-before-fire boolean; additionally `attachPeerLifecycleHandlers` guards the `close` handler with `peerObj !== peer` to drop late close events from replaced peer instances; (5) `disconnect()` resets `isHost` and `lastRoomCode` to avoid leaking stale state into the next session; (6) `PEER_DEBUG` is `2` on `localhost` and `0` everywhere else; (7) ICE servers are limited to the 5 known-good Google STUN servers — dead community VoIP STUN servers have been removed to reduce ICE gathering latency.
  - `sound.js` provides procedural audio via Web Audio API (IIFE, `Sound` global). All 11 sound effects are synthesized — zero audio files. Both host and guest generate sounds locally from their own simulation (`Sound.play()` calls in shared physics functions like `tryShoot`, `handlePlayerDeath`, `explodeBomb`, etc.). On reconnection, the one-shot `broadcastState` still includes `sounds` via `Sound.flushPending()` for backward compatibility. Stereo panning is position-based (event x-coordinate mapped to left/right speaker). Mute state is persisted in `localStorage('p2p-muted')`.
  - `themes/retro-neon.js` defines the built-in `retro-neon` theme as a plain JS object (`const retroNeonTheme`) with `id`, `name`, `colors` (~60 keys), `fonts.canvas` (13 keys), `rendering` settings, `sounds` (all `null` — procedural audio), and `cssVars`. Adding a new theme requires only this file plus one registration line.
  - `themes/index.js` is the theme registry (`const ThemeRegistry`). It holds all registered themes keyed by `id` and declares `defaultTheme`. Register a new theme by adding one entry: `ThemeRegistry.themes[myTheme.id] = myTheme`.
  - `core/ThemeManager.js` is the runtime singleton (IIFE, `const ThemeManager`). Public API: `init()` (loads saved theme from `localStorage`), `apply(themeId)` (mutates `CONFIG.COLORS` via `Object.assign`, calls `Renderer.applyThemeFonts()`, injects CSS vars via `root.style.setProperty`, persists to `localStorage`, dispatches `themechange` event), `getThemeList()`, and read-only getters (`active`, `colors`, `fonts`, `rendering`, `sounds`, `cssVars`). `ThemeManager.init()` is called at the top of `Game.init()`.
  - `index.html` contains the lobby UI and inline script for building the maze selector; it calls `Game.*` entry points directly. The lobby includes a `#theme-select` dropdown populated by `initThemeSelector()` (defined in the inline script).
  - **Mobile touch controls** — `#touch-controls` (inside `#gameContainer`) is a full-width bar shown only under `@media (orientation: portrait) and (max-width: 1024px)`. Left half: `#shoot-btn` (maps `touchstart/end` → `keys["Space"]`). Right half: `#joystick-zone` with a floating-knob joystick that maps 8 directional sectors → `keys["KeyW/A/S/D"]`. Wired in `initTouchControls()` in `game.js`, called from `init()`. `resizeCanvas()` subtracts `#touch-controls.offsetHeight` so the canvas never overlaps the controls bar.

## Project-specific conventions

- Modules are plain IIFEs in the global scope (no bundler, no imports). Keep new code consistent with this pattern.
- `activeMaze` is a shared global; when changing mazes call `parseMaze()` and update `selectedMazeKey` so Renderer/Physics stay in sync.
- Online guest uses WASD + Space for input (same as host). Local 2-player uses WASD/Space for P1 and arrows/Enter for P2.
- Maze rotation happens every 1 minute; all 6 mazes are shuffled and played in order, with the user's lobby-selected map always placed first via `shuffleMazeOrder(firstMazeKey)`. After all 6 mazes (6 minutes total), the match ends and the player with the most kills wins. A player can also win instantly by reaching `WIN_SCORE` (8) kills. If scores are tied at timeout, health is used as a tiebreaker; otherwise it's a draw.
- **Player movement** is diagonally normalized (no sqrt(2) speed exploit) — `processPlayerInput` accumulates `inputDX`/`inputDY` then divides by `Math.hypot` before multiplying by `PLAYER_SPEED`.
- **Zombie movement** — zombies are no longer static. Each physics tick, a zombie picks the nearest alive player and steps toward them at `ZOMBIE_SPEED` px/tick, using `Physics.canMoveTo` for per-axis wall sliding (same approach as player movement). Contact still freezes the player and removes the zombie.
- **Health packs** — a `healthPacks[]` array is host-spawned every `HEALTH_PACK_SPAWN_INTERVAL` ms (up to `HEALTH_PACK_MAX` on field). Walking within `PLAYER_SIZE * 1.2` px of a pack grants `+HEALTH_PACK_HEAL` HP (capped at `PLAYER_HEALTH`). Health packs are synced in both `sendCorrections()` and `broadcastState()` — `updateHealthPacks(dt, skipSpawn)` follows the same host/guest `skipSpawn` pattern as bombs and zombies.
- **Floating texts** — `floatingTexts[]` holds `{ text, x, y, color, startTime }` entries. `addFloatingText()` is called on bullet hit (`-1`), bomb blast (`-2`), player death (`ELIMINATED!` / `+1 KILL`), and health pack pickup (`+3 HP`). Texts float upward and fade out over `FLOATING_TEXT_DURATION_MS` ms; expired entries are pruned once per frame in the game loop.
- **Damage flash** — each player has a `damageFlashTimer` (ms) that is set to `DAMAGE_FLASH_MS` on every hit (bullet or bomb). The renderer draws a color-tinted full-screen overlay (`drawDamageFlash`) whose alpha scales with the remaining timer. Timer decrements in the `updateZombies` freeze-timer loop that already runs every tick.
- **Low-health vignette** — `drawLowHealthVignette` draws a pulsing radial gradient in each player's color when their HP is at or below `LOW_HEALTH_THRESHOLD`.
- **Speed boost pickups** — `speedBoostPickups[]` array; host-spawned every `SPEED_BOOST_SPAWN_INTERVAL_MS` (up to `SPEED_BOOST_MAX: 1` on field). Walking within range grants `speedBoostTimer = SPEED_BOOST_DURATION_MS` and applies `SPEED_BOOST_MULTIPLIER: 1.6×` speed in `processPlayerInput`/`processRemoteInput`. Synced via corrections + broadcastState. Rendered as a pulsing yellow lightning bolt by `Renderer.drawSpeedBoostPickups()`.
- **Weapon pickups** — `weaponPickups[]` array; host-spawned every `WEAPON_SPAWN_INTERVAL_MS` (up to `WEAPON_MAX: 1` on field). Each pickup is randomly `'rapidfire'` or `'scatter'`. Walking within range sets `player.weaponType` and `player.weaponTimer = RAPID_FIRE_DURATION_MS / SCATTER_DURATION_MS`. `tryShoot` checks `weaponType`: scatter fires 3 bullets spread ±`SCATTER_SPREAD_DEG`; rapidfire uses `RAPID_FIRE_RATE_MS` cooldown. Timer expires → `weaponType = 'normal'`. Synced via corrections + broadcastState. Rendered by `Renderer.drawWeaponPickups()` (blue = rapidfire, orange = scatter).
- **Bullets kill zombies** — in `updateBullets`, zombie collision is tested before player collision. A bullet hitting a zombie destroys both and shows a green "ZOMBIE!" floating text.
- **Maze urgency (last 10 s)** — once `mazeTimeLeft <= URGENT_MAZE_TIME_S`, the render loop fires `Sound.play("countdown", ...)` once per remaining second and sets `screenShakeDuration = 180 ms`. `lastUrgencySecond` tracks the last second fired to avoid double-triggering; reset to `-1` when `mazeTimeLeft > URGENT_MAZE_TIME_S` so it arms again for the next maze.
- **Screen shake** — `screenShakeDuration` (ms) decays by `rawDt` each frame. While active, `Renderer.beginFrame(sx, sy)` translates the canvas by a random ±3 px offset scaled to the remaining duration. `Renderer.endFrame()` restores the context. Shake is triggered by the urgency countdown.
- **Gameplay tuning constants** are in `CONFIG.GAMEPLAY` in `constants.js` (damage flash duration, low-health threshold, health pack rates, floating text duration, urgency seconds, speed boost/weapon pickup rates and durations). All per-feature values are alongside their entity constants (`BOMB.*`, `ZOMBIE.*`) so tuning everything is in one place.
- **Theme system** — all visual colors and canvas fonts are data-driven. `CONFIG.COLORS` (in `constants.js`) is the runtime color map; `ThemeManager.apply()` mutates it in-place with `Object.assign`. `RENDER_CONFIG.FONTS` (in `renderer.js`) is the runtime font map; `Renderer.applyThemeFonts()` mutates it in-place. CSS custom properties are injected directly onto `document.documentElement.style` so the DOM UI can use `var(--token)` without any bundler. **To add a new theme:** (1) create `src/themes/my-theme.js` that declares `const myTheme = { id, name, colors, fonts, rendering, sounds, cssVars }` mirroring `retro-neon.js`; (2) add `<script src="themes/my-theme.js">` before `themes/index.js` in `index.html`; (3) add `ThemeRegistry.themes[myTheme.id] = myTheme` in `themes/index.js`. No other files need modification. All `*RGB` color keys (e.g. `p1RGB`, `explosionShockwaveRGB`) are bare triplets (`"R,G,B"`) used in template-literal `rgba(${COLORS.xRGB}, alpha)` expressions for animation-driven transparency.
- **Immediate room creation** — tapping "CREATE GAME ROOM" calls `showOnlineUI('host')` which calls `Network.generateRoomCode()`, displays the room code and share link, and **immediately** registers with PeerJS via `setupHostRoom(roomCode, true)`. There is no longer a deferred Phase 2 — the peer is created right away.
- **Lobby retry limit** — `setupHostRoom` tracks retries via `lobbyRetryCount` (module-level). After `LOBBY_MAX_RETRIES = 5` failures, an error message is shown and the retry loop stops — the user can press Cancel to return to the lobby. The count is reset on success (`onConnected`), on `showOnlineUI('host')`, on `cancelOnline()`, and on `returnToLobby()`.
- **Ready handshake** — after the DataChannel opens, the guest immediately sends `{ type: "ready" }`. The host's `setupHostRoom.onData` intercepts this and only then calls `startOnlineGame(true)` + sends `{ type: "config" }`. This prevents the race where the host sends config before the guest's `onData` handler is wired. If the host is **already mid-game** when it receives "ready" (e.g. the guest refreshed or joined late), `handleNetworkData` catches it and responds with `config` + `resync` + `broadcastState(true)` to bootstrap the guest into the running match — config starts the guest's game loop, resync corrects maze/timer state, and broadcastState syncs all positions/scores.
- **Resync vs config** — mid-game reconnection uses a new `{ type: "resync" }` message instead of `{ type: "config" }`. The `resync` message syncs `mazeKey`, `mazeOrder`, `mazesPlayed`, `matchStartTime`, and `mazeRotationStart` without resetting scores or restarting the match. `handleNetworkData` handles both types.
- **Mobile reconnection** — `STATE.RECONNECTING` is entered when the DataChannel closes but the game is still in `online-host`/`online-guest` mode. `game.js` runs a **75 s** reconnect window (increased from 30 s to outlast PeerJS server’s ~60 s peer ID expiry TTL): every 3 s it calls `Network.restoreHost` or `Network.restoreGuest` which **reuse the existing Peer** (only closing the DataConnection, not the Peer itself) so there are no redundant signaling WebSocket connections or `unavailable-id` errors. A `visibilitychange` listener on `document` triggers the same flow proactively when the tab comes back to the foreground; it also handles the lobby phase — if the peer died while waiting for a guest, it re-registers the room with the same code. On success the host re-sends `resync` + a full state broadcast. On timeout, `handleDisconnect` falls through to the normal lobby return. The amber `drawReconnecting(secondsLeft)` overlay in `renderer.js` shows a countdown during this window.
- **Guest data-timeout detection** — `lastDataReceived` is updated in `handleNetworkData`. If the guest is in active play and receives no data for `DATA_TIMEOUT_MS = 5000` ms, `handleDisconnect()` is triggered automatically (covers silent host crashes / network drops with no WebRTC close event). `lastDataReceived` is reset to `Date.now()` in `startOnlineGame` and to `0` after a timeout trigger to prevent repeated triggers.
- **Session persistence (page reload)** — `startOnlineGame()` saves `{ role, roomCode }` to `sessionStorage` under key `p2p-maze-shooter`. On page load (`DOMContentLoaded` in `index.html`), `Game.rejoinSession()` reads this and auto-creates the host room with the same code or auto-joins as guest — before checking for `?room=` URL params. The session is cleared in `returnToLobby()`, `cancelOnline()`, `showOnlineUI()`, and on hard join errors so it doesn't loop; `pendingResumeState` is also nulled in those same paths to prevent stale game state from bleeding into a new match. This lets both host and guest survive a full page reload: the host re-registers the same peer ID and the guest reconnects to it. **Game state preservation** — `sendCorrections()` (10 Hz) also calls `saveSession` with the full game state (`mazeOrder`, `selectedMazeKey`, `mazesPlayed`, `matchStartTime`, `mazeRotationStart`, `p1Score`, `p2Score`). `rejoinSession()` for the host stores this as `pendingResumeState` before calling `setupHostRoom`. `startOnlineGame()` checks `pendingResumeState` when `isHost`: if set, it restores the saved maze/score state and sets `gameState = STATE.PLAYING` directly (skipping the countdown). Immediately after `config`, `setupHostRoom.onData "ready"` always follows with `resync` + `broadcastState(true)` so the guest is also bootstrapped with the restored state.

## Workflows

- No build system or tests. Open `src/index.html` directly in a browser (or serve statically if needed).
- PeerJS is loaded from `https://cdn.jsdelivr.net/...` in `index.html`; changes to networking should account for PeerJS behavior and events in `network.js`.
